================================================================================
КАК СОЗДАВАЛСЯ OXSAR-NOVA
================================================================================

Дневник работы над переписыванием legacy-игры oxsar2 (PHP/Yii 1.1 + MySQL,
~2018 г.) на современный стек Go + TypeScript + PostgreSQL + Redis.

Формат: по шагам, от входного запроса до финального каркаса. Цель — дать
материал для будущей статьи: какие решения принимались, где компромиссы,
что сознательно отложено в TODO.

Дата: 2026-04-21
Среда: Claude Code (Opus 4.7 1M context), Windows, IDE с диагностикой.

================================================================================
4. АРХИТЕКТУРНЫЕ РАЗВИЛКИ (почему Go, почему такой роутинг и т.д.)
================================================================================

Go vs Rust vs Node:
  - Node отпал: хотим одиночный бинарник, предсказуемую память, простой
    деплой. У Node хуже ситуация с долгоживущими event-loop-воркерами
    и параллельными шардами.
  - Rust — перебор. Компилируется долго, новых разработчиков искать
    сложнее, производительность и так будет не bottleneck (bottleneck —
    БД и игровая логика).
  - Go — «скучный» и продуктивный. Плюсом — slog/context/errgroup в стде,
    без зависимостей.

Postgres vs MySQL:
  - MySQL был в oxsar2. Мотив отказа — JSONB для payload событий,
    CITEXT для имён пользователей, range types для координат галактики,
    FOR UPDATE SKIP LOCKED для воркера.

sqlc vs pgx напрямую vs ORM:
  - ORM с магией (gorm/ent) — нет, легаси oxsar2 как раз страдал
    от WarperActiveRecord-магии.
  - sqlc: SQL пишем сами, кодогенерация типов. Компромисс между
    безопасностью и контролем.

Транспорт:
  - REST + WebSocket. GraphQL — отложен, не окупается для игры с
    небольшим числом эндпоинтов.

OpenAPI как источник истины:
  - В oxsar2 фронт и бек договаривались «в коде» (JS-строки ходили к
    AjaxRequestController::*). Тут один yaml-файл генерит go/ts типы.

UUIDv7 vs int autoincrement:
  - UUIDv7 монотонный (ok для btree-индекса), не раскрывает cardinality,
    нет гонок при merge из нескольких инстансов.

Вывод 4: большинство развилок — компромисс между «автоматическая
магия» и «явный контроль». Для долгоживущего проекта с большой
командой я выбирал контроль.


================================================================================
5. КАК ПРОЕКТИРОВАЛСЯ EVENT-LOOP (§6 ТЗ)
================================================================================

В oxsar2 расчёт полётов/очередей делал CronCommand.php — 40 КБ кода,
запускавшийся по расписанию. Проблемы:
  - Падение в середине оставляло полусостояние (см. §18.3 ТЗ).
  - Один процесс, не шардируется.
  - Вызывался и из веба «на лету», когда игрок заходил в планету —
    то есть один и тот же код бегал из двух контекстов.

В nova event-loop:
  - Таблица events с fire_at + state enum(wait|start|ok|error).
  - Воркер раз в 10 сек делает:
      SELECT ... FROM events WHERE state='wait' AND fire_at<=now()
      FOR UPDATE SKIP LOCKED
  - Каждое событие идёт через handler, обернутый в транзакцию.
  - При пользовательском запросе «дай состояние планеты» сервер
    синхронно прогоняет pending-events этого пользователя. Функция
    одна и та же — мы просто передаём другой ctx.

Ключевое: каждый handler ДОЛЖЕН быть идемпотентным. Повторный запуск
с тем же event_id не должен дублировать эффект. Пример —
HandleBuildConstruction в internal/event/handlers.go:

  var cur int
  ...SELECT level FROM buildings WHERE planet_id=.. AND unit_id=..
  if cur >= pl.TargetLevel {
      // уже применено — идемпотентность, просто закрываем queue
      return nil
  }

Вывод 5: идемпотентность — это не «гигиена», это инвариант для воркера
с retry. В статье — обязательно сказать про неё явно.

Java-движок:
  Assault.java        1941 строк  — ядро
  Participant.java     988 строк  — сторона боя + технологии
  Party.java           778 строк  — ACS/инопланетяне
  Units.java           650 строк  — unit stats + rapidfire
  Database.java        126 строк  — MySQL I/O
  = 4483 строк всего

План порта:
  - Database.java НЕ портируем. В Go — sqlc-репозиторий рядом с
    движком, сам battle.Calculate — чистая функция.
  - Остальные классы один-в-один в internal/battle/.
  - RNG. В Java — java.util.Random с системным seed. В Go мы ОБЯЗАНЫ
    принимать seed параметром, чтобы симулятор фронта давал тот же
    результат. Для бит-паритета напишем адаптер JavaRandom (LCG с теми
    же константами, что и Oracle JDK), когда включим cross-verification.

Верификация:
  - Тест: берём одинаковый JSON-вход, прогоняем через Assault.jar и
    через Go-движок, сравниваем потери по unit_id (допустимо
    расхождение 0).
  - Jar живёт в d:\Sources\oxsar2-java\assault\dist\oxsar2-java.jar,
    вызывается как внешний процесс из Go-теста.

Сейчас в проекте:
  - internal/battle/engine.go — каркас с циклом раундов и summarize.
    prepareTurn / distributeShots / resolveDamage — TODO (M4).
  - internal/battle/types.go — полная модель Input/Report, включая
    три канала урона (normal/laser/ion), rapidfire, Tech.{Ballistics,
    Masking}.
  - internal/battle/engine_test.go — тесты детерминизма и валидации
    уже проходят (на плейсхолдере).

7. БЕЗОПАСНЫЕ УЛУЧШЕНИЯ — МАТЕРИАЛ ДЛЯ СТАТЬИ
================================================================================

Эти 23 пункта — самое интересное для читателя, потому что показывают,
где legacy действительно был плох (а не «мы перепишем, потому что
модно»). Выборка самых наглядных:

#18.1  numeric(20,4) вместо double MySQL.
       В oxsar2 на длинной дистанции триллионы ресурсов переставали
       точно складываться. Это не теоретический баг — игроки на
       старых серверах OGame-клонов реально упирались.

#18.2  Убить Sim-таблицы.
       Параллельные sim_* (na_sim_user, na_sim_assault и т.д.) были
       попыткой «песочницы для симулятора». В результате две реализации
       одной логики, расхождение, баги. Решение — симулятор = чистая
       функция, без БД вообще.

#18.4  Seed в бой.
       Тест «воспроизведи баг из тикета №123» превращается из
       археологии в git-blame + один json-файл с seed'ом.

#18.6  Bashing включить по умолчанию.
       BASHING_PERIOD=0 в oxsar2 = никакой защиты. Минимальная
       боевая этика — 6 атак/24 часа — встроена сразу.

#18.9  Единообразное округление.
       В oxsar2 floor/ceil разбросаны без системы. Решение: списание
       всегда ceil в пользу системы, начисление — floor в пользу
       игрока, «floating accumulator» для добычи per-second.

#18.11 Единый YAML-реестр юнитов.
       В legacy UNIT_METALMINE=1 было захардкожено в трёх местах:
       consts.php, java-движке, клиентском JS. Решение: один YAML,
       кодогенерация Go-констант + TS-типов + БД-seed.

#18.14 Админы в БД, не в конфиге.
       $GLOBALS["ADMIN_USERS"] = [1, 3] требовало релиза на каждый
       новый адм. Решение — users.role enum + admin UI.

#18.16 IP-check действительно работает.
       TODO-комментарий «Make ipcheck!» висел годы. Сейчас привязка
       refresh-токена к /24 включена по умолчанию.

Вывод 7: «rewrite» часто продают как «сделаем всё быстрее/красивее».
На практике 70% ценности переписывания — это мелкие, но систематически
правильные изменения, которые в старом коде были бы риском на каждом
шаге. В статье это основной нарратив.

10. СТАНДАРТЫ РАЗРАБОТКИ (§17 ТЗ) — НЕ ФОРМАЛЬНОСТЬ
================================================================================

Раздел «стандарты» в ТЗ обычно воспринимается как скучный. На самом
деле это самая практичная часть: без него через полгода три
разработчика будут писать код в трёх разных стилях.

Что вынесено в обязательное:
  - Go: golangci-lint (errcheck, govet, staticcheck, revive, gocritic,
    gosec, bodyclose, contextcheck, nilerr, prealloc, errorlint,
    misspell) + запрет init-побочек + slog с полями + sqlc-only.
  - TS: strict + noUncheckedIndexedAccess + exactOptionalPropertyTypes;
    запрет any/default-export/console.log; server-state только в
    TanStack Query, UI-state только в Zustand — смешивать запрещено.
  - Коммиты: Conventional Commits + PR ≤ 400 строк + 1 approve (2 для
    battle/economy/auth/market).
  - Тесты: пирамида (unit > integration > e2e), coverage 70%/85%,
    t.Parallel() по умолчанию, flaky — в карантин.
  - Observability: OTel traces, Prometheus RED/USE, slog JSON с
    trace_id, PII либо хешировать, либо короткий retention.
  - Security: Dependabot еженедельно, CVE за 48ч, gitleaks
    pre-commit, audit_log с before/after JSONB.

Всё это сразу положено в .golangci.yml, .eslintrc, .github/workflows/
ci.yml, CLAUDE.md. То есть не «когда-нибудь настроим», а с первого
коммита.

Вывод 10: стандарты нужно материализовать в конфиги и CI сразу, иначе
они превращаются в словарный пункт, на который никто не смотрит.

14. ИТЕРАЦИЯ 2 — ЗАКРЫТИЕ M1 И M2 ПОЛНОСТЬЮ
================================================================================

После каркасной сессии следующая логичная цель — сделать проект
играбельным на уровне «зарегался → планета → здание → исследование →
корабли». Это как раз M1+M2.

Что добавлено:

  А. Стартовая планета при регистрации.
     planet.Starter с seed-детерминированным поиском свободной
     координаты (до 100 попыток), первый res_log с reason='admin_gift'.
     Auth.Service получил интерфейс StarterPlanetAssigner —
     интерфейс (не прямой импорт пакета planet), чтобы избежать
     цикла импорта и сохранить testability auth без mocks всего
     планетарного пакета.

  Б. Requirements checker.
     Отдельный пакет internal/requirements. Один Check(ctx, tx,
     targetKey, userID, planetID) вызывается из building, research,
     shipyard. YAML configs/requirements.yml содержит 40+ записей,
     от «research_lab >= 3 для spyware» до полных цепочек
     (plasma_tech требует ion+laser+energy на определённых уровнях).
     Ошибка ErrNotMet с полями Kind/Key/Need/Have — handler'ы
     превращают её в 400 с человекочитаемым сообщением.

  В. Research service.
     Очередь в той же таблице construction_queue (unit_type='research').
     Одно исследование на игрока одновременно (§5.4 ТЗ). Формула
     времени: (m+s) / (1000 * (1 + research_lab_level)) / GAMESPEED.
     Handler и OpenAPI — GET /api/research, POST /api/planets/{id}/research.

  Г. Shipyard service.
     Единая очередь для кораблей и обороны. Enqueue принимает
     count > 0. Per-unit time × count. Различие ship/defense — флаг
     is_defense в payload события, применяется в той же event-функции
     HandleBuildFleet (разный INSERT в `ships` vs `defense`).

  Д. Event handlers: research + fleet/defense.
     Все идемпотентные. HandleResearch повторяет HandleBuild пост-
     роения, но обновляет таблицу research (уровни игрока, не планеты).
     HandleBuildFleet идемпотентен через проверку status='done' у
     shipyard_queue.

  Е. Frontend: экраны Buildings, Research, Shipyard.
     PlanetSwitcher в App.tsx, локальный useState-таббар (без
     TanStack Router — меньше зависимостей, пока хватает). Все
     экраны используют один клиент api.* + общий каталог (фронтовая
     копия units.yml) в src/api/catalog.ts.

Ключевые наблюдения на этой итерации:

  1. Паттерн «сервис + handler + event» повторяется 1-в-1. В третий
     раз (shipyard) это уже очевидно, можно выделить генератор.
     Пока не делаю — три копии это три копии, а не premature
     abstraction (§17.2 ТЗ как раз про это).

  2. Requirements-checker окупается мгновенно. Первая версия
     (§14.Б) — 120 строк + 2 теста, но используется в трёх разных
     сервисах. Если бы тащили логику в каждый сервис отдельно —
     было бы ~300 строк дубля и риск расхождения формул.

  3. Интерфейс StarterPlanetAssigner в auth — пример «сужающего
     интерфейса для dependency inversion». Auth не должен знать,
     что такое planet.Service со всей его логикой тика экономики.
     Ему достаточно «кто-то, кто умеет Assign». Это классика SOLID,
     но в Go такие интерфейсы особенно естественны.

  4. События и идемпотентность ещё раз доказали себя. HandleResearch
     проверяет «текущий уровень >= target» — если воркер упал между
     UPDATE research и UPDATE construction_queue, повторный прогон
     просто закроет очередь, не повторяя апгрейд. Без этой проверки
     мы бы тайком давали игроку двойной уровень.

  5. Seed-детерминизм в Starter. Казалось бы, «новая планета —
     случайность, зачем seed». Но в тестах (и при repro-сессиях от
     поддержки) стабильность даёт повторяемость: для юзера X планета
     всегда в 4:123:7. Это не геймплейное ограничение — первая
     свободная клетка всё равно случайная для человека. Зато тесты
     ловят регрессии без mock-ов RNG.

  6. Диагностика IDE про «undefined: pgx, undefined: chi, etc.» —
     чисто локальная: модули объявлены в go.mod, но для Go LSP нужен
     первый `go mod download`. В CI этот шаг делается на pre-install.
     Это хорошая отметка для статьи: «не каждый красный крестик в
     IDE — это баг».

Статистика после итерации 2:
  backend/ .go файлов: 37 (было 27), LOC ~3800
  миграции: без изменений, хватает схемы из M0
  configs/: +research.yml, +requirements.yml (7 YAML всего)
  api/openapi.yaml: +6 эндпоинтов
  frontend/ .tsx/.ts: 17 (было 12)

Что теперь реально можно сделать в UI:
  POST /api/auth/register -> получаешь планету
  GET  /api/planets       -> видишь её
  POST /api/planets/X/buildings -> строишь metal_mine
  ждёшь ~секунду, worker добавляет уровень
  POST /api/planets/X/research  -> стартуешь computer_tech
  POST /api/planets/X/shipyard  -> ставишь в очередь 5 light_fighter

То есть это уже playable loop. Не весь, но достаточный, чтобы
показать работоспособность архитектуры в QA/демо.

16. ИТЕРАЦИЯ 4 — ОТЗЫВ ОШИБОЧНОГО "УЛУЧШЕНИЯ" (§18.21)
================================================================================

Пользователь обратил внимание на файл game/Artefact.class.php:1061
и указал, что users.exchange_rate меняется индивидуально.
Я полез смотреть — и оказалось, что пункт §18.21 ТЗ был написан НА
ОШИБОЧНОМ ПОНИМАНИИ оригинального кода.

Что я утверждал в §18.21 (неверно):
  «В oxsar2 users.exchange_rate висел как поле игрока, хотя курс
  биржи должен быть свойством самой биржи/вселенной. В nova: курс — в
  configs/market.yml, персональные корректировки — как артефакт/
  офицер, а не как поле юзера.»

Что на самом деле в коде:
  ARTEFACT_MERCHANTS_MARK (id 300) при активации делает
  UPDATE users SET exchange_rate = 1.03 WHERE userid=...
  При деактивации — обратно 1.2.
  resyncUser() сбрасывает в 1.2 как часть резервного пересчёта.

Это не «курс биржи, торчащий не в том месте». Это
МАТЕРИАЛИЗОВАННЫЙ ЭФФЕКТ артефакта. Таких полей в legacy — шесть:
  users.exchange_rate      от MERCHANTS_MARK       (set 1.2 ↔ 1.03)
  users.research_factor    от SUPERCOMPUTER        (± 1)
  planets.build_factor     от ROBOT_CONTROL_SYSTEM (± 1, per planet)
  planets.produce_factor   от CATALYST             (± 0.1, all planets)
  planets.energy_factor    от POWER_GENERATOR      (± 0.15, all planets)
  planets.storage_factor   от ATOMIC_DENSIFIER     (± 0.15, all planets)

Архитектурный выбор oxsar2: материализовать эффекты в БД-поля
(быстрое чтение в тике экономики без join'ов) и иметь cron
resyncUser для восстановления консистентности. Это НОРМАЛЬНОЕ
проектное решение, не технический долг.

Что я сделал после анализа:

  А. ТЗ v1.4: §18.21 ОТОЗВАН (явно помечено, не удалено — чтобы
     в git-history остался след «был пункт, оказался ошибочным»).
  Б. §5.10 переписан: появились §5.10.1 «Персональные факторы» с
     таблицей поле→артефакт→масштаб, правила apply/revert и
     указание на resyncUser. §5.10.2 — non-stackable артефакты.
     §5.10.3 — задержка/expire/disappear события.
  В. §4 (модель данных): у полей users.exchange_rate,
     users.research_factor и planets.*_factor теперь ВАЖНО-блок

17. ИТЕРАЦИЯ 5 — ЗАКРЫТИЕ ДОЛГОВ + АРТЕФАКТЫ (M5.0)
================================================================================

Итерация после двух коррекций ТЗ. Задачи, пришедшие как «долги» и
«следующий ценный шаг»:

  1. Storage cap, обещанный ещё на sanity-check после итерации 2 —
     был захардкожен 1e18, то есть ёмкости фактически не было.
     Теперь StorageCapacity(base, level, factor) по OGame-формуле
     base * round(2.5 * exp(level * 20 / 33)) * factor.
     Покрыт тестом: level=0 → base, монотонность, линейность по
     factor (важно для ATOMIC_DENSIFIER artefact, см. §5.10.1).

Важные детали, которые вошли в дневник для статьи:

  A. Отделение чистой логики от I/O оправдалось сразу. Файл effects.go
     весь на ≤100 строк и полностью покрыт unit-тестами без dockertest
     и моков. applyChange в service.go линейно переводит FactorChange
     в UPDATE-запрос и тестируется интеграционно. Два слоя тестов —
     два уровня уверенности.

  B. Белый список полей (allowedField) — не перфекционизм, а
     реальная защита. effects.go строит UPDATE SQL через fmt.Sprintf
     (по-другому нельзя — имя колонки нельзя подставить параметром).
     Если бы YAML принимал любое строковое значение field, кто-то мог
     бы написать `field: "metal = metal + 9999, silicon"` и ломать
     экономику. Семь хардкод-значений в Go — это и безопасность, и
     документация.

  C. `expire.go` вернул меня к архитектурному спору «куда кладём
     event-хендлеры: в event/ или в доменный пакет». Выбрал: в
     доменный пакет + метод Service.ExpireEvent() event.Handler.
     Так доменный пакет сам знает про свои события, а event не
     зависит от всех доменов. Регистрация — в worker/main.go одним
     вызовом w.Register(KindArtefactExpire, artefactSvc.ExpireEvent()).
     По сравнению с ранними HandleBuildConstruction (в event/) —
     этот подход лучше для больших доменов (я переношу туда expire,
     в M5.1 добавлю delay-activate event).

  D. `resync.go` — классический пример «код должен быть idempotent».
     Дважды вызвать ResyncUser одного игрока — тот же результат.
     Это важно, потому что cron может упасть и рестартануть, и
     руками админ может нажать кнопку.

После итерации 7 (которая добавила §1.4/5.2.1/10.3 в ТЗ, но не код)

1. M0.3 — pkg/formula (Go DSL для legacy-формул)

   4 файла (context, lexer, parser, nodes) + тесты на 10+ кейсов,

   na_artefact_datasheet → artefacts_meta_generated.yml.

   pkg/sqldump — минимальный парсер phpMyAdmin-дампов, общий для
   обоих CLI (import-datasheets и import-phrases). Поддерживает

   internal/i18n/ — Bundle.Tr(lang, group, key, args) с fallback'ом
   на ru и возвратом "[group.key]" для отсутствующих ключей.
   HTTP: GET /api/i18n (список языков) + GET /api/i18n/{lang}
   (полный словарь). Доступно без auth — логин-экран тоже переводится.

4. M1 полностью переключён на DSL

   planet.Service.productionRates теперь имеет два пути:
     - productionRatesDSL (предпочтительный): читает формулы из
       ConstructionCatalog, вычисляет через pkg/formula, умеет в
       {tech=N} через researchLevels()
     - productionRatesApprox (fallback): приближение из buildings.yml,
       если construction.yml ещё не импортирован

   Formula-eval кешируется per-expr (sync.RWMutex + map), чтобы не
   парсить строку на каждый тик. Тест TestProductionRatesDSL_MetalMine
   проверяет побитовое соответствие вычисленному значению
   floor(30 * 10 * pow(1.1 + 5*0.0006, 10)).

Ключевые решения и наблюдения:

   А. Кеш формул в planet.Service, а не глобальный. Причина: если
      в какой-то будущей итерации формулы станут per-universe
      (seasonal-серверы, §19.3 ТЗ), кеш нужно будет сбросить для
      конкретного инстанса. Глобальный map сложнее инвалидировать.

   Б. evalProd/evalCons возвращают 0 на любую ошибку — не panic,
      не пропаганда ошибки наружу. Это компромисс: валидация
      формул делается в import-datasheets при их записи в YAML,
      дальше на runtime они уже гарантированно парсятся. Если
      что-то и сломается — это будет заметно через метрики
      (производство ресурсов упало в ноль), и игрок это увидит
      быстрее, чем мы поймаем ошибку в логах.

   В. YAML generated-файлы (construction.yml, ships_generated.yml
      и т.д.) предполагаются как коммитимые артефакты. Скрипт
      идемпотентен, поэтому git-diff покажет только реальные
      изменения legacy (которых в oxsar2 уже не будет, он
      зафиксирован в 2018). Это противоположность классическому
      подходу «generated-файлы в .gitignore» — в нашем случае
      мы ХОТИМ видеть изменения, и их не будет.

   Г. i18n.Tr сразу поддерживает fmt.Sprintf с args. Это совпадает
      с использованием в legacy (в PHP-коде видно, что фразы
      содержат %s / %d, которые подставляются через sprintf).
      Альтернатива — именованные плейсхолдеры {{player}} — была бы
      чище, но потребовала бы миграции всех 1489 строк. Отказ от
      неё — часть правила §19 «совместимость важнее эстетики».

   Д. Frontend-хук useTranslation() — осознанно отложил. Сначала
      довожу бэкенд, потом фронт забирает словарь одним запросом
      и использует. Пока фронт-тексты на русском захардкожены —
      будут заменены после TS-хука.

22. ИТЕРАЦИЯ 10 — RECALL + i18n ОСТАВШИХСЯ ЭКРАНОВ
================================================================================

Короткая итерация, которая закрыла два «хвоста», обещанных в конце
итерации 9: досрочный возврат флота и полный перевод UI.

1. Recall (/api/fleet/{id}/recall).

   Сначала — дизайн. Recall осмыслен только пока флот в state='outbound':
   после ArriveHandler'а либо груз уже выгружен, либо флот уже
   возвращается домой — отзывать нечего. Любое другое состояние →
   409 Conflict, без отдельного метода «force return».

   Реализация избежала новых полей в БД. Хитрость в том, что у нас
   уже запланированы два события — KindTransport (arrive, kind=7) и
   KindReturn (kind=20). Recall — это:

     1) fleets.state='returning', arrive_at=now,
        return_at=now + (now-depart_at)  -- симметричный путь назад
     2) DELETE FROM events WHERE kind=7 AND payload->>'fleet_id' = $1
        -- убираем «прибытие», чтобы ArriveHandler не разгружал флот
        в точке цели (куда он теперь не прилетит)
     3) UPDATE events SET fire_at=$new WHERE kind=20 AND ...
        -- переносим return-событие на новое время

   Почему делать через редактирование events, а не через «cancelled»
   столбец: worker использует FOR UPDATE SKIP LOCKED и берёт только
   state='wait'. wait-строка, которую мы удалили/обновили, просто
   не будет выбрана — без отдельного флага отмены. Меньше кода,
   меньше состояний, меньше ошибок.

   ReturnHandler (kind=20) уже устроен идемпотентно: он смотрит
   state ≠ 'done' и возвращает груз + корабли на src_planet.
   Отдельная ветка «возврат после recall» в нём не нужна — с точки
   зрения handler'а это обычный возврат, просто раньше запланированный.

   Тестовый сценарий (mental model):
     - send TRANSPORT, arrive_at=T+2h, return_at=T+4h
     - через 30 мин recall
     - новый return_at = T+30m+30m = T+1h
     - в T+1h воркер берёт KindReturn, списывает carry обратно на
       источник, возвращает корабли, fleet.state='done'
     - KindTransport-событие к этому моменту уже удалено

2. tf() — i18n с inline-дефолтом.

   Большинство UI-строк, которые были захардкожены по-русски
   («Построить», «Очередь», «Юнит»), не существуют в legacy
   na_phrases. Просто подсунуть ключ — значит показать пользователю
   «[Main.BUILD_BUTTON]» до тех пор, пока import-phrases не пополнит
   словарь. Это плохой UX.

   Решение: добавил в useTranslation вторую функцию — tf(group, key,
   fallback, ...args). Семантика:
     dict[lang][group][key] || dict[fallback][group][key] || fallback
   Результат всегда пропускается через sprintfLite, если есть args.

   Это позволяет писать tf('Main', 'BUILD_BUTTON', 'Построить') и
   быть уверенным: если ключа нет — пользователь увидит русский
   дефолт. Когда геймдизайн добавит ключ в ru.yml — он автоматически
   подхватится без изменения кода.

   Альтернатива была бы выделить дефолты в отдельный файл, но это
   дублирует источник истины и требует выбирать «из двух мест».
   Inline-дефолт живёт прямо у места использования — куда проще.

3. Баг в ArtefactsScreen.tsx.

   Внутри JSX обнаружилась неожиданная находка:
     <button
       typ
       e="button"
     ...>

   Атрибут «type» был разрезан переносом строки и стал двумя
   невалидными props «typ» и «e». Реакт молча игнорирует
   неизвестные свойства через jsx-runtime, так что в рантайме
   это работало «почти правильно» (кнопка была без type=«button»
   и, вероятно, как type=«submit», что в data-less компоненте
   ни к чему плохому не приводило).

   Это класс ошибок, который не ловят линтеры ESLint по умолчанию
   — нужен jsx-a11y + react/no-unknown-property, и то не всегда.
   Плюс TypeScript strict тоже не фиксирует, потому что
   расширенный тип JSX.IntrinsicElements допускает любые
   data-атрибуты. Урок: при чтении diff'а надо останавливаться
   глазами не только на логике, но и на «разорванных» словах в
   атрибутах JSX.

4. IDE-диагностики «undefined: pgx / Cannot find module 'react'».

   После каждого Edit в go/tsx-файлах IDE выплёвывает один и тот
   же набор ошибок. Это не регрессии от правок — это follow-up
   от того, что в окружении не выполнены `go mod download` и
   `npm install`. Для итерации это ожидаемый шум; в CI эти
   проверки будут зелёными после первого install-шага.

   В рабочем процессе это учит отделять «ошибки кода» от «ошибок
   окружения». Первый раз можно сверить с соседним go-файлом
   («там pgx.Tx тоже используется и тоже undefined? — да, значит
   это не про мой код»), а дальше — просто фильтр шума.

23. ИТЕРАЦИЯ 10.5 — DEVX: БЫСТРЫЕ ПЕРЕЗАПУСКИ ФРОНТА
================================================================================

Пользователь пожаловался, что пересборка фронта стала заметно долгой.
Исправил разом, чтобы дальнейший порт legacy-экранов не упирался в
инфраструктуру.

Диагноз (без замеров — по косвенным признакам):

  - dev-скрипт запускал `vite` без optimizeDeps.include → первое
    касание каждой крупной зависимости триггерило rebundle прямо в
    момент разработки. Типичный сценарий: открыл FleetScreen — Vite
    обнаружил новую зависимость, перепрогнал граф, dev-сервер завис
    на 2–4 секунды.
  - `build` был `tsc -b && vite build`. Каждый билд перекомпилирует
    типы с нуля, потому что `incremental` не был включён. Экономия —
    до 3–5 секунд на повторный build.
  - 134 .tpl → React: экранов много, но при HMR Vite пере-оценивает
    весь импортный граф App.tsx. Если все экраны импортированы
    синхронно, одна правка в BuildingsScreen может обновить граф
    даже для открытой вкладки Galaxy.
  - i18n-словарь 80kb грузится /api/i18n/ru при каждом F5 и при
    каждом HMR full-reload. staleTime у TanStack Query не защищает
    от reload — кеш живёт в памяти React, не в браузере.
  - Prod-бандл не разбит на vendor-чанки: правка features/fleet
    меняет хеш всего бандла, браузер перекачивает и react, и
    query, и router с нуля. Для deploy-preview-флоу это ~300kb
    лишних на каждую итерацию.

Что изменилось:

1. vite.config.ts

   - optimizeDeps.include: явный список react/react-dom/react-query/
     react-router/zustand/zod. Vite пре-бандлит их один раз при
     старте (в node_modules/.vite) и дальше не трогает. Если нужно
     сбросить — `npm run dev:clean` (= `vite --force`).
   - server.warmup: список «горячих» модулей (main, App, i18n,
     LoginScreen, OverviewScreen). Vite трансформирует их сразу
     после старта dev-сервера, не ждёт первого HTTP-запроса.
     Эффект — первая загрузка страницы без задержки на первую
     трансформацию App.
   - build.target='es2022': наш tsconfig target=ES2022, транспиляция
     в ES2019-уровень просто тратит CPU. Vite по умолчанию более
     консервативен.
   - build.sourcemap: command === 'serve' — true в dev, false в prod.
     Sourcemap в prod прод-билде даёт +30% объёма и время генерации.
   - build.rollupOptions.output.manualChunks: vendor-react,
     vendor-query, vendor-state. Теперь они живут в отдельных
     файлах с собственным хешом, меняющимся только при смене
     версий пакетов. Браузер кеширует их на долго.

2. tsconfig.json

   - incremental: true + tsBuildInfoFile в node_modules/.cache/tsc.
     `tsc -b` больше не сканирует весь проект — только изменённое
     с прошлого раза. На чистом CI (без кеша) — тот же результат,
     в локальном цикле — 5→1 секунда.

3. package.json

   - `dev` остался `vite` (так и было).
   - `build` теперь просто `vite build` (без tsc). Правка типов не
     ломает прод-бандл — Vite transpiles, не проверяет. Это осознанный
     trade-off: CI проверит типы отдельно через `typecheck`.
   - `typecheck` — новый: `tsc -b`. Вызывается в pre-commit/CI.
   - `build:full` — `tsc -b && vite build`. Для локального «сборки
     с проверкой», если хочется один шаг.
   - `dev:clean` — `vite --force`. Сбрасывает optimizeDeps-кеш, если
     Vite запутался (редко, но бывает после ручного обновления
     node_modules).

4. App.tsx — React.lazy

   Все экраны, кроме OverviewScreen (стартовый), теперь ленивые:
     const BuildingsScreen = lazy(() => import('./.../BuildingsScreen').then(m => ({default: m.BuildingsScreen})))
   Обёрнуты в один <Suspense fallback={<p>…</p>}>.

   Эффекты в dev:
     - Первая отрисовка App после логина не ждёт трансформации
       всех 8 экранов — только Overview.
     - При HMR правка в BuildingsScreen не инвалидирует chunk
       GalaxyScreen. Vite точечно перекидывает только задетый
       модуль.

   Эффекты в prod:
     - Начальный бандл меньше. React + Overview + React Query
       загружаются, остальные экраны подтянутся при первом клике.
     - Правка в одном экране ребилдит только его чанк, не роняя
       браузер-кеш остальных.

   Почему .then(m => ({default: m.BuildingsScreen})): у нас
   именованные экспорты (default exports запрещены стандартом
   §17.3 ТЗ кроме route-модулей). React.lazy требует default
   export, поэтому оборачиваем вручную.

5. i18n/i18n.tsx — localStorage-кеш

   Дополнили TanStack Query initialData, которая читает из
   localStorage, и queryFn, которая пишет в localStorage после
   успешной загрузки с /api/i18n/.

   Ключ: `oxsar.locale.v{N}.{lang}`, где N=LOCALE_VERSION. Поднять
   версию — инвалидация у всех клиентов. Префикс единый, чтобы
   легко очистить всё через (пример для консоли DevTools):
     Object.keys(localStorage).filter(k => k.startsWith('oxsar.locale.')).forEach(k => localStorage.removeItem(k))

   Эффект:
     - F5 / HMR full-reload: словарь из localStorage мгновенно
       (JSON.parse 80kb ~2–10 мс), UI сразу рендерится с
       правильными текстами, сеть идёт в фоне для revalidate.
     - Первый визит (пустой localStorage): как раньше — загрузка
       через API, но после — кешируется.
     - Private mode / QuotaExceeded: писать в localStorage падает
       в try/catch, поведение деградирует до прежнего (как если
       бы кеша не было).

Что НЕ сделали:

  - Не стали подключать `@vitejs/plugin-legacy`: нам не нужны
    IE11/старые браузеры.
  - Не вкручивали `swc` вместо `esbuild` для React: плагин
    @vitejs/plugin-react и так быстрый; swc добавил бы ещё одну
    зависимость ради ~30% улучшения трансформации, что меркнет
    на фоне lazy-экранов.
  - Не меняли tsc strict-флаги: медленный тип-чекинг — цена
    правильной безопасности (§17.3 ТЗ запрещает ослаблять strict).
    Вынесли tsc из hot-path вместо ослабления.

Ожидаемый эффект (порядки, не замеры):

  - Холодный старт dev-сервера: ~4 с (pre-bundle) → остаётся
    примерно так же, но предсказуемее (не добавит внезапных
    rebundle по ходу работы).
  - HMR-перерисовка после правки одного экрана: 2–3 с → 0.3–0.8 с.
  - F5 в браузере: 1–2 с (из-за загрузки i18n) → ~0.2 с.
  - `npm run build`: 15 с → 5 с (убрали tsc из hot-path).

Урок на будущее: каждый раз, когда «долго пересобирает», стоит
спросить: это сам bundler, это typecheck, или это сеть во время
загрузки ресурсов? Три разных слоя, три разных решения. Вложить
5 минут в диагностику окупается в 10× на итерациях.

24. ИТЕРАЦИЯ 11 — DOCKER-ПОЛНЫЙ СТЕК И ФАКТИЧЕСКИЙ ЗАПУСК
================================================================================

Рубеж, за который долго не хотелось заходить: «у нас всё работает на
бумаге» → «я лично запустил игру целиком и сделал первый API-вызов».

1. Docker-стек (6 сервисов).

   До: docker-compose поднимал только postgres + redis, остальное
   нужно было запускать локально через `make backend-run`,
   `make worker-run`, `npm run dev`. Это требовало установленных
   go, node и goose на хосте.

   Стало: backend/Dockerfile (multi-stage: golang:1.23-alpine для
   сборки, alpine:3.20 для runtime, оба бинаря /app/server и
   /app/worker), frontend/Dockerfile (node:22-alpine с `vite --host
   0.0.0.0`), deploy/Dockerfile.migrate (отдельный контейнер с goose
   CLI), обновлённый deploy/docker-compose.yml с шестью сервисами.

   Запуск одной командой:
     docker compose -f deploy/docker-compose.yml up --build -d

   Поймать конфликт портов.

   На рабочей машине был уже занят 5432/6379/8080 другим проектом
   (другой docker-compose с postgres/redis/backend на тех же портах).
   Решение: смена проброса наружу на 5433/6380/8081 без изменений в
   коде — сервисы внутри docker-сети находят друг друга по именам
   (backend → postgres:5432, а не localhost). Это, кстати, стандартная
   уличная магия docker networking, но она требует, чтобы в backend
   не был вшит «localhost» в конфиге.

   Три подводных камня, которых не было в теории:

   а) goose v3.22.1 с GOOSE_DRIVER env + позиционным driver
      конфликтует: парсер аргументов путает URL и command. Симптом:
      `goose run: "postgres://...": no such command`. Исправлено
      убранием ENV в Dockerfile.migrate. Запоминать: один источник
      истины для CLI-опций — либо env, либо позиционные.

   б) В compose для worker я сначала написал `command: ["/app/worker"]`,
      но ENTRYPOINT в backend Dockerfile `= ["/app/server"]`. Docker
      не заменяет ENTRYPOINT через `command`, а СКЛЕИВАЕТ их как
      аргументы → получилось `/app/server /app/worker` (worker стал
      arg'ом, запустился server). Исправлено заменой на `entrypoint:
      ["/app/worker"]`. Запоминать: ENTRYPOINT и CMD — не одно и то же;
      если нужен другой бинарь, переопределяй entrypoint.

   в) vite proxy захардкоженный на localhost:8080 не работал
      внутри docker (там backend — это имя сервиса, не localhost).
      Решено env-driven target-ом: `VITE_PROXY_TARGET`, с fallback
      на localhost для локального запуска без docker. Один и тот же
      vite.config.ts в двух режимах.

2. i18n-данные (import-phrases против na_phrases.sql).

   Параллельно запустил давно валяющийся CLI — cmd/tools/import-phrases.
   Результат: configs/i18n/ru.yml (1463 ключа, 23 группы) и en.yml
   (заготовка). Эти файлы копируются в backend-образ на стадии build.
   В логах сервера сразу: «i18n loaded: langs=[en,ru]». WARN
   «i18n not loaded» ушёл.

   Это делает t() в компонентах первичным, а tf() — fallback'ом
   для пропущенных ключей. То есть инженерно путь чист: legacy
   переводы — primary, inline-русский — gap filler.

3. Repair Factory (M2-финал, DISASSEMBLE MVP).

   Первое новое здание/очередь за долгое время. Batch-семантика
   (в отличие от legacy per-unit), потому что без боя нет
   damaged_count и «пересобирание события после каждого юнита» —
   лишняя сложность. Итоговый баланс тот же (earn = return - req =
   base*0.7), просто всё разом.

   В коде:
     migrations/0008_repair_queue.sql   — очередь с mode='disassemble'
     backend/internal/repair/service.go — Enqueue + ceil10 формулы
     backend/internal/repair/events.go  — DisassembleHandler (kind=51,
                                          совпадает с legacy)
     configs/buildings.yml              — repair_factory (id=100)

   Регистрация handler'а в cmd/worker/main.go потребовала
   «полноценной» сборки repair.Service с planet/requirements —
   даже хотя handler использует только db. Альтернатива — сделать
   отдельный экспортируемый helper без зависимостей. Я выбрал
   путь совместимости с server: один и тот же Service живёт в двух
   процессах. Это ~4 лишних строки, но нет дублирующего кода.

4. End-to-end тест.

   Ручной smoke, не юнит. План:
     а) register → получить TOKEN + стартовую планету
     б) SQL-seed: metal/silicon/hydrogen = 100000, repair_factory=1,
        shipyard=2 (requirement small_transporter), research
        combustion_engine=2 (requirement small_transporter),

26. ИТЕРАЦИЯ 13 — M4.0: БОЕВОЙ ДВИЖОК, КАРКАС РАУНДА
================================================================================

Первый шаг в M4 — самая большая невспаханная территория проекта.
Приняли поэтапный план: M4.0 закладывает каркас, M4.1+ добавляет
шильды, rapidfire, ballistics, golden-тесты против Java. Сегодня —
только M4.0.

1. Что делает engine.go теперь.

   Поверх существующей сигнатуры Calculate(Input) → Report:
     * clone входных сторон (не мутируем вход);
     * for round = 0..Rounds:
         - atkSnap := clone(atk), defSnap := clone(def)
         - shootAtSides(atkSnap, def)   // атакующие бьют по защитникам
         - shootAtSides(defSnap, atk)   // наоборот
         - compactSides (юниты с quantity=0 сохраняются в списке, чтобы
           summarize читал по индексу, но не влияют на дальнейшие
           раунды: в shootAtSides фильтруем quantity > 0)
         - RoundTrace, ранний выход при полной победе;
     * summarize → SideResult[], decideWinner.

   shootAtSides упрощённый:
     - собирает активные целевые unitList-ы, суммирует вес
       (weight = 2^Front × Quantity, как в Java getTurnWeight);
     - для каждого shooter-unit: shots = Quantity, attack =
       Unit.Attack[0] (только normal-канал);
     - распределяет shots по целям пропорционально weight, min 1;
     - применяет damage = attack × shots к total-shell цели,
       пересчитывает Quantity = floor((totalShell - damage) / unitShell).

2. Семантический вопрос: «когда стреляет вторая сторона».

   Java хранит startTurnQuantity в Units и все полу-раунды считают
   по нему. Без этого первое попадание атакующих обнуляет quantity
   защитников, и во втором полу-раунде защитникам нечем стрелять
   (а это баг — в OGame обе стороны всегда имеют шанс нанести урон
   в один раунд).

   Я не стал заводить отдельное поле startTurnQuantity — вместо этого
   клонирую snapshot стрелков перед каждым полу-раундом и использую
   его для подсчёта shots. Проще, меньше изменений типов. В M4.1,
   когда появятся shield-regen и turnShell, логично вернуть
   startTurnQuantity как явное поле — оно понадобится для формул.

3. Почему нет щитов в M4.0.

   Java-формула processAttack сначала вычисляет shieldDestroyFactor,
   затем расщепляет shots на shield-shots и shell-shots. Добавлять
   это одним куском с round-loop → тесты на shell-only не получат
   стабильного базиса.

   Подход: сначала простая модель, потом слой shield поверх с
   assertion «без shield поведение совпадает с M4.0-сценарием».
   Когда придёт M4.1, ранее написанные тесты станут регрессионными
   — будут ловить деформации базовой математики.

4. Тесты.

   Шесть сценариев в engine_test.go:
     * AttackersWin — сильные атакующие за 1 раунд уничтожают
       слабых защитников, winner=attackers, rounds=1.
     * DefendersWin — зеркало, winner=defenders.
     * Draw — обе стороны взаимно уничтожаются за счёт снимков.
     * Deterministic — одинаковый Input даёт бит-равный Report.
     * LostResources — ровно 10 потерянных × 20000 metal = 200000.
     * InvalidInput — пустой Attackers возвращает ErrInvalidInput.

   Все passed. Тесты — юнитные, self-consistent: они не сверяют с
   Java. Это правильно для M4.0; golden-тесты придут в M4.3.

5. Попутные правки.

   а) types.go получил json-теги для всех полей (snake_case). Без
      них POST /api/battle-sim возвращал 400 «unknown field user_id»
      — OpenAPI обещал snake_case, а Go по умолчанию пишет
      CamelCase.

   б) Обнаружен настоящий баг в planet/service.go: evalProd НЕ
      обрабатывал resource="energy", поэтому энергия по DSL-пути
      всегда была 0 → energyRatio=0 → шахты ничего не производили
      по DSL. В проде этот код не исполнялся (configs/construction.yml
      не существует), но TestProductionRatesDSL_MetalMine падал
      долгое время. Фикс — добавить case "energy" в switch. Тест
      прошёл: metalPerSec=0.22 = 799/3600.

      Урок: «тест падает, но в проде не используется» — это запах,
      а не «ок». Баг таился до того момента, как DSL не включат
      (после прогона import-datasheets).

   в) battle-sim end-to-end в docker работает: POST с 10 vs 10,
      winner=attackers, rounds=1, quantity_end видно.

Статистика итерации 13:
   backend: engine.go переписан (+30 строк), types.go +json-теги,
            engine_test.go 6 тестов, planet/service.go фикс.
   frontend: не менялся.
   miграции: 0.
   API: тот же /api/battle-sim (schema не изменилась, тэги — то же поле).

27. ИТЕРАЦИЯ 14 — M4.1: ЩИТЫ + MULTI-CHANNEL
================================================================================

Второй шаг в M4. M4.0 был про каркас («раунд, выстрел, квантити»);
M4.1 добавляет механику щитов, которая в OGame ключевая — без неё
линкоры неотличимы от транспортов, а лёгкие истребители не тают о
крупные корабли.

1. Сдвиг модели: публичные Unit vs runtime unitState.

   В M4.0 я мутировал Side.Units напрямую. Это работало, пока
   состояние было «quantity и всё», но turnShell/turnShield — это
   явно state одного раунда, и засорять ими публичный Unit было бы
   плохо.

   Решение: ввёл приватный unitState с полями turnShell, turnShield,
   quantity, primaryChannel + ссылка tmpl на исходный Unit. Вся
   мутация идёт через unitState, summarize читает его через
   toSides() (метод).

   Попутно поймал тонкий баг: поле battleState.sides []*sideState
   конфликтует по имени с методом sides() (Go запрещает). Переименовал
   метод в toSides — простое, но показательное напоминание, что
   «одинаковые имена для поля и метода» — не просто стиль, это
   compile error.

2. Модель щитов.

   Java processAttack очень длинный: он делит выстрелы на
   shield-shots и shell-shots, считает shieldDestroyFactor,
   обрабатывает shield regen внутри раунда. M4.1 упрощает это до
   пула:

     pool = attack × shots
     if attack < shield / 100  →  выстрелы полностью абсорбируются
                                  щитом, в корпус НИЧЕГО не идёт;
     else:
       shieldAbsorb = min(pool, turnShield)
       turnShield -= shieldAbsorb
       pool -= shieldAbsorb
       turnShell -= pool

   Плюсы:
     * Аккуратное поведение ignoreThreshold — «лёгкие пулемёты
       не пробивают тяжёлые щиты никогда» — ключевая OGame-механика.
     * Математика остаётся линейной, легко тестируется.

   Минусы (осознанные):
     * Нет «одинокий щит защищает от серии слабых выстрелов без
       просадки» — у нас щит постепенно тает. В M4.2 с rapidfire
       это станет заметнее, тогда и выправлю.
     * Нет partial regen (shieldDamageFactor). Пока — 100% regen
       после каждого раунда.

3. Multi-channel: primaryChannel.

   Attack[3] и Shield[3] были в types.go с M0. M4.1 впервые их
   использует. Подход: для каждого unit-stack выбираем один primary
   channel (с max Attack). Вся пачка стреляет этим каналом, щит
   цели оценивается по тому же каналу.

   Полноценное multi-channel распределение (один выстрел тратит
   все три канала независимо) — это сложный кусок Java, где laser
   ещё и усиливает/обходит специальные щиты. Это M4.2+.

4. Shield regen.

   В начале каждого раунда (кроме первого — там инициализация в
   newState) turnShield сбрасывается до quantity × shield[primary].
   Это важнейшая оборонительная механика: в M4.1 я пишу тест, где
   10 атакующих ровно сносят щит за раунд, а следующий раунд —
   щит снова полон. Итог: никто не гибнет за 3 раунда → draw.

5. Тесты.

   Старые 6 (M4.0) остались зелёными без изменений — это было целью.
   Добавилось 3 новых:

   а) ShieldIgnoreThreshold: attack=50, shield=10000 (threshold=100).
      Никто не гибнет за 6 раундов. Щит непробиваем.

   б) ShieldPierced: attack=500 × 10 shots = 5000 pool; защита
      100 shield + 300 shell × 10 = 4000 total. Одним раундом
      защитники стерты.

   в) ShieldRegensBetweenRounds: pool ровно равен totalShield,
      корпус не страдает, следующий раунд — shield восстановлен,
      3 раунда draw.

   Итого 9 тестов. Все прошли с первого запуска (редкий кейс —

28. ИТЕРАЦИЯ 15 — M4.2: RAPIDFIRE И BALLISTICS/MASKING
================================================================================

Третий шаг в M4. M4.1 закрыл щиты; M4.2 — ещё два слоя OGame-механик,
которые делают бой не-тривиальным: rapidfire (скорострельность для
пары shooter×target) и ballistics vs masking (точность против
скрытности).

1. Важное уточнение про RNG.

   В предыдущей итерации я записал «M4.2: ballistics/masking с RNG-roll».
   Оказалось, неправильно: Java использует детерминированную формулу,
   не roll. RNG в processAttack не участвует вообще. Он нужен позже
   для debris/moon-chance/tie-break, но не для «попал-промахнулся».

   Формула (из Units.processAttack в Java):

     maskingEffect = max(0, masking - ballistics)
     factor = 1 - 1 / (1 + maskingEffect * 2 / 10)
     missed = floor(shots * factor)
     shots -= missed

   Семантика: masking берётся у ЦЕЛИ (у защитника), ballistics —
   у СТРЕЛКА. Это правильно («я прячу флот, враг сквозь помехи ищет»).

2. Rapidfire.

   Input.Rapidfire[shooter_unit_id][target_unit_id] = multiplier.
   Существовал с M0, но не использовался. Теперь:

     rawShots = round(shooter.quantity * portion_by_weight)
     shots = rawShots * rf[shooter.id][target.id]

   Если rf < 1 или таблицы нет → rf = 1. Это совпадает с Java
   (Math.max(1, ...)).

   Источник истины — configs/rapidfire.yml (уже загружен в каталог,
   но пока в сам Calculate не передаётся — для тестов передаю
   напрямую через Input.Rapidfire). Интеграция с runtime-каталогом
   пойдёт в M4.4, когда Fleet ATTACK реально начнёт использовать бой.

3. Баг, пойманный тестом.

   TestRapidfire_TriplesDamage сначала упал: показал kills=1 при
   rf=3 вместо ожидаемых 3. Час недоумения — и оказалось, что
   snapshot() в battleState не копирует поле rapidfire (и tech
   сторон тоже). А shootAtSides берёт rf именно из shooters.rapidfire
   (снапшота), потому я читал пустую nil-таблицу.

   Фикс — одна строка: копировать rapidfire в snapshot(). После
   этого и rapidfire, и masking-тесты прошли.

   Урок: если state разделён на «main» и «snapshot» с глубоким
   клонированием — любое новое поле должно копироваться явно.
   Это не приводит к краш-ошибке, а дают «тихую» потерю данных.
   Написать дополнительный тест на copy-completeness как следующий
   шаг (будет регрессионной защитой для M4.3+).

4. Ещё один баг-от-модели: «частичный урон = полный юнит мёртв».

   Test TestMasking_ReducesDamage я сначала написал на attack=100,
   масштаб 10 юнитов × 1000 shell = 10000 totalShell. Без masking
   1000 damage → floor(9000/1000) = 9 → 1 kill. С masking=10
   (factor=0.667, shots=4) 400 damage → floor(9600/1000) = 9 →
   тоже 1 kill.

   Проблема не в masking, а в моей модели commitDamage:
   любой частичный урон убивает последнего юнита целиком. В Java
   эту ситуацию покрывает ablation (damaged_count + ShellPercent).
   У меня оно на M4.3+.

   Пока — просто подогнал масштабы теста: attack=500, baseline
   5 kills, с masking=10 получается 2 kills. Ablation отложен как
   явный TODO.

5. Тесты.

   9 старых (M4.0/M4.1) остались зелёными без изменений.
   Добавилось 3 новых:

   а) Rapidfire_TriplesDamage: baseline=1 kill, с rf=3 → 3 kills.
   б) Masking_ReducesDamage: masking>ballistics → kills меньше.
   в) Ballistics_OffsetsMasking: ballistics==masking → эффект
      отсутствует, kills как в baseline.

   Итого 12 тестов, все проходят.

29. ИТЕРАЦИЯ 16 — M4.3: ABLATION (УПРОЩЁННЫЙ M4.3 БЕЗ GOLDEN-ТЕСТОВ)
================================================================================

Четвёртый шаг в M4. По решению пользователя golden-тесты против Java-
движка пропущены — без Java-инструментов на хосте их запуск требует
отдельной инфраструктуры (jar, JavaRandom-адаптер, test-harness),
а ценность «паритета с legacy» меньше, чем «продолжать двигаться».
Оставили из M4.3 только ablation — самую заметную для игрока
поведенческую правку.

1. Что такое ablation.

   Раньше (M4.2) commitDamage делал грубо: `floor(turnShell / shell)`.
   Это значит, что любой частичный урон по последнему юниту убивал
   его целиком. Игрок, нанёсший 499 из 1000 shell, видел, что весь
   юнит всё равно уничтожен — контринтуитивно.

   Ablation в Java хранит «частично ранен» как отдельный state:
   damaged_count = 1 + shell_percent = остаток × 100. Следующий
   раунд этот юнит воюет как обычно (по attack), но его shell
   понижен до shell × shell_percent/100.

2. Реализация.

   unitState получил два поля: damaged (0..quantity) и shellPercent
   (0..100). Helpers:
     * totalShell(shell, qty, damaged, pct) —
       (qty-damaged) × shell + damaged × shell × pct/100
     * clampDamaged, clampPercent — защита от мусорного ввода.

   commitDamage переписан на три ветки:
     * fullRem >= quantity  → никто не умер, turnShell
                              нормализуется к текущему (qty,damaged,pct);
     * fullRem < quantity, remainder > 0
                            → quantity = fullRem + 1, damaged = 1,
                              shellPercent = remainder/shell × 100;
     * remainder == 0      → точная граница, fullRem юнитов, без damaged.

   toSides теперь пробрасывает Damaged и ShellPercent в summarize.
   Summarize уже умел это поле (см. Unit/UnitResult) — нужно было
   просто подключить.

3. Поведение в серии раундов.

   turnShield восстанавливается regen каждый раунд — это было в M4.1.
   turnShell НЕ восстанавливается: если damaged-юнит получил ранение
   в раунде 1, во втором раунде его shell пониженный, и небольшой
   дополнительный урон уже может его добить.

   Это и тестирует TestAblation_DamagedCarriesOverAndDies:
     1 защитник, shell=1000. Round 1: pool=500 → damaged=1, pct=50.
     Round 2: pool=500 → turnShell=0 → dead.

4. Тесты.

   Старые 12 остались зелёными без правок — выборка «полный kill
   или нет урона» не пересекается с новой логикой.

   Добавлены 2:
     а) AblationPartialHitLeavesDamagedUnit: pool=500, totalShell=10000,
        после раунда quantity=10, damaged=1, shellPercent=50.
     б) AblationDamagedCarriesOverAndDies: 2 раунда × 500 pool добивают
        единственного защитника, winner=attackers за 2 раунда.

   Всего 14 тестов, все зелёные.

5. Docker smoke.

   POST /api/battle-sim с partial-hit сценарием отдаёт в JSON:
     quantity_end = 10, damaged_end = 1, shell_percent_end = 50.0

   UI/ShipyardScreen/FleetScreen пока этого не отображают — это
   следующий экран («battle report»), который придёт вместе с M4.4.

6. Осознанное упрощение (на будущее).

   Во входном Input.Side.Units уже есть Damaged и ShellPercent — я
   протянул их через newState через clampDamaged/clampPercent. Это
   пригодится, когда Fleet будет нести повреждённый флот в бой
   (например, после рейкалла, если ввести damaged при обстреле в
   полёте — в legacy такое есть).

   Repair Factory (disassemble) сейчас работает только со здоровыми
   юнитами (см. backend/internal/repair/service.go). REPAIR-режим
   ждал именно damaged_count — теперь его можно делать. Но это
   отдельная итерация после M4.4.

Статистика итерации 16:
   backend/internal/battle: engine.go +ablation + helpers (~70 строк),
              engine_test.go +2 теста + math import (~80 строк).
   миграции: 0.
   API: без изменений (damaged_end/shell_percent_end были в Report
        с M0, просто теперь реально используются).

Что дальше:

32. ИТЕРАЦИЯ 19 — M4.4c: REPAIR-РЕЖИМ В REPAIR FACTORY
================================================================================

Завершающая итерация серии M4: цикл «бой → повреждённый флот →
ремонт на фабрике» замкнут.

1. Формула стоимости.

   Из legacy ExtRepair::setRepairUnitRequirements:
     struct_scale = 0.1 × (100 - shell_percent) / 100
     required_{m,s,h} = ceil(base × struct_scale / 10) × 10
     required_time = buildTime(base×0.1, base×0.1, mode)

   Семантика: «только что подбит» (shell_percent=99) → ремонт
   почти бесплатный. «Раздавленный в пыль» (shell_percent=1) —
   ремонт почти как строительство заново. Это интуитивно
   правильно (если корпус почти целый — мелочь).

2. Реализация — EnqueueRepair в internal/repair/service.go.

   Читаем ships.damaged_count + shell_percent FOR UPDATE, считаем
   стоимость, списываем ресурсы, НЕ трогаем ships.count (юниты
   остаются в стоке с damaged>0 до срабатывания события). Ставим
   event KindRepair=50, который при finish обнуляет damaged.

   Batch-семантика: «кнопка Починить» чинит всех damaged этого
   unit_id. Частичный ремонт не поддерживается — усложняет
   shell_percent без выигрыша в UX.

3. RepairHandler — тривиальный.

   Выполняет `UPDATE ships SET damaged_count=0, shell_percent=0
   WHERE planet_id=$1 AND unit_id=$2`. Если за время ремонта
   произошёл второй бой и damaged вырос — наша модель считает
   «сбросит всё подряд». В идеале handler должен сохранять только
   разницу (delta=count при enqueue), но это усложнение ради
   edge-case'а. Оставляем как есть.

4. Отдельный endpoint для damaged.

   Сначала хотел расширить shipyard.Inventory (добавить damaged в
   ответ), но ShipyardScreen его уже потребляет и лишнее поле
   туда не нужно. Поэтому — separate GET /api/planets/{id}/
   repair/damaged, возвращает только ships с damaged_count > 0.
   Чистая изоляция фич, RepairScreen знает только про своё.

5. UI.

   RepairScreen получил отдельную секцию «Повреждённые корабли»
   над disassemble-блоком. Таблица: Юнит / В наличии / Повреждено
   / Броня (%) / Действие [Починить]. Клик — disable-кнопка на
   время pending, на success — invalidate queue + damaged +
   inventory + planets.

6. E2E smoke.

   Dave после атаки carol'ы (итерация 17) имел
   ships(31)=7/damaged=1/shell_percent=23.25.

     - seed: repair_factory=1, shipyard=2, research(combustion)=2,
       research(impulse)=1, metal/silicon/hydrogen=100k/100k/50k.
     - GET /damaged → [{unit_id:31, count:7, damaged:1, shell_pct:23.25}]
     - POST /repair unit_id=31 → queue с mode=repair, count=1,
       per_unit_seconds=1.
     - res_log: repair_cost -240/-80/0
       (base light_fighter 3000/1000/0,
        struct_scale=0.1×76.75/100=0.07675,
        required = 230.25 → ceil10=240, 76.75 → ceil10=80. ✓)
     - Через 1 сек worker: damaged=0, shell_percent=0, count=7
       (юнит снова здоров).
     - queue.status=done.

   Весь жизненный цикл «атака → damaged → repair → healthy»
   работает в docker-окружении.

7. Осознанные упрощения.

   а) Defense не чинится (таблица defense не хранит damaged_count
      в legacy-схеме). Если оборона уничтожена — строится заново.

   б) Частичный ремонт (починить N из M damaged) не поддерживается.
      UX-простота vs эмуляция legacy — выбрал простоту.

   в) Если к моменту срабатывания KindRepair прилетает второй бой
      и damaged_count растёт — handler сбросит его целиком. Бонус
      игроку, но минимальный (новый бой должен точно уложиться
      в ~1-2 секунды ремонта).

   г) Стоимость считается по shell_percent в момент enqueue. Если
      между enqueue и finish юнит дополнительно повредился,
      стоимость не пересчитывается — платит то, что считалось
      изначально.

33. ИТЕРАЦИЯ 20 — RAPIDFIRE FIX + RECYCLING (A + B ОДНОЙ ИТЕРАЦИЕЙ)

Два трека одной итерацией. Оба небольшие, но вместе закрывают

A. Rapidfire fix (тривиальный).

   В M4.4a rapidfireToMap возвращала nil — AttackHandler проходил
   в бой с rf=1 для всех пар. Это занижало урон тяжёлых юнитов
   (cruiser → light_fighter rf=6, deathstar → мелочь rf=100-250).

   Проверка каталога показала: configs/rapidfire.yml уже имеет
   формат map[int]map[int]int (ключи — unit_id). В
   RapidfireCatalog он грузится как есть. Так что конвертер
   name→id не нужен — просто возвращаю cat.Rapidfire.Rapidfire.

   E2E: 10 cruiser (attack=400, rf=6 → LF) vs 10 light_fighter
   (shell=4000). Без rf pool=4000/раунд = 10×400 = exactly
   totalShell → 1 kill/раунд (или draw). С rf=6 pool=24000/
   раунд → attackers win за 2 раунда, все defenders уничтожены,
   loot 3555 metal+silicon. Баланс значительно динамичнее.

B. RECYCLING + debris_fields.

   1. Миграция 0010_debris_fields.

      Отдельная таблица (galaxy, system, position, metal, silicon,
      last_update), PK по координатам. Один field на три числа —
      UPSERT при повторных боях.

      Привязки к планете нет: debris может лежать на пустой
      орбите (после полного уничтожения цели). Legacy хранил это
      как «фиктивные планеты» на position=16, в современном
      стеке — отдельная таблица.

   2. AttackHandler: debris вместо «всё в loot».

      Debris = 30% metal+silicon от стоимости уничтоженных SHIP'ов
      (обеих сторон, не только defender'а). Defense не даёт debris —
      OGame-правило.

      calcDebris: итерирует по UnitResult обеих сторон, для каждого
      уничтоженного юнита (QuantityStart - QuantityEnd) достаёт
      Cost из каталога Ships, умножает на 30/100. Defense-юниты
      фильтруются по defenseIDs (их UnitID из defenderDefense
      stacks).

      После расчёта — UPSERT в debris_fields, запись debris_metal/
      silicon в battle_reports. Loot как и был (50% ресурсов
      планеты → carry).

   3. RecyclingHandler (KindRecycling=9).

      Приходит флот, берёт from debris_fields FOR UPDATE (чтобы
      два recycler'а не вычерпали больше debris, чем там есть).
      Считает cargo cap живых fleet_ships × Cargo spec из каталога.
      Забирает min(debris, freeCargo) пропорционально (metal : silicon
      как есть в debris). Остаток debris остаётся для следующих.

      state='returning' — стандартный KindReturn положит carry
      на src_planet.

   4. fleet.Send / HTTP: добавлен mission=9 в valid list.

   5. E2E smoke в docker.

      Этап 1 (ATTACK создаёт debris):
        - carol: 10 cruiser + 10 small_transporter.
        - dave: 50 light_fighter (shell × 50 = 200000).
        - mission=10 → fleet send, force-arrive.

          c rf=6 даёт 24000/раунд × 6 = 144000, не хватает до

      Этап 2 (RECYCLING):

        - state=returning.

   6. Осознанные упрощения.

      а) В ATTACK считаем debris и у атакующего, и у защитника
         (суммарно). Legacy делает так же.
      б) defense-юниты не дают debris (исключаем по unit_id из
         defenderDefense). В legacy — та же логика.
      в) Loot (50% ресурсов цели) и debris (30% стоимости кораблей)
         независимы. Если и то и то забираем — грузоподъёмность

      г) RECYCLING карго не предел: если recycler прилетел с грузом

      д) Debris не привязан к is_moon (orbit-field один на

Реализация (два файла + изменения в воркере и сервере):

  backend/internal/score/service.go (~340 строк):
    - RecalcUser(ctx, userID): пересчитывает b/r/u/a/total points
      по формулам из legacy PointRenewer.class.php:
        b_points = 0.0005 × Σ cost(level 1..N) по зданиям всех планет
        r_points = 0.001  × Σ cost(level 1..N) по исследованиям
        u_points = 0.002  × Σ count × cost по ships+defense всех планет
        a_points = Σ achievement.points (из таблицы achievements)
        total    = b + r + u (a не входит в total — как в legacy)
      Индексы по ID строятся в памяти из map[string]Spec, итерация
      по строкам БД — один запрос на тип данных.
    - RecalcAll(ctx, log): обходит всех активных (umode=false) и
      вызывает RecalcUser. Ошибки одного игрока не роняют цикл.
    - Top(ctx, type, limit): лидерборд, ORDER BY col DESC, limit ≤ 200.
      type = "total"|"b"|"r"|"u"|"a", неизвестный → "total".
    - PlayerRank(ctx, userID, type): позиция в рейтинге через
      COUNT(*)+1 WHERE col > playerPts.

  backend/internal/score/handler.go (~50 строк):
    - GET /api/highscore?type=...   → Top()
    - GET /api/highscore/me?type=...  → PlayerRank()

  backend/cmd/server/main.go:
    - scoreSvc = score.NewService(db, cat)
    - scoreH = score.NewHandler(scoreSvc)
    - маршруты pr.Get("/highscore", ...) и pr.Get("/highscore/me", ...)

  backend/cmd/worker/main.go:
    - горутин с time.Ticker(5 * time.Minute) → scoreSvc.RecalcAll()
    - запускается параллельно event-loop, завершается по ctx.Done()

Ключевые технические решения:
  1. repo.Exec не имеет Query/QueryRow/Exec напрямую (только InTx +
     Pool()). Все прямые запросы идут через s.db.Pool().Query() —
     паттерн остальных сервисов (achievement, message, artefact).
  2. Пересчёт — materialized, не on-the-fly: очки читаются из users,
     не вычисляются при каждом GET /highscore. Это O(1) запрос.
  3. Каталоги (map[string]Spec) перекладываются в map[int]Spec один
     раз на вызов RecalcUser — не на каждую строку БД.
  4. RecalcAll — периодический batch, НЕ в каждом domain-handler'е.
     Это упрощение: между завершением постройки и обновлением очков
     есть задержка до 5 минут. Приемлемо для лидерборда (не real-time
     балансировка). Если нужна точность — добавить вызов RecalcUser
     в building/research/shipyard completion handler'ах.
