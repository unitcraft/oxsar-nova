# ADR-0001 (приложение): сравнение Go с альтернативами

- Status: Informational (приложение к [ADR-0001](0001-stack-choice.md))
- Date: 2026-04-25

## Зачем этот документ

[ADR-0001](0001-stack-choice.md) фиксирует решение «бэкенд на Go» одной строкой
без разбора альтернатив. Этот документ — ретроспективный анализ для будущих
участников: почему именно Go, в каких нишах он проигрывает другим языкам, и
когда имеет смысл точечно вынести модуль на другой язык (например, бой-движок).

Решение по основному стеку не пересматривается. Документ — справочный.

## Альтернативы, которые рассматривались

### Elixir + Phoenix

**Сильные стороны для oxsar-nova:**

- **Акторная модель ложится на домен**. Каждый летящий флот, каждая планета,
  каждый бой — отдельный процесс BEAM. Не нужно тянуть state через БД на
  каждый тик: процесс хранит состояние в памяти, БД используется как снапшот.
- **OTP** — `GenServer`, `Supervisor`, `Registry` — стандартная либа «как
  писать отказоустойчивые системы». Падает один флот — супервизор поднимает
  только его.
- **Распределённость из коробки** — процессы на разных нодах общаются
  одинаково, горизонтальное масштабирование почти бесплатно.
- **Phoenix LiveView** — реактивный UI без JS через WebSocket-diff. Для игры
  с real-time обновлениями (флот прилетел, ресурс тикнул) радикально проще,
  чем React + TanStack Query + ручные WS-подписки.

**Минусы для нашего случая:**

- Бой из Java — императивный код на мутабельных массивах. Перевод в
  функциональный стиль = почти переписать, а golden-тесты против jar-а
  требуют бит-в-бит совпадения.
- Сообщество в ~10× меньше Go.
- AI-ассистенты пишут Elixir заметно хуже, чем Go (для проекта, где код
  пишется в основном AI, это серьёзный минус).
- Нет аналога sqlc такого же уровня — Ecto это ORM, другой подход.

**Пример (флот летит и обрабатывает прилёт):**

```elixir
defmodule Oxsar.Fleet.Transport do
  use GenServer

  def init(%Fleet{state: "outbound"} = fleet) do
    ms = DateTime.diff(fleet.arrive_at, DateTime.utc_now(), :millisecond)
    Process.send_after(self(), :arrive, max(ms, 0))
    {:ok, fleet}
  end

  def handle_info(:arrive, fleet) do
    {:ok, updated} = Repo.transaction(fn ->
      case Planet.find_at(fleet.dst_coords) do
        {:ok, target} ->
          Planet.credit_resources!(target, fleet.carried)
          ResourceTransfer.log!(fleet, target)
        :not_found -> :skip
      end
      Fleet.mark_returning!(fleet)
    end)

    PubSub.broadcast(fleet.owner_id, {:fleet_arrived, updated})
    Process.send_after(self(), :return, updated.return_ms)
    {:noreply, updated}
  end
end
```

### Rust

**Сильные стороны:**

- Производительность и память на уровне C, без GC-пауз. На VPS — в 2-3× меньше
  RSS, чем у Go.
- Типовая безопасность на порядок выше Go: суммарные типы (`enum` с payload),
  `Option<T>` вместо nil-паник, `Send + Sync` маркеры исключают гонки данных
  на уровне компилятора.
- Идеально для порта боя из Java: те же массивы и циклы, плюс гарантии
  отсутствия UB. Бит-в-бит детерминизм через `rand_chacha`.
- `sqlx` — compile-time проверка SQL против живой БД (или offline-кеша
  схемы). Не настолько удобен как `sqlc`, но близко.
- Зрелые async-стек: `tokio` + `axum` + `sqlx` + `tokio-tungstenite`.

**Минусы:**

- Скорость разработки в 1.5–2× медленнее Go: borrow checker, лайфтаймы,
  трейты требуют времени даже у опытных. Для одиночной разработки + AI это
  ощутимый налог.
- Async-сложности: `async fn` + лайфтаймы + `Send` bounds — легко упереться
  в загадочные ошибки компилятора.
- Компиляция: ~50k строк incremental собирается 30-60 сек против 2-5 сек у Go.
- AI-ассистенты пишут Rust хуже Go (но лучше Elixir).

**Пример:**

```rust
pub async fn handle_arrive(pool: &PgPool, event: ArriveEvent)
    -> Result<(), FleetError>
{
    let mut tx = pool.begin().await?;

    let fleet: Option<Fleet> = sqlx::query_as!(
        Fleet,
        r#"SELECT id, state as "state: FleetState", dst, carried, owner_id
           FROM fleets WHERE id = $1 FOR UPDATE"#,
        event.fleet_id
    ).fetch_optional(&mut *tx).await?;

    let Some(fleet) = fleet else { return Ok(()); };
    if fleet.state != FleetState::Outbound { return Ok(()); }

    if let Some(target) = Planet::find_at(&mut tx, fleet.dst).await? {
        Planet::credit_resources(&mut tx, target.id, &fleet.carried).await?;
        ResourceTransfer::log(&mut tx, &fleet, &target).await?;
    }
    Fleet::mark_returning(&mut tx, fleet.id).await?;
    tx.commit().await?;
    Ok(())
}
```

`query_as!` проверяет SQL при компиляции, `FleetState` — суммарный тип,
`?` — короткий return-on-error. Опечатка в имени колонки = ошибка
компилятора, а не рантайма.

### C / C++

**Сильные стороны:**

- Производительность как у Rust, иногда чуть выше за счёт зрелости
  компиляторов.
- Гигантская экосистема, любая задача решена многократно.
- Идеален для порта Java-боя — те же массивы, циклы, мутабельность.

**Минусы — для серверного бэкенда в 2026 дисквалифицирующие:**

- Безопасность памяти на разработчике. Use-after-free, double-free, buffer
  overflow, data race — компилируется молча. По статистике Microsoft и Google
  ~70% CVE в C/C++ — баги памяти. Для веб-сервиса с пользовательским вводом
  это бомба замедленного действия.
- Нет стандартного менеджера зависимостей: CMake + vcpkg / Conan / submodules
  / системные пакеты — каждый проект решает по-своему.
- Async-стек незрелый: C++20 coroutines — фундамент без готовых решений,
  библиотеки сверху (Asio, libunifex, cppcoro) между собой не совместимы.
- HTTP/WS/SQL стек слабее Go и Rust: libpqxx без compile-time проверки SQL,
  HTTP-фреймворки нишевые.
- Скорость разработки самая медленная из четырёх: заголовочные файлы,
  долгая компиляция, отладка по корам.
- AI-ассистенты пишут C++ хуже Go и Rust — языковая сложность плюс 10+
  диалектов (C++11/14/17/20/23).

C — отдельная история: слишком примитивен для прикладного бэкенда.
Уместен в ядрах ОС, embedded, libc, SQLite.

**Правило 2026 года:** если 10 лет назад вы выбирали бы C++ — сегодня
выбирайте Rust. Те же гарантии скорости, без UB и с современным тулингом.

## Почему именно Go для нашего контекста

Сумма факторов под конкретный проект:

1. **Порт java-кода боя** — Go ближе к Java по типизации и парадигме, чем
   Elixir. Rust был бы технически чуть лучше, но дороже в разработке.
2. **Одиночная разработка + AI-ассистент** — Go даёт лучшее качество AI-кода
   из четырёх. Это компенсирует слабости в типовой системе.
3. **Дешёвый VPS** ([docs/ops/vps-sizing.md](../ops/vps-sizing.md)) —
   статический бинарь, ~30 МБ RAM idle. Не лучшее (Rust лучше), но
   достаточное.
4. **Скорость доставки фич** — Go проигрывает только Elixir, но Elixir
   отпадает из-за порта боя.
5. **Зрелость SQL-стека** (sqlc + goose + pgx) — лучшая в подборке.
6. **WebSocket + HTTP под нагрузкой** — один бинарь, низкая память, быстрый
   старт. Достаточно для тысяч одновременных коннектов на 1 VPS.

Где Go проигрывает и мы это принимаем:

- Нет суммарных типов с payload — компенсируем дисциплиной (sentinel-ошибки,
  exhaustive switch'и, линтеры).
- Нет compile-time проверки SQL по схеме — компенсируем sqlc, который
  генерирует типы из CREATE TABLE.
- Нет акторной модели — event-loop через таблицу `events` в БД работает,
  ценой большего количества DB-чтений на тик.

## Когда имеет смысл точечно вынести модуль на другой язык

**Бой-движок как отдельный модуль на Rust — допустимо**, если профайлер
покажет, что бой = >30% CPU воркера и масштабирование БД не помогает.
Сценарий:

1. Бой выносится в shared library или отдельный бинарь.
2. Go-воркер вызывает через cgo / unix socket / gRPC.
3. Тот же golden-тест-стек продолжает работать.

C++ для этого **не использовать** — Rust даёт ту же скорость без UB.

Elixir для отдельных модулей не имеет смысла: вынос акторной модели в
отдельный сервис требует отдельной инфраструктуры (BEAM cluster), что
превышает выгоду.

## Когда стоит пересмотреть

Сигналы к пересмотру основного стека (маловероятные, но возможные):

- Бой стал узким местом и Rust-модуль не спасает → возможно, переписать
  бой на Rust целиком.
- Реальная нагрузка превысила одну ноду и Go-стек требует сложного
  шардирования → подумать об Elixir-кластере (но это уже не «переписать
  бэкенд», а «переархитектурить»).
- Появилась команда из 5+ разработчиков с сильным Rust-бэкграундом — но
  это уже не наш сценарий.

Пока ни один из сигналов не виден, **Go остаётся правильным выбором**.
