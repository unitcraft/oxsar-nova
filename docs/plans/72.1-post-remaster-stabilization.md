## Цель

Полный dev-стек (`docker compose -f deploy/docker-compose.yml up`) после
завершения plan'ов ремастера ни разу не запускался end-to-end в полном
составе. При первом фактическом прогоне выявилось 5 разных мелких
рассогласований между сервисами — окружение, имена переменных, удалённый
column, забытый hydrate. Цель плана — **сделать стек дев-готовым:
любой свежий запуск `docker compose up` → все сервисы healthy → залогиниться
в портал/админку/игру тестовым юзером без ручной правки конфигов**.

## Что было сломано (5 классов багов — части 1-5)

### 1. Воркеры падали со старта — нет `IDENTITY_JWKS_URL`

Backend и uni02-backend получали переменную через `IDENTITY_JWKS_URL`,
а у `worker` и `uni02-worker` она не была прописана в compose. После
плана 36 Ф.12 общий config обязательно требует JWKS-URL (HS256 удалён).

**Проявление**: `docker logs deploy-worker-1` →
`ERROR worker exit err="IDENTITY_JWKS_URL is required (legacy AUTH_JWKS_URL also accepted)"`.

**Фикс**: добавил `IDENTITY_JWKS_URL: http://identity-service:9000`
в секции `worker` и `uni02-worker` в `deploy/docker-compose.yml`.

### 2. Vite-фронты не могли проксировать /auth/* — расхождение имён переменных

Compose передавал во все три фронта (nova, uni02-nova, portal)
`VITE_IDENTITY_TARGET=http://identity-service:9000`. При этом
`vite.config.ts` всех трёх фронтов читал переменную **`VITE_AUTH_TARGET`**,
fallback на `http://localhost:9000`. Внутри docker-контейнера `localhost:9000`
указывает на сам контейнер фронта → ECONNREFUSED → vite-proxy отдавал
HTTP 500 на любой запрос `/auth/*`.

**Проявление**: попытка логина на `http://localhost:5174/login` →
HTTP 500, в `docker logs deploy-portal-frontend-1`:
`[vite] http proxy error: /auth/login AggregateError [ECONNREFUSED]`.

**Решение**: переменная должна называться `VITE_IDENTITY_TARGET`
(консистентно с `IDENTITY_JWKS_URL`, `IDENTITY_SERVICE_URL`,
`IDENTITY_URL` — пользовательское решение, см. итеративный диалог в
сессии). Откатил compose; переименовал в трёх `vite.config.ts` и в
`.env.example` origin-фронта.

**Файлы**:
- `projects/portal/frontend/vite.config.ts` (2 occurrence)
- `projects/game-nova/frontends/nova/vite.config.ts` (2 occurrence)
- `projects/game-nova/frontends/origin/vite.config.ts` (2 occurrence)
- `projects/game-nova/frontends/origin/.env.example` (комментарий + пример)

### 3. Шапка портала «Войти» оставалась после успешного логина

`LoginPage` после `setAuth(res); window.location.href='/'` делает
полную перезагрузку. Стор Zustand пересоздаётся с `accessToken` и
`refreshToken` из `localStorage`, но **`user: null`**. `Layout`
показывает «Войти/Регистрация» когда `user === null`.

**Решение**: добавил метод `hydrate()` в стор — если есть
`accessToken`, но нет `user`, дёргает `GET /auth/me` и заполняет
`user`. Битый/просроченный токен — чистит localStorage. Вызов один
раз в `App.tsx` через `useEffect`.

**Файлы**:
- `projects/portal/frontend/src/store/auth.ts` — `hydrate()`.
- `projects/portal/frontend/src/App.tsx` — `useEffect(() => { void hydrate(); }, [])`.

**Гэп при отладке**: на Windows + Docker Desktop volume mount inotify
не передаётся в контейнер vite — фронт продолжал отдавать
закэшированный bundle без `hydrate`. Лечится `docker compose restart
<frontend>` + hard-reload в браузере. Зафиксировал в memory как
feedback-правило (после правок в docker'е верифицировать через
`docker exec grep` + `curl http://<port>/src/...`).

### 4. Game-nova `/api/me` падал 500 на удалённый `users.role`

После плана 52 миграция удалила столбец `users.role` (роли уехали в
identity, доступны через `claims.Roles` в JWT). Но `Me` handler
([backend/internal/auth/handler.go:35](../../projects/game-nova/backend/internal/auth/handler.go))
всё ещё селектил `COALESCE(role::text, '')` из `users` →
`ERROR: column "role" does not exist (SQLSTATE 42703)`.

**Проявление**: при попытке зайти в игру под identity-юзером
(`admin1`) — `EnsureUserMiddleware` создавал запись в game-nova
успешно, потом `/api/me` падал 500 → фронт не мог открыться.

**Решение**: убрал `role` из SQL в `Me`, добавил `roles` в JSON-ответ
из `RSAClaims(r.Context()).Roles`.

**Файл**: `projects/game-nova/backend/internal/auth/handler.go`.

**Заметка**: после фикса D подтверждено end-to-end что для входа в игру
под identity-юзером **не нужен portal handoff flow**. Прямой login через
`http://localhost:5173` (LoginScreen → identity `/auth/login` через
vite-proxy) → `EnsureUserMiddleware` lazy-провижионит юзера в game-nova
→ `Starter.Assign()` создаёт стартовую планету (Homeworld 6:258:11).
Полный flow:
- ✅ `POST /auth/login` → JWT
- ✅ `GET /api/me` → 200, `roles:["admin"]`, `credit:5`
- ✅ `GET /api/planets` → 1 планета, ресурсы тикают

Portal handoff (`POST /auth/universe-token` + `POST /auth/token/exchange`
+ `/auth/handoff` page в game-frontend) реализован в коде, но не
подключен к кнопке «Играть» в `HomePage.tsx` portal'а — она ведёт
прямо на `https://uni01.oxsar-nova.ru` без `?code=`. В dev это
эквивалентно ручному логину на game-frontend, в проде нужен handoff.
**Вынесено в отдельную задачу**, в этом плане не сделано.

### 5. Admin-консоль не запускалась — три рассогласования

Подняли admin-стек поверх dev:
- `docker-compose.admin.yml` дефолт `IDENTITY_URL=http://identity-service:9001`
  — опечатка, identity нигде не слушал 9001.
- `volumes: ./admin-ips.conf:/etc/nginx/admin-ips.conf:ro` — файла
  нет, только `.example`. Без него монтирование падает.
- `BFF_COOKIE_SECURE` дефолт `true` — на dev по `http://localhost`
  cookie с `Secure` не доедет, юзер не залогинится. Default правильный
  для прода, но при локальном запуске нужно `false` (через env).

**Решение**:
- `docker-compose.admin.yml`: дефолт `9001 → 9000` (опечатка) +
  комментарий в шапке.
- `deploy/admin-ips.conf`: создан файл `0.0.0.0/0 1;` (dev — открыто).
  Файл уже в `.gitignore`, в репо лежит только `.example`.
- `BFF_COOKIE_SECURE=false` передаётся через env при dev-запуске
  (default в compose остался `true` для прода).

**Также**: при первом запуске `ADMIN_FRONTEND_PORT` дефолт был `8081`,
который занят dev-backend. Заранее (в начале сессии при аудите портов)
заменили на `8086`.

## Lessons learned (зафиксировано в memory)

- На Windows + Docker Desktop inotify-события не передаются из хоста в
  vite-контейнер через volume mount → vite держит закэшированный bundle
  даже при свежих файлах на диске. Симптом: фикс не подхватывается.
  Лечится `docker compose restart <fe>` + hard-reload.
- После любых правок файла, который активно обслуживается docker-
  контейнером, **верифицировать применение** через
  `docker exec <c> grep <маркер> /app/<path>` и (для dev-серверов)
  `curl http://localhost:<port>/src/<file>` — иначе можно отрапортовать
  «починил», а юзер увидит старое поведение и потеряет час.

### 7. uni03-frontend = origin-фронт

Изначально uni03-frontend был nova-фронтом. Логичнее было использовать
origin-фронт (legacy pixel-perfect клон): «legacy баланс на бэке +
legacy UI на фронте» — внутренне согласовано.

Origin-фронт раньше в docker не запускался (своего Dockerfile нет),
но `nova/Dockerfile` достаточно generic — bind-mount перекрывает
исходники. Ловушка: Docker-image при build кэширует `npm install` под
nova-package.json, а origin требует другие зависимости (например,
`react-router-dom`). Bind-mount + named volume для node_modules
оставляли node_modules **пустыми** — vite падал с «react-router-dom not
found». Лечится переопределением `command: sh -c "npm install &&
npm run dev"` для uni03-frontend (npm install прогоняется в runtime,
node_modules заполняется правильно для origin'а).

Особенности:
- Vite origin'а слушает 5174 (см. `vite.config.ts` origin'а), поэтому
  port mapping `5176:5174` (а не `5176:5173`).
- Добавлено `VITE_PORTAL_BASE_URL=http://portal-backend:8090` для
  S-045 Support (план 56) — origin делает прямой fetch на portal
  с Bearer-токеном (не через vite-proxy).

### 10. Воркеры игнорировали `UNIVERSE_ID` + переименование `worker` → `uni01-worker`

При проверке логов uni02-worker и uni03-worker обнаружилось, что **все
три воркера** грузили `balance bundle universe_id=uni01,
override_applied=false`. Причина — в env воркеров не задана
`UNIVERSE_ID`, общий config-loader падал на default `uni01`. Это
значит uni02-worker и uni03-worker писали в свои БД (uni02-db, uni03-
db — это правильно по `DB_URL`), но тикали события по uni01-балансу
(не по uni02 ускорению ×4 и не по uni03 origin-override).

Семантически — это близкий родственник бага из части 1 (`IDENTITY_JWKS_URL`
у воркеров): backend получил env, воркер — нет. Аналогичный недосмотр
при copy-paste compose-секций.

**Фикс**: добавил `UNIVERSE_ID: uni01` / `uni02` / `uni03` в env
соответствующих воркеров. После пересоздания:
- uni01-worker → `universe_id=uni01, override_applied=false` (для
  uni01 нет override-файла, как и было задумано).
- uni02-worker → `universe_id=uni02, override_applied=false` (override
  отсутствует — `configs/balance/uni02.yaml` нет; loader корректно
  возвращает дефолт).

### 14. ✅ i18n не загружался — 3-уровневые YAML + битые `t()`-вызовы

После handoff'а в game-фронте (uni02/uni03) везде показывались
плейсхолдеры `[global.menuMain]`, `[overview.planetDiameter]`. На
uni01 переводы случайно работали — браузер кешировал словарь в
localStorage с прошлого захода (см. `i18n.tsx`

uni03:5176) localStorage пуст → запрос на `/api/i18n/ru` →
**404** (handler возвращает 404 если bundle nil).

**Корневая причина**: backend `internal/i18n/i18n.go:32` объявляет
`type Dict map[string]map[string]string` — двухуровневый. YAML-
файлы `configs/i18n/{ru,en}.yml` имели **3-уровневые блоки**:

```yaml
  validation:               # подгруппа (3-й уровень)
    pickArtefact: "..."
  errors:                   # подгруппа (3-й уровень)
    insufficientArtefacts: "..."

event:                      # group
  exchange:                 # подгруппа
    expire:                 # 4-й уровень

    ban:
      subject: "..."
```

`yaml.Unmarshal` при попытке прочитать map[string]string из map (3-й
уровень) падал с `cannot unmarshal !!map into string`. На уровне
i18n.Load это превращалось в WARN «i18n not loaded» → bundle = nil
→ handler 404. **Все три бэкенда** (uni01/02/03) на самом деле НЕ

1. `event.exchange.{expire,ban}.subject` — **удалён целиком** в обоих

2. `exchange.validation.*` (6 ключей) → корневая группа `validation:`

пользователем — это короткий путь к устранению блокера. В будущем

### 15. ✅ Bootstrap race — стартовая планета не успевала создаться

**Симптом**: при первом входе нового юзера в game-вселенную (через
handoff) origin-фронт показывал «Нет планет. Попробуйте перезагрузить
страницу.», а в логах backend'а появлялись пары предупреждений:

```
"retrying starter planet assign" user_id=019dd849-...
"retry starter planet assign failed" err="duplicate key value violates
unique constraint planets_galaxy_system_position_is_moon_key"
```

**Корневая причина**: race-condition в `EnsureUserMiddleware`:

1. Origin-фронт сразу при загрузке делает 4-5 параллельных запросов
   (`/api/me`, `/api/planets`, `/api/messages/unread-count`,
   `/api/fleet/incoming` и т.д.).
2. Все попадают в middleware. Первый INSERT проходит, `RowsAffected==1`
   → `go bootstrapNewUser(...)` (асинхронно).
3. Параллельные запросы видят `RowsAffected==0` (ON CONFLICT DO
   NOTHING) → `go retryBootstrapIfNeeded(...)`. Этот retry проверяет
   `cur_planet_id IS NULL` — но первый bootstrap ещё не дошёл до
   `Starter.Assign`, флаг ещё не установлен → условие true →
   параллельно вызывается **второй** `Starter.Assign`.
4. Оба `Starter.Assign` пытаются вставить планету в БД на одни и те
   же координаты → один проходит, второй падает с UNIQUE-violation.
5. Параллельно `/api/planets` возвращает **пустой массив** (юзер уже
   есть, но планеты ещё нет) → фронт показывает «Нет планет».

**Фикс**: `Starter.Assign` теперь делается **синхронно** при
`RowsAffected == 1` в `EnsureUserMiddleware` — до возврата управления
handler'у. Это гарантирует что:
- К моменту когда **любой** handler получит запрос (включая
  `/api/planets`) — планета уже создана.
- Параллельные запросы получают `RowsAffected==0` и идут через
  `retryBootstrapIfNeeded`, который видит `cur_planet_id != NULL`
  → no-op. Никаких race-call'ов на `Starter.Assign`.

Welcome-сообщение и `universe_membership` в identity остаются
**асинхронными** (выделены в отдельную функцию `bootstrapAsyncTail`)
— они не блокируют первый рендер UI. План 36 Critical-5 (membership
в identity) не страдает: до следующей выдачи JWT membership
зарегистрируется, а первый JWT уже выдан с пустым `active_universes`
(handoff не зависит от этого поля).

**Trade-off**: первый запрос юзера медленнее на ~50ms (синхронный
INSERT планеты + начальные ресурсы + здания + флот). Это приемлемо
— одноразовый цена при регистрации в каждой вселенной.

### 16. ✅ Origin MainScreen — pixel-perfect доводка под legacy main.tpl

**Симптом**: на http://localhost:5176/ origin показывает почти пустую
таблицу (имя планеты + 3 ресурса), сравнение с эталоном
http://localhost:8092 (legacy-php) выявило что origin/MainScreen
реализует ~10% от legacy main.tpl. Параллельно TopHeader работает как
placeholder (выводит UUID юзера и `—` вместо ресурсов).

**Эталон** — `projects/game-legacy-php/src/templates/standard/main.tpl`.

#### Требования и ограничения (согласованы с пользователем)

1. **Pixel-perfect зеркало legacy-PHP**. Origin портируется от
   `projects/game-legacy-php`. Все различия с эталоном — баги, не
   фичи; все элементы legacy должны быть воспроизведены если есть
   данные в API. Дизайн-отклонения недопустимы.
2. **Правки только в `projects/game-nova/frontends/origin/`**.
   Nova-фронт `projects/game-nova/frontends/nova/` **запрещено
   изменять** в рамках этой задачи (часть 16).
3. **Без захардкоженных текстов**. Все строки UI идут через i18n
   (`t('group', 'key')`). Английский неполный, но обязательно
   поддерживается.
4. **Формат i18n-плейсхолдеров — `{{var}}`** (наш стандарт, как у
   `i18next`/Vue I18n). Старый PHP-формат `{@var}` мигрируется на
   `{{var}}` при касании (technical debt). ICU MessageFormat не
   используем — overhead.
5. **Подбор i18n-ключей**: сначала искать существующий ключ в
   подходящей группе (`overview.*`, `main.*`, `global.*`); новый
   ключ только если в существующих нет ничего подходящего. Имя
   группы — семантическое (где UI это показывает).
6. **Без расширения API** в этой части. Поля которые нужны в UI
   но отсутствуют в `/api/me` или `/api/planets` (военный опыт,
   очки, ранг, шахтёр-уровень, межгал. исследования, картинки
   планет) — отложены в out-of-scope, отдельная часть/план.
7. **Удаление дублей с TopHeader**. Старая редакция MainScreen
   дублировала ресурсы в самой таблице (legacy так не делает —
   ресурсы только в шапке topHeader). Дубли удалены в рамках
   pixel-perfect-выравнивания.

**Скоуп части 16** (ограничен полями уже доступными в API):
- TopHeader: реальные ресурсы (металл/кремний/водород/энергия) +
