# План 72.1.53 — In-game ConfirmDialog (замена window.confirm)

> Подплан 72.1.2. После пользовательского запроса
> «сделай конфирм в стиль игры» (вместо native `window.confirm`).

## 1. Контекст

Native `window.confirm()` визуально не вписывается в game-frame:
- Использует системный шрифт + цвета.
- На некоторых browser'ах выглядит как фишинг-warning.
- Невозможно стилизовать под legacy `ntable`/`button` визуал.

Используется в 22+ местах (`grep -rn 'window.confirm'`). Замена на
переиспользуемый компонент в духе игры улучшает UX без изменения
логики.

## 2. Цель

Создать переиспользуемый `<ConfirmDialog>` + hook `useConfirm()`,
заменить **критичные** call sites сразу. Остальные — постепенно
по мере правок соответствующих экранов.

## 3. Реализовано

### A. Компонент

[`features/common/ConfirmDialog.tsx`](../../projects/game-nova/frontends/origin/src/features/common/ConfirmDialog.tsx):
- Использует HTML5 `<dialog>` element — native focus-trap, Esc closes,
  backdrop без сторонних зависимостей.
- API: декларативный `<ConfirmDialog open={...} onConfirm={...} onCancel={...} />`
  + императивный hook `useConfirm()` возвращает `{ confirm, dialogProps }`.
- Поддержка `destructive` flag — кнопка confirm красная для разрушительных
  действий (delete, recall, abandon).
- Стилизация через `.game-confirm-dialog` класс (см. base.css).

[`styles/base.css`](../../projects/game-nova/frontends/origin/src/styles/base.css):
- `.game-confirm-dialog` — обнуление native dialog padding/border.
- `::backdrop` — полупрозрачный чёрный overlay.

### B. Заменённые call sites (5 в 4 экранах)

1. [`MainScreen.tsx`](../../projects/game-nova/frontends/origin/src/features/main/MainScreen.tsx):
   recall флота из строки fleet-event (legacy retreatFleet).
2. [`MonitorPlanetScreen.tsx`](../../projects/game-nova/frontends/origin/src/features/monitor-planet/MonitorPlanetScreen.tsx):
   pre-scan confirmation для 5000H списания.
3. [`FleetOperationsScreen.tsx`](../../projects/game-nova/frontends/origin/src/features/fleet-operations/FleetOperationsScreen.tsx):
   recall в FleetRow (общий с #1, но отдельный экран).
4. [`MessagesScreen.tsx`](../../projects/game-nova/frontends/origin/src/features/messages/MessagesScreen.tsx)
   × 2: deleteAll + deleteOne сообщения.

Это 5 наиболее видимых destructive action'ов.

## 4. Backlog (постепенная миграция)

Остальные ~17 call sites `window.confirm` оставлены под постепенный
rollout (заменяем при следующих правках соответствующих экранов):

- `BuildPanel.tsx` × 2 (используется в shipyard/defense — VIP/cancel)
- `ConstructionsScreen.tsx` × 2 (cancel + demolish)
- `ResearchScreen.tsx` × 3 (cancel + VIP + demolish-equivalent)
- `AllianceMembersScreen.tsx` × 1 (kick member)
- `AllianceMyScreen.tsx` × 2 (leave + disband)
- `AllianceRanksScreen.tsx` × 1 (delete rank)
- `BuildingInfoScreen.tsx` × 2 (pack/demolish)
- `FriendsScreen.tsx` × 1 (remove friend)
- `ChatScreen.tsx` × 1 (delete chat msg)
- `StockScreen.tsx` × 2 (recall lot + ban)

Все оставленные `window.confirm` функционально работают — UI
консистентность будет восстановлена постепенно. Это **не блокер**:
visual mismatch, не функциональный gap.

## 5. Что **не делается**

- Не вводим toast/snackbar (другой паттерн, для info/success).
- Не реализуем dialog с custom payload (только confirm/cancel).
- Не используем headless-ui / radix-ui — лишние deps для одного
  компонента; HTML5 `<dialog>` достаточен.
- Не покрываем компонент vitest'ом отдельно — это thin wrapper
  поверх native dialog, поведение тестируется в e2e (план 73 Ф.3).
