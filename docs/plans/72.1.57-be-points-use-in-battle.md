# 72.1.57 — be_points: применение в бою (legacy 1:1)

**Дата создания:** 2026-05-03 (план 72.1.56 C12)
**Уточнён:** 2026-05-03 (verify-handsweep по реальному коду PHP+Java)
**Статус:** STUB — детальное ТЗ для отдельной сессии. Не выполнять в той
итерации что 72.1.56.
**Размер:** ~неделя работы (UI + backend + battle-engine + battle-sim
тесты + i18n).

## Контекст

72.1.56 закрыл 11 малых-средних 1:1 нарушений. `be_points` —
расхождение с legacy: сейчас поле `users.be_points` существует (план
72.1 ч.17) и **отображается** на MainScreen, но:

1. Не **начисляется** при бое (legacy тоже не начисляет автоматически —
   см. ниже).
2. Не **расходуется** при отправке атаки (legacy расходует — это
   реальное расхождение).
3. Не **применяется** в боевых формулах (legacy применяет — реальное
   расхождение).

## Источники истины (verified 2026-05-03)

### Где be_points начисляется в legacy?

**НЕ начисляется автоматически.** Поиск
`grep -rn "be_points" /d/Sources/oxsar2/www/` показывает **только две**
SQL-операции:

- `EventHandler.class.php:363` — **списание** при отправке атаки
  (UPDATE be_points - used_exp_points).
- `EventHandler.class.php:1130` — **возврат** при cancel/recall
  (UPDATE be_points + used_exp_points).

Других UPDATE'ов на `be_points` в legacy НЕТ. Это значит:

- be_points начисляется **только админом вручную** через Yii-форму
  (`SimUser_YII::_form.php:115-117` — admin edit field).
- Фактически это **донат-резерв** или **награда за участие в
  активностях вне игры** (форумы, конкурсы), а не игровая механика
  «опыт от боёв».

В nova план 72.1 ч.17 ошибочно описал «be_points += experience при
бое» — это **выдумка плана**, не legacy.

### Форма выбора уровней (Mission.class.php:1638-1671)

```php
$exp_active_levels = min(MAX_ADD_TECH_LEVELS, floor(be_points / POINTS_PER_ADD_TECH_LEVEL));
// MAX_ADD_TECH_LEVELS = 20, POINTS_PER_ADD_TECH_LEVEL = 100

if (ADVANCED_BATTLE) {
    $aval_tech = [GUN, SHIELD, SHELL, BALLISTICS, MASKING, LASER, ION, PLASMA];
} else {
    $aval_tech = [GUN, SHIELD, SHELL, BALLISTICS, MASKING];
}

foreach ($aval_tech as $techid) {
    $value = (int)$post["attack_lvl_$techid"];
    $is_adv_tech = in_array($techid, [LASER, ION, PLASMA]);
    $min_value = $is_adv_tech ? -$exp_active_levels : 0;  // продвинутые могут быть отрицательными!
    $value = clampVal($value, $min_value, $exp_active_levels);
    $exp_active_levels -= abs($value);   // общий бюджет уменьшается
    $used_exp_levels += abs($value);      // cost накапливается
    $data["add_tech_$techid"] = $value;
}
$data["used_exp_points"] = $used_exp_levels * POINTS_PER_ADD_TECH_LEVEL;
```

**Ключевые наблюдения:**

- Бюджет K = `min(20, be_points/100)` — общий на ВСЕ техники, не
  per-tech. То есть при K=10 можно взять Gun=5, Shield=3, Shell=2 и
  больше ничего.
- **Базовые техники** (Gun/Shield/Shell/Ballistics/Masking) — только
  **положительные** значения 0..K.
- **Продвинутые техники** (Laser/Ion/Plasma) — `-K..+K`, отрицательные
  тоже платные. Доступны только при `ADVANCED_BATTLE=true` для вселенной.
- **Стоимость:** `cost = Σ|value_i| × 100` be_points.

### Применение в бою (Java Participant.java:520-528)

```java
boolean isAttacker = this.isAttacker();
int structure = rs.getInt("basic_metal") + rs.getInt("basic_silicon");
int unitShell = (int) Math.round(structure * (1 + getShellLevel() / 10.0) / 10.0);
int unitAttack = (int) Math.round(rs.getInt(isAttacker ? "attacker_attack" : "attack") * (1 + getAttackLevel() / 10.0));
int unitShield = (int) Math.round(rs.getInt(isAttacker ? "attacker_shield" : "shield") * (1 + getShieldLevel() / 10.0));
int unitFront = rs.getInt(isAttacker ? "attacker_front" : "front");
double unitBallistics = rs.getInt(isAttacker ? "attacker_ballistics" : "ballistics") + getBallisticsLevel();
double unitMasking = rs.getInt(isAttacker ? "attacker_masking" : "masking") + getMaskingLevel();
```

**Формулы (verified):**

- `attack *= 1 + level_gun / 10` (мультипликативно).
- `shield *= 1 + level_shield / 10` (мультипликативно).
- `shell *= 1 + level_shell / 10` (мультипликативно).
- `ballistics += level_ballistics` (**аддитивно** к tech-уровню).
- `masking += level_masking` (**аддитивно**).

Применяется один раз при `setupBattle()`, дальше движок работает с
модифицированными значениями.

**Расхождение с моим первоначальным stub-планом:**

- Я писал «формула одинаковая для всех 5 техник» — **неверно**.
  Ballistics/Masking аддитивные, остальные — мультипликативные.
- Я писал что Laser/Ion/Plasma «advanced fire mode из 3-channel
  attack» — **неверно**. Они тоже модификаторы, но ADVANCED_BATTLE
  фича, и их baseline игнорируется в nova (3-channel attack удалён в
  плане 27, см. simplifications.md `[M4.1] Multi-channel attack`).

### Возврат при cancel (EventHandler.class.php:1130)

При recall/abort атаки `used_exp_points` возвращаются на
`users.be_points`. В nova `RecallFleet` нужно зеркалить.

## План работ

### Ф.1 — UI на MissionScreen

- В `frontends/origin/src/features/mission/MissionScreen.tsx` для
  миссий attack=1, attack_alliance=2, destroy_building=26, halt=...
  добавить блок «Усиления в бою» (collapsible, по умолчанию свёрнут).
- 5 number-input'ов для базовых техник (Gun/Shield/Shell/Ballistics/
  Masking), `min=0 max=K`. K = `Math.min(20, Math.floor(be_points/100))`.
- Если `ADVANCED_BATTLE` для вселенной → дополнительно 3 input'а для
  Laser/Ion/Plasma с `min=-K max=K`. Иначе скрываем.
- **Общий бюджет** виден над формой: «Доступно K=15, использовано 8,
  осталось 7». При вводе пересчёт live.
- Submit disabled если `Σ|values| > K`.
- i18n keys в группе `mission.battleLevels.{gun,shield,shell,ballistics,
  masking,laser,ion,plasma,available,used,remaining}`.
- `ADVANCED_BATTLE` per-вселенной — нужна универсальная инфраструктура
  (см. simplifications.md `[config] DEATHMATCH-режим` — похожий паттерн
  per-universe флагов; решить в Ф.0 или поднять отдельным мини-планом).

### Ф.2 — OpenAPI + dispatch validation

- `POST /api/fleets/dispatch` body расширить:
  ```yaml
  battle_levels:
    type: object
    properties:
      gun:        { type: integer, minimum: 0,   maximum: 20 }
      shield:     { type: integer, minimum: 0,   maximum: 20 }
      shell:      { type: integer, minimum: 0,   maximum: 20 }
      ballistics: { type: integer, minimum: 0,   maximum: 20 }
      masking:    { type: integer, minimum: 0,   maximum: 20 }
      laser:      { type: integer, minimum: -20, maximum: 20 }
      ion:        { type: integer, minimum: -20, maximum: 20 }
      plasma:     { type: integer, minimum: -20, maximum: 20 }
  ```
- В `internal/fleet/transport.go::Dispatch` валидация:
  - SELECT users.be_points + ADVANCED_BATTLE-флаг вселенной FOR UPDATE.
  - `K = min(20, be_points/100)`;
  - Для каждой техники: clamp(value, base_techs ? 0 : -K, K).
  - `Σ|value_i| ≤ K` (бюджет).
  - Если ADVANCED_BATTLE=false → laser/ion/plasma игнорируем (или
    400 error если они переданы > 0).
  - `cost = Σ|value_i| × 100`;
  - UPDATE users SET be_points = be_points - cost WHERE id = $1.
- Сохранение `add_tech_*` в payload event'а KindAttackSingle/Alliance/etc.

## Риски

- **Балансовый**: K=20 даёт attack×3, что может ломать существующие
  стратегии. Перед мерджем — battle-sim прогнать топ-10 сценариев из
  CI и сравнить с pre-be_points версией.
- **be_points начислять негде**: в legacy be_points даётся только
  админом. Без admin UI / источника начисления механика «не активна»
  для обычного игрока. Решить:
  - (a) оставить как в legacy — только админ выдаёт через grant API,
  - (b) ввести своё начисление за достижения / сезонные награды (это
    уже **новая фича game-nova**, не 1:1).
  - **Рекомендую (a)** — полный 1:1 с legacy, отдельный план для
    источников начисления при необходимости.
- **ADVANCED_BATTLE**: флаг вселенной нужен для laser/ion/plasma. См.
  пересечение с `[config] DEATHMATCH-режим` — общая инфра-задача
  «per-universe флаги». Стоит вынести в отдельный мини-план или Ф.0.
- **UI complexity**: 5+ слайдеров — много. Решение: collapsible
  секция «Расширенные настройки», по умолчанию свёрнута + tooltip
  с пояснением что такое be_points.

## Что НЕ делаем (вне scope)

- **Начисление** be_points за бой / достижения — это **новая фича**
  не из legacy. Если нужно — отдельный план.
- **3-channel attack** (laser/ion/plasma реально применять в бою) —
  оставлено как в плане 27 (scalar mode), запись хранится в payload
  для будущего расширения но не влияет на формулы.
- **Frontend cap-визуализация** для не-owner'ов в ACS-группе — кто
  что выбрал. Privacy: показываем только свой выбор.
