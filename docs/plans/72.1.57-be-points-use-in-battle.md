## Legacy-механика

### Накопление (УЖЕ РЕАЛИЗОВАНО)

При каждом бое:
- `be_points += experience`, где experience зависит от потерь
  противника (~100-1000 за рядовой бой).
- Поле `users.be_points` уже есть в схеме nova (план 72.1 ч.17).

### Использование (НЕ РЕАЛИЗОВАНО)

При **отправке** атаки игрок выбирает 5 уровней усиления:
- `attack_lvl_gun` (-K..+K) — модификатор атаки;
- `attack_lvl_shield` (-K..+K);
- `attack_lvl_shell` (-K..+K);
- `attack_lvl_ballistics` (-K..+K);
- `attack_lvl_masking` (-K..+K).

Где `K = min(20, be_points / 100)`. Например при be_points=1500
доступен диапазон −15..+15 по каждой технологии.

**Стоимость:** при отправке списывается `sum(|level_i|) × 100`
be_points (сумма абсолютных значений). Отрицательные уровни тоже
платные — игрок может, например, **уменьшить** свою защиту чтобы
выглядеть слабее (тактический ход).

**Применение в бою:** в `Participant::setupBattle` после tech-research
домножается:
```
attack *= (1 + level_gun / 10)
shield *= (1 + level_shield / 10)
shell *= (1 + level_shell / 10)
ballistics *= (1 + level_ballistics / 10)
masking *= (1 + level_masking / 10)
```

То есть +10 = ×2, −10 = ×0. Полный диапазон: ×0..×3 при K=20.

### Возврат при cancel

`EventHandler::cancelMission` возвращает уплаченные be_points при
recall до прибытия (legacy строка 363). При abort/cancel в
KindAttackSingle handler (если ещё не applied) — то же самое.

## План работ

### Ф.1 — UI на MissionScreen

- В `frontends/origin/src/features/mission/MissionScreen.tsx` для
  миссий attack=1, attack_alliance=2, destroy_building=26 добавить
  блок «Усиления в бою».
- 5 слайдеров −K..+K (K вычисляется из `be_points / 100`, max 20).
- Live-показ стоимости: `cost = Σ|levels| × 100 be_points`.
- Disable submit если cost > current be_points.
- i18n keys: `mission.battleLevels{Gun,Shield,Shell,Ballistics,Masking}`,
  `mission.battleLevelsCost`, `mission.battleLevelsAvailable`.

### Ф.2 — OpenAPI + backend dispatch

- `POST /api/fleets/dispatch` body расширить:
  ```yaml
  battle_levels:
    type: object
    required: [gun, shield, shell, ballistics, masking]
    properties:
      gun: { type: integer, minimum: -20, maximum: 20 }
      shield: { type: integer, minimum: -20, maximum: 20 }
      shell: { type: integer, minimum: -20, maximum: 20 }
      ballistics: { type: integer, minimum: -20, maximum: 20 }
      masking: { type: integer, minimum: -20, maximum: 20 }
  ```
- В `internal/fleet/transport.go::Dispatch` валидация:
  - `K = min(20, be_points / 100)`;
  - `|level_i| <= K` для каждой техники;
  - `cost = Σ|level_i| × 100`;
  - `cost <= be_points` (FOR UPDATE на users).
- Списание `users.be_points -= cost` в той же tx.
- Сохранение levels в payload event'а KindAttackSingle/Alliance/etc.

### Ф.3 — Battle engine

- Расширить `battle.Side` полем `BattleLevels{Gun,Shield,Shell,
  Ballistics,Masking int}`.
- В `internal/battle/engine.go` при сборке `Unit` (после tech-апа):
  ```go
  attack *= 1 + float64(side.BattleLevels.Gun) / 10
  shield *= 1 + float64(side.BattleLevels.Shield) / 10
  shell *= 1 + float64(side.BattleLevels.Shell) / 10
  // ballistics/masking — модификатор tech-уровня
  ```
- В `attack.go::startAttackCommon` читать `pl.BattleLevels` из payload
  и устанавливать в Side.
- ACS: каждая сторона ACS-группы имеет свои levels (legacy mission
  payload per-fleet).

### Ф.4 — Возврат при cancel

- В `internal/fleet/recall.go::RecallFleet` при recall до прибытия:
  если levels были применены (cost > 0) — вернуть на `users.be_points`
  в той же tx.
- ACS-cancel: вернуть только тому, кто инициировал свой fleet.

## DoD

- Полный flow: бой → +be_points → отправка атаки с levels → списание →
  применение в бою → cancel возвращает → repeat.
- battle-sim воспроизводит legacy-сценарий с levels (±2% delta).
- Все тесты зелёные, lint ок, screenshot-diff CI ок.
- 1 PR на 1 фазу (Ф.1-Ф.6 = 6 PR'ов).

## Риски

- **Балансовый**: K=20 даёт attack×3, что может ломать существующие
  стратегии. Перед мерджем — battle-sim прогнать топ-10 сценариев из
  CI и сравнить с pre-be_points версией.
- **Спам микро-атак**: игроки могут отправлять много мелких атак с
  отрицательными levels чтобы накопить bait-данные у защитника. Если
  обнаружится — ввести cooldown / rate-limit.
- **UI complexity**: 5 слайдеров на MissionScreen — много. Решение:
  collapsible-секция «Расширенные настройки», по умолчанию свёрнута.
