## Источники истины (verified 2026-05-03)

### Где be_points начисляется в legacy?

**Начисляется в Java assault.jar (не в PHP!).** Поиск только в PHP
ничего не дал, но `grep be_points /d/Sources/oxsar2-java/` нашёл:

`oxsar2-java/assault/src/assault/Participant.java:957-966`:
```java
int experiencePoints = isAttacker() ? Assault.atterBattleExperience
                                    : Assault.defenderBattleExperience;
sql = "UPDATE " + prefix + "user set "
    + " e_points = e_points + '" + experiencePoints + "'"
    + ", be_points = be_points + '" + experiencePoints + "'"
    + ", battles = battles + 1"
    + " where userid = '" + userid + "'";
stmt.executeUpdate(sql);
```

То есть после каждого боя **e_points = be_points = experiencePoints**
(тождественно). Java assault.jar подключается к БД напрямую (минуя
PHP) и пишет результаты.

### Формула experience (Assault.java:817-847)

```java
battleTurnsCoefficient = pow(battleTurnsNumber, 1.1) / BATTLE_MAX_TURNS

atterExperience = (atan(defPower/atkPower * 1.5 - 1.5) + 1) * 0.4 * 3 * battleTurnsCoefficient + 1
defenderExperience = (atan(atkPower/defPower * 1.5 - 1.5) + 1) * 0.4 * 3 * battleTurnsCoefficient + 1

// бонус за победу:
if attacker won: atterExperience *= 3
if defender won: defenderExperience *= 3
if draw:         atterExperience *= 1.5; defenderExperience *= 1.7

// масштаб от мощи боя:
battlePower = sqrt(atkPower * defPower) / 1_000_000
battlePowerCoefficient = (atan(battlePower * 10 * 0.2 - 1.6) + 1) * 0.4 * 19 + 1

if planetid == 0:  // expedition / debris field
    battlePowerCoefficient *= 0.5

atterExperience *= battlePowerCoefficient
defenderExperience *= battlePowerCoefficient

atterBattleExperience = round(atterExperience)
defenderBattleExperience = round(defenderExperience)
```

**Минимум:** `+1` слагаемое в формуле гарантирует, что даже при
полном превосходстве одной стороны другая получит хотя бы 1 опыт за
участие.

**Применимость в nova:** наш Go battle-engine (port Java) знает
`battleTurnsNumber`, `startBattleAtterPower`, `startBattleDefenderPower`,
`assaultResult` (1=atk_won, 2=def_won, 0=draw), `planetid==0` для
expedition. Всё нужное доступно. Реализуется в `internal/battle/engine.go`
после finishBattle, через возврат `BattleExperience{Attacker, Defender int}`
в `Report`. Caller (`fleet/attack.go::finalizeAttack`) обновляет
`users.be_points` и `users.e_points` для каждого участника.

ACS: каждый attacker получает **полную** experience? Или делится?
Java получает один общий `experiencePoints` для каждого атакующего
участника независимо (цикл `for each isReported user`). То есть **каждый
ACS-участник получает столько же be_points, сколько одиночный атакующий
с теми же параметрами**. Не делится.

### Форма выбора уровней (Mission.class.php:1638-1671)

**Scope nova:** только 5 базовых техник (`ADVANCED_BATTLE`-флаг
вселенной + Laser/Ion/Plasma — отложено, не входит в 72.1.57).
Решение пользователя 2026-05-03: ADVANCED_BATTLE требует подумать
отдельно (не делаем сейчас).

```php
$exp_active_levels = min(MAX_ADD_TECH_LEVELS, floor(be_points / POINTS_PER_ADD_TECH_LEVEL));
// MAX_ADD_TECH_LEVELS = 20, POINTS_PER_ADD_TECH_LEVEL = 100

if (ADVANCED_BATTLE) {
    $aval_tech = [GUN, SHIELD, SHELL, BALLISTICS, MASKING, LASER, ION, PLASMA];
} else {
    $aval_tech = [GUN, SHIELD, SHELL, BALLISTICS, MASKING];
}

foreach ($aval_tech as $techid) {
    $value = (int)$post["attack_lvl_$techid"];
    $is_adv_tech = in_array($techid, [LASER, ION, PLASMA]);
    $min_value = $is_adv_tech ? -$exp_active_levels : 0;  // продвинутые могут быть отрицательными!
    $value = clampVal($value, $min_value, $exp_active_levels);
    $exp_active_levels -= abs($value);   // общий бюджет уменьшается
    $used_exp_levels += abs($value);      // cost накапливается
    $data["add_tech_$techid"] = $value;
}
$data["used_exp_points"] = $used_exp_levels * POINTS_PER_ADD_TECH_LEVEL;
```

**Ключевые наблюдения:**

- Бюджет K = `min(20, be_points/100)` — общий на ВСЕ техники, не
  per-tech. То есть при K=10 можно взять Gun=5, Shield=3, Shell=2 и
  больше ничего.
- **Базовые техники** (Gun/Shield/Shell/Ballistics/Masking) — только
  **положительные** значения 0..K.
- **Продвинутые техники** (Laser/Ion/Plasma) — `-K..+K`, отрицательные
  тоже платные. Доступны только при `ADVANCED_BATTLE=true` для вселенной.
- **Стоимость:** `cost = Σ|value_i| × 100` be_points.

### Применение в бою (Java Participant.java:520-528)

```java
boolean isAttacker = this.isAttacker();
int structure = rs.getInt("basic_metal") + rs.getInt("basic_silicon");
int unitShell = (int) Math.round(structure * (1 + getShellLevel() / 10.0) / 10.0);
int unitAttack = (int) Math.round(rs.getInt(isAttacker ? "attacker_attack" : "attack") * (1 + getAttackLevel() / 10.0));
int unitShield = (int) Math.round(rs.getInt(isAttacker ? "attacker_shield" : "shield") * (1 + getShieldLevel() / 10.0));
int unitFront = rs.getInt(isAttacker ? "attacker_front" : "front");
double unitBallistics = rs.getInt(isAttacker ? "attacker_ballistics" : "ballistics") + getBallisticsLevel();
double unitMasking = rs.getInt(isAttacker ? "attacker_masking" : "masking") + getMaskingLevel();
```

**Формулы (verified):**

- `attack *= 1 + level_gun / 10` (мультипликативно).
- `shield *= 1 + level_shield / 10` (мультипликативно).
- `shell *= 1 + level_shell / 10` (мультипликативно).
- `ballistics += level_ballistics` (**аддитивно** к tech-уровню).
- `masking += level_masking` (**аддитивно**).

Применяется один раз при `setupBattle()`, дальше движок работает с
модифицированными значениями.

**Расхождение с моим первоначальным stub-планом:**

- Я писал «формула одинаковая для всех 5 техник» — **неверно**.
  Ballistics/Masking аддитивные, остальные — мультипликативные.
- Я писал что Laser/Ion/Plasma «advanced fire mode из 3-channel
  attack» — **неверно**. Они тоже модификаторы, но ADVANCED_BATTLE
  фича, и их baseline игнорируется в nova (3-channel attack удалён в
  плане 27, см. simplifications.md `[M4.1] Multi-channel attack`).

## План работ

### Ф.0 — Battle engine: возврат experience (фундамент)

- Расширить `battle.Report` полем
  `Experience{Attackers []int64, Defenders []int64}` (по индексу
  стороны, аналогично `Units`).
- В `engine.go::Run` после finishBattle вычислить experience по
  legacy-формуле (Assault.java:817-847):
  ```go
  battleTurnsCoefficient = math.Pow(turnsNumber, 1.1) / float64(MaxTurns)
  atkExp = (math.Atan(defPower/atkPower*1.5-1.5)+1)*0.4*3*battleTurnsCoefficient + 1
  defExp = (math.Atan(atkPower/defPower*1.5-1.5)+1)*0.4*3*battleTurnsCoefficient + 1
  // win-multiplier
  switch winner { case "attackers": atkExp *= 3 ... }
  // power-coefficient
  battlePower = math.Sqrt(atkPower*defPower) / 1e6
  battlePowerCoefficient = (math.Atan(battlePower*10*0.2-1.6)+1)*0.4*19 + 1
  if expedition { battlePowerCoefficient *= 0.5 }
  atkExp *= battlePowerCoefficient
  defExp *= battlePowerCoefficient
  ```
- Pure helper `calcBattleExperience(turnsNumber, atkPower, defPower,
  winner string, isExpedition bool) (atk, def int64)` для unit-тестов.
- ACS: каждый ACS-участник получает full experience (Java loop over
  reportedUsers).
- Тесты: 8+ кейсов (атакующий победил, защитник победил, draw,
  expedition, big power gap, equal power, single turn, max turns).

### Ф.1 — Backend: начисление в users (победитель/проигравший)

- В `internal/fleet/attack.go::finalizeAttack` (single + ACS) после
  получения `report.Experience` для каждого участника:
  ```sql
  UPDATE users SET be_points = be_points + $1,
                   e_points  = e_points + $1,
                   battles   = battles + 1
   WHERE id = $2
  ```
  где `$1 = experience` от `report.Experience`.
- Игнорируем rocket-attack (`isRocketAttack=true` в legacy
  пропускается, см. `Assault.java:811`).
- Alien-attack handler — тоже обновляет (alien это NPC-attacker, но
  defender — реальный игрок, должен получать experience).
- Expedition handler — defender пуст (пираты), attacker получает
  experience с `isExpedition=true` (×0.5 coefficient).
- nova поле `e_points` пока не существует — по решению пользователя
  можно либо добавить миграцию (полный 1:1), либо начислять только
  `be_points` (e_points = синоним, нигде не используется в nova).
  **Решить перед Ф.1**: нужен ли `e_points` отдельно?

## Риски

- **Балансовый**: K=20 даёт attack×3, что может ломать существующие
  стратегии. Перед мерджем — battle-sim прогнать топ-10 сценариев из
  CI и сравнить с pre-be_points версией.
- **e_points field в nova**: его пока нет в users-схеме. Решить
  перед Ф.1: добавляем миграцию (полный 1:1) или начисляем только
  be_points (e_points = синоним, нигде в nova не используется).
- **Накопленные be_points у существующих игроков**: после Ф.0+Ф.1
  игроки начнут получать be_points за каждый бой задним числом? Нет
  — только за бои **после** деплоя Ф.1. Backfill не делаем.
- **ACS дисбаланс**: каждый ACS-участник получает full experience
  как в legacy Java (loop over reportedUsers). Это значит в ACS-
  группе из 10 человек на 1 защитника — 10 атакующих получат +X
  каждый, защитник тоже +X. Проверить что не вызывает экспоненциальный
  рост у активных ACS-игроков.
- **UI complexity**: 5 слайдеров. Решение: collapsible секция
  «Усиления в бою», по умолчанию свёрнута + tooltip с пояснением
  что такое be_points.
