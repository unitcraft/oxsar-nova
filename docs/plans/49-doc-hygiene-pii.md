## Цель

Убрать из технических и нарративных документов проекта (`docs/plans/`,
`docs/release-roadmap.md`, `docs/project-creation.txt`,
`docs/ops/payment-integration.md`) упоминания **налогового статуса
автора** (самозанятый / НПД / плательщик НПД / «Мой налог» —
применительно к автору лично) и упоминания **ФИО автора** в
технических контекстах, где смысл сохраняется при замене на
нейтральное «автор» / «правообладатель».

В юридических документах (`LICENSE`, `COMMERCIAL-LICENSE.md`, `CLA.md`,
`docs/legal/{offer,privacy-policy,game-rules,refund-policy}.md`,
`docs/origin-rights.md`, `README.md` §Лицензия) — формулировки
**оставить как есть**: там идентификация правообладателя обязательна
по сути юридического документа.

---

## Что меняем

### 1. Налоговый статус автора («самозанятый», «НПД», «плательщик НПД»,
   «Мой налог» как личный сервис автора)

В технических документах **убираем** или заменяем на нейтральные
формулировки:
- «для самозанятого» → «для оператора, применяющего НПД» (если контекст
  про возможности шлюза) или просто убрать статусную привязку;
- «&lt;ФИО автора&gt; (самозанятый)» → «автор» / «правообладатель»;
- «чек в "Мой налог"» (применительно к автору) → «формирование чека
  через ОФД платёжного агрегатора»;
- сравнительные таблицы шлюзов («ЮKassa поддерживает самозанятых,
  Робокасса — самозанятых») — это объективные факты о шлюзах,
  **не** про автора лично, оставлять можно. Чёткий критерий: упоминание
  «самозанятый» — про **автора** или про **возможность шлюза**?
  Про автора — убирать; про шлюз — оставлять.

Файлы:
- `docs/plans/07-payments.md` — заголовок «Сравнение для самозанятого/
  физлица» переформулировать;
- `docs/plans/39-license-ru-jurisdiction.md` — формулировка про реквизиты
  убирает упоминание автора как «самозанятого/физлица», оставить
  «реквизиты автора отдельным договором»;
- `docs/plans/42-yookassa.md` — упоминания «для самозанятого через API
  ЮKassa» переформулировать как «автоматическое формирование чека НДС-0
  через API ЮKassa»;
- `docs/plans/44-personal-data-152fz.md` — убрать «&lt;ФИО,
  налоговый статус&gt;» из обоснования и формулировок реквизитов оператора;
- `docs/plans/46-age-rating-ugc.md` — «штраф для ИП/самозанятых» это
  формулировка КоАП, оставить (это юридическая норма, не про автора лично);
- `docs/plans/47-offer-tos.md` — три места про «&lt;ФИО,
  налоговый статус&gt;» и одна формулировка реквизитов;
- `docs/ops/payment-integration.md` — runbook для платежей, упоминания
  «для самозанятого» в инструкциях оставить там, где это влияет на
  процедуру (например, «выбрать тип "Физлицо/самозанятый"» в ЛК
  Робокассы), убрать риторические;
- `docs/project-creation.txt` — две записи дневника, переформулировать
  на нейтральные.

### 2. ФИО автора (любые формы — кириллицей и латиницей)

В технических документах заменяем на «автор» / «правообладатель»
там, где смысл сохраняется. Где формулировка цитирует будущий
публичный текст (например, план 45 цитирует итоговую запись для README
после регистрации товарного знака с упоминанием правообладателя) —
оставить, это
часть конечного публичного текста.

Файлы:
- `docs/plans/41-origin-rights.md` (стр. 18, 34) — формулировка про
  автора кода → нейтральная «автор-правообладатель»;
- `docs/plans/44-personal-data-152fz.md` — формулировка про оператора
  ПДн → нейтральная «автор как оператор персональных данных»;
- `docs/plans/45-trademark.md` — оставить (это цитата итогового текста
  для README после регистрации товарного знака).

### 3. Где НЕ трогаем

- `LICENSE` — Copyright требует ФИО, не трогаем.
- `COMMERCIAL-LICENSE.md` — реквизиты оператора нужны по сути сделки.
- `CLA.md` — Owner идентифицируется ФИО, нужно для передачи прав.
- `docs/legal/offer.md` — публичная оферта по ст. 437 ГК требует
  идентификации оферента; реквизиты оператора (включая применимый
  налоговый режим) здесь обязательны по сути сделки (обсуждалось при
  выполнении плана 47).
- `docs/legal/privacy-policy.md` — 152-ФЗ требует идентификации
  оператора ПДн (ФИО + юрисдикция РФ + email + регномер РКН).
- `docs/legal/{game-rules,refund-policy}.md` — реквизиты оператора
  для логичной связи с офертой.
- `docs/origin-rights.md` — документ о происхождении прав, ФИО автора
  здесь сущностно обязательно.
- `README.md` §Лицензия — короткая отсылка к LICENSE, ФИО уже там.

### 4. Memory-правило

Расширить `feedback_legal_docs_public.md` явным указанием, что правило
«не светить налоговый статус» применяется **также** к:
- `docs/plans/*.md` (планы и подзадачи);
- `docs/release-roadmap.md`;
- `docs/project-creation.txt` (дневник).

И добавить новый пункт: **ФИО автора в технических документах** —
заменять на «автор» / «правообладатель» там, где смысл не теряется;
в юридических документах оставлять обязательно.

## Что НЕ делаем

- Не вводим псевдоним автора, не скрываем ФИО из юридических документов
  (там оно обязательно по сути).
- Не переписываем LICENSE / COMMERCIAL-LICENSE / CLA / docs/legal/* /
  origin-rights / README §Лицензия — реквизиты там обязательны.
- Не создаём ИП/ООО для смены модели правообладателя (это отдельная
  задача — выходит за рамки гигиены документации).
- Не правим atomар-формулировки в коде (комментарии, строки) — текущий
  scope ограничен документацией в `docs/`.
