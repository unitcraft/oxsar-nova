## 1. Контекст

Legacy [`Profession.class.php`](../../projects/game-legacy-php/src/game/page/Profession.class.php)
имеет `setPostAction("save", "changeProfession")` — выбор/смена
профессии. По §20.12 чек-листу обнаружены 4 расхождения:

1. **umode-блок** (legacy `Profession.class.php:31-34`):
   ```php
   if(NS::getUser()->get("umode")){
       Logger::dieMessage('UMODE_ENABLED');
   }
   ```
   У нас в [`service.go::Change`](../../projects/game-nova/backend/internal/profession/service.go)
   проверки нет — игрок в отпуске может менять профессию.

2. **same-profession check** (legacy `Profession.class.php:36`):
   ```php
   if(NS::getUser()->get("profession") != $id ...)
   ```
   У нас отсутствует — `Change("miner")` при текущем `miner` спишет
   1000 кредитов и сбросит cooldown.

3. **AutoMsg `MSG_CREDIT_PROFESSION_CHANGED`** (legacy
   `Profession.class.php:54-65`) — отправляется в папку `MSG_CREDIT`
   при успешном списании. У нас НЕТ.

4. **`description` поле** профессий (legacy шаблон
   `profession.tpl`/`assign('desc', getItem(name+'_DESC'))`):
   у нас DTO отдаёт только `label`, без `description`.

## 2. Что нужно

### Backend

#### `service.go::Change`
- Перед `FOR UPDATE`-чтением — прочитать `umode` из users; если true
  → `ErrInVacation`.
- После чтения текущей `profession` — если `professionKey == current`
  → no-op (вернуть nil без списания, как legacy).
- После успешного `UPDATE users` (в той же транзакции) — `automsg.
  SendDirect(folder=MSG_CREDIT, title, body)` с шаблоном.
  Folder=MSG_CREDIT в legacy — нужно найти константу. Беру folder=8
  (план 11 шаг 2 — кредитные транзакции в folder=8).

#### `service.go` ошибки
```go
ErrInVacation = errors.New("profession: cannot change in vacation mode")
```

#### `handler.go`
- `Change` — мапнуть `ErrInVacation` → 400.

#### DTO `ProfessionDTO`
```go
type ProfessionDTO struct {
    Key         string         `json:"key"`
    Label       string         `json:"label"`
    Description string         `json:"description,omitempty"`
    Bonus       map[string]int `json:"bonus,omitempty"`
    Malus       map[string]int `json:"malus,omitempty"`
}
```

#### `config/professions.yml` или `ProfessionSpec`
- Добавить поле `description: string` в `config.ProfessionSpec`.
- В YAML конфиге — заполнить (если уже есть тексты в i18n —
  использовать `name+_DESC` lookup) или хранить ключ.

Проверю текущий конфиг — если description уже идёт через i18n-ключ
типа `profession.<key>.desc`, тогда фронт может tr-ить сам без
изменений в DTO. Backend задача — отдать **ключ** или **уже
переведённый текст**? Решение: backend отдаёт `description_key:
"profession.<key>.desc"`, frontend `t()`-ит. Это чище: backend
language-agnostic.

Уточнение: legacy резолвит `getItem($name+'_DESC')` на сервере
(шаблон уже получает строку). Нам в Go-сервере подключение
i18n-bundle для лоадов нежелательно (фронт многоязычный, юзер
выбирает в UI). Лучше отдать `description_key` и tr-ить фронтом.

### Frontend

#### `api/types.ts::Profession`
```ts
export interface Profession {
  key: string;
  label: string;
  description_key?: string;
  bonus?: Record<string, number>;
  malus?: Record<string, number>;
}
```

#### `ProfessionScreen.tsx`
- Под названием профессии — описание из `t(description_key)` (или
  fallback на пустую строку).

### i18n

Группа `profession`:
- `descMiner`, `descBuilder`, `descScientist`, … — описания каждой
  профессии (ru/en). Ключи синхронизированы с `description_key`.
- `errInVacation` — «Нельзя менять профессию в режиме отпуска» /
  «Cannot change profession in vacation mode».
- `autoMessages.creditProfessionChanged.title/body` — для AutoMsg.

### AutoMsg

Шаблон `creditProfessionChanged`:
- Title: «Списание кредитов: смена профессии» / «Credit charge: profession change»
- Body: «Списано {{amount}} кредитов за смену профессии на «{{profession}}».» (с переменными)

`automsg.SendDirect` без идемпотентности (можно несколько смен в год).
