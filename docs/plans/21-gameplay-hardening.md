## Блок A: Lancer — прямой нерф стоимости (крит)

Диагностика: см. [balance-audit.md BA-002](../balance/audit.md#ba-002-lancer-ship--аномальное-attack-per-resource).

### Вариант A1 — повысить стоимость

```yaml
# ships.yml — Lancer (102)
cost:
  metal: 50000      # +25000 (был 25000)
  silicon: 10000    # +2500 (был 7500)
  hydrogen: 20000   # +5000 (был 15000)
# attack per 1k опустится с 61 до ≈30 — близко к Cruiser/Frigate
```

### Вариант A2 — снизить attack

```yaml
attack: 3500        # -2000 (был 5500) — при стоимости 90k даёт 38/1k
shell: 8000         # -2000 (был 10000) — более хрупкий
```

### Рекомендация

**A1** — повышение стоимости. Геймплейно честнее: Lancer остаётся
сильным, но становится элитным (не spam-юнитом). A2 ломает статы в
тестах паритета с Java JAR.

### ADR

Принять конкретные числа. Баланс проверить симуляцией перед деплоем.

## Блок B: Экспедиции — закрыть farm-эксплойт (крит)

Диагностика: см. [balance-audit.md BA-003](../balance/audit.md#ba-003-экспедиции--фарм-ресурсов-минимальным-флотом).

Этот блок дополняет **внутри одной экспедиции** то, что уже частично
закрыто планом 17 B1 (полное уничтожение флота при `lost`) и планом 20 Ф.7
(ограничение количества одновременных экспедиций через ASTRO_TECH).

### B1. Минимальный флот для экспедиции

Требование: стоимость флота ≥ N% от средней награды на текущем `exp_power`.
Если флот меньше порога — экспедиция проваливается с сообщением
«экипаж слишком мал, чтобы справиться с неизвестным».

```go
// fleet/expedition.go — в валидации перед стартом
minFleetValue := expectedReward(expPower) * 0.1  // 10% от EV
if fleetValue(ships) < minFleetValue {
    return ErrExpeditionFleetTooSmall
}
```

Эффект: нельзя фармить экспедиции 1-шипным флотом. Чтобы получить
big reward — нужно рисковать big fleet.

### B2. Масштабирование `lost_weight` с `exp_power`

Текущее: `lost=10` статично → 0.19% при exp_power=11.
Предложение: `lost = 10 × (1 + exp_power × 0.1)` → при power=11 вес 21 → ~0.4%,
при power=30 → вес 40 → ~1%.

Не слишком жёстко, но гарантирует: чем сильнее флот — тем выше риск
полностью потерять его при неудаче.

### B3. Совместно с планом 17 B1

План 17 B1 делает `outcome="lost"` полным уничтожением флота (как в legacy).
Это + B2 (рост шанса с power) + B1 (порог размера) → закрытие эксплойта.

### B4. Cap на награды относительно флота

Награда ресурсов не может превышать `3× fleet_value`. Экспедиция —
не лотерея на 1000×, а рискованная разведка с умеренным upside.

```go
reward := computeReward(expPower, r)
reward = math.Min(reward, fleetValue(ships) * 3)
```

Эффект: малый флот — малая награда, большой флот — большой risk/reward.

## Блок C: Щиты — проверка и cap (средний приоритет)

Диагностика: см. [balance-audit.md BA-005](../balance/audit.md#ba-005-щиты-близки-к-неуязвимости-при-high-tech).

### C1. Golden-тест: лёгкий флот vs high-tech Small Shield

Сценарий:
- Defender: 100× Small Shield при shield_tech=10 (эффективный 5180)
- Attacker: 10 000× Light Fighter (attack=50)
- Проверить: за сколько раундов оборона падает

Если падение > 50 раундов при соотношении 100:10000 — щиты слишком
прочные, нужен нерф множителя shield_tech (1.1→1.05) или cap на
`ignoreAttack ≤ baseShield / 100` (не масштабируется от tech).

### C2. Cap ignoreAttack независимо от shield_tech

```go
// engine.go::computeIgnoreAttack
ignoreAttack := baseShield / 100.0   // только base, без tech-множителя
// tech по-прежнему увеличивает абсорбцию щита, но не порог игнора
```

Эффект: лёгкий флот всегда может нанести хотя бы царапину. Щит-башни
сильнее по абсорбции, но не становятся абсолютными.

### C3. Риск

Это может ослабить оборону в целом → проверить совместно с планом 18
(который уже нерфит Bomber и Cruiser против защит).
