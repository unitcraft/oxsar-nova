## 3. Реализовано

### A. Client-side validation для rank-name (1-32 символа)

[`AllianceRanksScreen.tsx`](../../projects/game-nova/frontends/origin/src/features/alliance/AllianceRanksScreen.tsx)
кнопка «Сохранить» disabled если `name.trim().length < 1 || > 32`.
Backend `ErrRankNameInvalid` дублирует, но FE не должна позволять
отправить пустое.

### B. Client-side validation для alliance tag/name

[`AllianceManageScreen.tsx`](../../projects/game-nova/frontends/origin/src/features/alliance/AllianceManageScreen.tsx):
- tag: 3-5 ASCII alphanumeric (`/^[A-Za-z0-9]+$/`).
- name: 3-64 символа.
Backend сентинели `ErrInvalidTag` / Create-time проверка name 3-64
дублируют. Кнопка «Сохранить» disabled если правила не выполнены.

### C. Documented simplification [P72.S2.ALLIANCE_PREFS]

В [`docs/simplifications.md`](../simplifications.md) добавлена запись
о том что 6 legacy полей `updateAllyPrefs` сознательно не реализованы:
- `logo` / `homepage` — UGC URL, требует image-proxy + SSRF-защиты
  (plan 46-tier moderation).
- `foundername` — кастомный title основателя (nova: `rank='owner'`).
- `showmember` / `showhomepage` — privacy preferences (nova: всё
  всем доступно).
- `memberlistsort` — preference сортировки (nova: client-side / fixed).

Не блокер для core gameplay; восстанавливается отдельным планом при
появлении UGC-moderation pipeline.

## 4. Что **не делается**

- backend изменения (всё уже работает).
- per-field PATCH'и (legacy `changetag`/`changename` отдельно — bulk
  удобнее).
- legacy permission bits `CAN_SEE_MEMBERLIST/CAN_SEE_APPLICATIONS/
  CAN_BAN_MEMBER/CAN_SEE_ONLINE_STATE/CAN_WRITE_GLOBAL_MAILS` — у
  nova другой набор 7 permission'ов (can_invite, can_kick,
  can_send_global_mail, can_manage_diplomacy, can_change_description,
  can_propose_relations, can_manage_ranks). Архитектурный refactor
  prison-системы, не legacy 1:1.
