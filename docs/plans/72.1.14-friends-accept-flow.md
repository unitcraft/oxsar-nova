## 1. Контекст

Legacy [`Friends.class.php`](../../projects/game-legacy-php/src/game/page/Friends.class.php)
реализует двустороннюю дружбу:
- Таблица `buddylist (relid, friend1, friend2, accepted)`.
- Add: `INSERT (friend1=me, friend2=target, accepted=0)`. Дубликаты
  ловятся проверкой по обоим направлениям.
- Accept: `UPDATE accepted=1 WHERE relid=$1 AND friend2=me`.
- Remove: `DELETE WHERE relid=$1 AND (friend1=me OR friend2=me)`.
- Список: `WHERE friend1=me OR friend2=me ORDER BY points DESC`,
  показывает обе стороны (отправленные мной + входящие мне).

Origin [`backend/internal/friends/handler.go`](../../projects/game-nova/backend/internal/friends/handler.go)
+ миграция `0053_friends.sql`:
- `friends (user_id, friend_id, created_at)` PK `(user_id, friend_id)`.
- Однонаправленный: A→B = «A добавил B», B о факте не знает.
- Endpoints: List/Add/Remove (без accept).

§20.12 плана 72.1 требует строгий функциональный паритет с legacy
и явно перечисляет в запрещённом «Кнопка/действие в legacy работает,
в origin не реализовано» (accept) и «Поле/колонка в legacy есть, в
origin отсутствует» (`accepted`).

## 2. Что нужно

### Backend

#### Миграция `0086_friends_pending.sql`
```sql
ALTER TABLE friends
  ADD COLUMN accepted boolean NOT NULL DEFAULT false;

-- Бэкфилл: все существующие записи считаем accepted (текущая модель
-- односторонняя без подтверждения; не превращать живые friendship'ы
-- в pending).
UPDATE friends SET accepted = true WHERE accepted = false;

-- Симметричная пара для каждого существующего friendship.
-- Текущий unidirectional UI пишет одну запись (A,B); legacy ожидает
-- запись для каждой из сторон. Создаём (B,A,accepted=true) для всех
-- существующих (A,B), если ещё нет.
INSERT INTO friends (user_id, friend_id, created_at, accepted)
SELECT f.friend_id, f.user_id, f.created_at, true
FROM friends f
LEFT JOIN friends g ON g.user_id = f.friend_id AND g.friend_id = f.user_id
WHERE g.user_id IS NULL
ON CONFLICT DO NOTHING;

-- Partial-index для быстрых выборок «входящие запросы мне».
CREATE INDEX friends_pending_idx ON friends (friend_id) WHERE NOT accepted;
```

Down: симметричная пара остаётся (безопасный rollback), DROP COLUMN.

#### Handler

Изменения в [`handler.go`](../../projects/game-nova/backend/internal/friends/handler.go):

- **`List`** — два списка:
  - `accepted = true` (друзья — обе стороны видят друг друга).
  - Опциональный фильтр `?pending=incoming|outgoing|all`:
    - `incoming` = `friend_id = me AND NOT accepted` (мне пришёл запрос).
    - `outgoing` = `user_id = me AND NOT accepted` (я отправил, ждёт).
- **`Add`** (POST `/api/friends/{userId}`) — переписать логику:
  - Если уже есть пара `(friend, me, accepted=true)` (т.е. they
    добавили меня раньше) → автоматически принять: `INSERT (me, friend,
    accepted=true)` + `UPDATE` встречной записи `accepted=true` (она
    уже true). Случай «mutual instant-accept» (legacy эквивалент:
    `friend1=target, friend2=me, accepted=0` найдено — принять).
  - Иначе обычный pending: `INSERT (me, friend, accepted=false)`.
  - Запрет дубликата по обоим направлениям.
- **`Accept`** (новый POST `/api/friends/{userId}/accept`):
  - Существует `(friend, me, accepted=false)` (входящая мне).
  - Транзакция: `UPDATE accepted=true WHERE user_id=friend AND friend_id=me`
    + `INSERT (me, friend, created_at, accepted=true)` (симметричная
    пара, чтобы оба видели друг друга).
- **`Remove`** (DELETE `/api/friends/{userId}`) — удалить **обе**
  стороны: `DELETE WHERE (user_id=me AND friend_id=$1) OR
  (user_id=$1 AND friend_id=me)` (в одной транзакции).

Все мутации — Idempotency-Key header (R9).

#### `FriendsScreen.tsx`

Три секции в одном экране (legacy показывает всё в одной таблице,
у нас разделим для ясности):
1. **Друзья** (accepted=true) — как сейчас, с remove-кнопкой.
2. **Входящие запросы** (incoming pending) — с кнопками Accept/Decline.
3. **Отправленные** (outgoing pending) — с кнопкой Cancel (= remove
   pending записи).

Секция «Входящие» рендерится только если есть записи (как в legacy).
