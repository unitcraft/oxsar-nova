# План 24: ИИ-персонажи (боты, имитирующие живых игроков)

**Дата**: 2026-04-24
**Статус**: Концепт принят — см. [ADR-0003](../adr/0003-ai-players.md)
**Затрагивает**: backend (новый пакет `aiplayer`), worker, migrations, configs

> **Главное ограничение**: без внешних нейросетей. Все решения — это
> **детерминированные правила + взвешенный рандом + state machines**.
> ИИ-игрок выглядит как живой, но его поведение описывается конечным
> числом политик, которые можно тестировать и настраивать.

---

## Зачем это нужно

1. **Заполнение пустоты галактики**. Когда реальных игроков мало, вселенная
   кажется мёртвой. ИИ-игроки создают активность: строят, воюют, торгуют.
2. **PvP-тренировка для новичков**. Живые игроки бьют новичков → новички
   уходят. ИИ слабее профи → новичку есть на ком учиться.
3. **Геймплейная прогрессия**. Разные тиры ИИ (от «пират-одиночка» до
   «империя») дают цели: сначала фармим слабых, потом боремся с сильными.
4. **Экономический драйв**. ИИ торгуют на рынке, создают спрос и
   предложение → курсы живут.
5. **Альтернатива Alien AI**. Сейчас (план 15) есть только HOLDING-механика
   от пришельцев. Живые ИИ-игроки — следующий уровень: свои планеты, свой
   флот, свои цели.

---

## Что это НЕ

- **НЕ** OpenAI/Claude/LLM — это внешняя зависимость, стоимость на токены,
  латенс, недетерминированность в тестах.
- **НЕ** RL/нейросети, обученные на данных — требуют ML-инфраструктуры,
  датасетов, отладки.
- **НЕ** просто «респавн пришельцев» (это план 15) — там монстр без
  экономики и развития.
- **НЕ** MMO-NPC с жёстко заскриптованным поведением — это будет скучно.

Это **правило-ориентированный симулятор**: игрок-бот имеет политики,
ресурсы, планеты, исследования, флот — всё как у живого. Отличие —
решения принимаются кодом по правилам, а не человеком.

---

## Архитектура: как бот «играет»

### Ключевая идея — **behavior tree + policy-weighted actions**

Бот — это запись в БД (`ai_players`), у которой:
- **Profile** — тип личности (см. ниже): тянет базовые веса действий
- **State** — текущие планеты, флот, исследования, кредиты (те же таблицы,
  что у живого игрока, просто `users.is_ai = true`)
- **Memory** — короткая история: кого недавно атаковали, где добыли
  debris, сколько раз получали по шапке
- **Goals stack** — стек текущих целей (например: «построить 3-й
  metal_mine → исследовать Laser 3 → построить флот → атаковать соседа»)

Каждые **N секунд** (воркер `ai-tick`, N = 30–600 в зависимости от
tier) бот «думает»:

1. **Perceive** — читает своё состояние + окружение (соседние планеты в
   галактике, рынок, события в папке сообщений)
2. **Score** — для каждого возможного действия (build, research, send
   fleet, trade, expedition) считает скор = `policy_weight × situational_bonus`
3. **Pick** — выбирает действие взвешенным рандомом из топ-N скоров
   (не всегда лучшее — иначе предсказуемо; softmax)
4. **Act** — вызывает тот же API, что и живой игрок (через
   internal-функции, не через HTTP)

**Почему взвешенный рандом**: живые игроки не играют оптимально.
Случайность делает бота человечным.

### Profile-ы (типы личности)

Набор готовых политик. Каждый задаёт веса действий и триггеры:

| Profile | Приоритеты | Триггеры атак | Ожидаемый уровень |
|---|---|---|---|
| **Farmer** | build/research 80%, fleet 15%, attack 5% | только если атаковали | Новичок (очки 0-50k) |
| **Miner** | build 60%, research 20%, trade 15%, fleet 5% | никогда | Турбо-экономика (50-200k) |
| **Pirate** | fleet 50%, attack 30%, expedition 15%, build 5% | атакует всех слабее | Агрессор (30-100k) |
| **Raider** | fleet 40%, expedition 30%, attack 20%, build 10% | редко, только «дохлых» | Средний (100-500k) |
| **Warlord** | fleet 40%, attack 30%, research 20%, build 10% | ищет равных | Топ (500k-5M) |
| **Explorer** | expedition 50%, research 25%, fleet 15%, build 10% | редко | Средний (100-300k) |
| **Turtle** | defense 50%, build 30%, research 15%, fleet 5% | никогда | Любой (оборонный) |
| **Merchant** | trade 60%, build 25%, fleet 10%, attack 5% | только оборона | Средний рынок |

Профиль задаёт **структурные предпочтения**, а не жёсткий план. В рамках
профиля бот действует ситуативно.

### Tier-ы (уровень силы)

Бот стартует на tier-1 (свежий игрок) и растёт. Tier привязан к очкам
(`score_total`) и определяет:
- Частоту тиков (tier 1 = раз в 10 мин, tier 5 = раз в 30 сек)
- Лимиты действий за тик
- Агрессивность (высокий tier = больше шанс атаки)
- Качество decision-scoring (высокий tier лучше считает ROI)

Tier **не** даёт ботам чит-ресурсы. Бот растёт как живой игрок —
строит, фармит, воюет. Баланс между скоростью роста и реалистичностью —
настройка воркера.

## Ключевые подсистемы

### Ф.1: База данных и модель бота

**Новые таблицы**:

```sql
-- migration 00XX
CREATE TABLE ai_players (
  user_id UUID PRIMARY KEY REFERENCES users(id),
  profile TEXT NOT NULL,              -- 'farmer', 'pirate', ...
  tier INT NOT NULL DEFAULT 1,        -- 1-5
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  last_tick_at TIMESTAMPTZ,
  next_tick_at TIMESTAMPTZ NOT NULL,
  state JSONB NOT NULL DEFAULT '{}',  -- memory, goals stack, персональные флаги
  disabled_at TIMESTAMPTZ              -- для временного выключения
);

CREATE INDEX ai_players_next_tick ON ai_players(next_tick_at)
  WHERE disabled_at IS NULL;

-- Историческая логика решений для дебага и аналитики
CREATE TABLE ai_decisions (
  id UUID PRIMARY KEY,
  user_id UUID NOT NULL REFERENCES users(id),
  decided_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  action TEXT NOT NULL,               -- 'build', 'attack', 'research', ...
  score NUMERIC(10,2),
  payload JSONB                       -- что именно решили (target, unit_id, ...)
);

CREATE INDEX ai_decisions_user_time ON ai_decisions(user_id, decided_at DESC);
```

**Флаг в users**: `users.is_ai BOOLEAN NOT NULL DEFAULT false` — чтобы
ботов можно было фильтровать в рейтингах, списках, админке.

Боты не отличаются от живых в большинстве таблиц — это **фича**:
один и тот же `fleet/attack.go` работает и для живых, и для ботов.

### Ф.2: Движок принятия решений (`backend/internal/aiplayer/`)

```
backend/internal/aiplayer/
├── engine.go          // главный цикл: Perceive → Score → Pick → Act
├── profiles.go        // profile definitions (farmer, pirate, ...)
├── policies/
│   ├── build.go       // что строить: metal_mine если metal < X, иначе ...
│   ├── research.go    // какую технологию качать
│   ├── fleet.go       // какие корабли строить, куда слать
│   ├── attack.go      // выбор цели, подходит ли момент
│   ├── expedition.go  // когда отправить экспедицию
│   └── trade.go       // когда торговать на рынке
├── scoring.go         // policy_weight × situational_bonus
├── memory.go          // обновление state.memory (recent events, grudges)
├── seed.go            // первичный посев ботов в галактику
└── aiplayer_test.go
```

**Главный цикл** (`engine.go`):

```go
func Tick(ctx, tx, userID) error {
  bot := loadBot(tx, userID)
  world := perceive(tx, bot)             // состояние мира глазами бота

  candidates := []Action{}
  candidates = append(candidates, policies.Build(bot, world)...)
  candidates = append(candidates, policies.Research(bot, world)...)
  candidates = append(candidates, policies.Fleet(bot, world)...)
  // ... etc

  // softmax по score, берём топ-3, кидаем кости
  picked := pickWeighted(candidates, bot.profile.Temperature)
  recordDecision(tx, bot, picked)

  for _, a := range picked {             // обычно 1-3 действия за тик
    a.Execute(ctx, tx, bot)
  }

  scheduleNextTick(tx, bot)              // next_tick_at = now + profile.Interval
  return nil
}
```

**Без внешних зависимостей**. Все решения — это `if/else` и арифметика.

### Ф.3: Политики (policies/*)

Каждая политика = набор **правил + веса**, которые при заданной ситуации
дают список (action, score). Пример для `policies/build.go`:

```go
func Build(bot *Bot, world *World) []Action {
  actions := []Action{}
  for _, planet := range bot.Planets {
    // Приоритет: добывающие здания, если ресурсов мало
    if planet.MetalPerHour < targetMetal(bot.Tier) {
      actions = append(actions, Action{
        Kind: "build",
        Payload: map[string]any{
          "planet_id": planet.ID,
          "building": "metal_mine",
        },
        Score: 100 * bot.Profile.BuildWeight,
      })
    }
    // Энергетический баланс
    if planet.EnergyBalance < 0 {
      actions = append(actions, Action{
        Kind: "build",
        Payload: map[string]any{"building": "solar_plant"},
        Score: 150 * bot.Profile.BuildWeight,  // выше — срочнее
      })
    }
    // Верфь и защита — зависит от профиля
    if bot.Profile == PirateProfile && planet.FleetPower < tierFleetTarget(bot.Tier) {
      actions = append(actions, Action{
        Kind: "build",
        Payload: map[string]any{"building": "shipyard"},
        Score: 80,
      })
    }
    // ... ещё правила
  }
  return actions
}
```

**Главная идея**: политики пишутся в стиле «если X — то рассмотри Y с
весом Z». Сумма весов → softmax → pick. Разнообразие даёт интересную
игру.

### Ф.4: Первичный посев галактики

Воркер `ai-seed` (разовый запуск или при расширении галактики):

```go
func SeedGalaxy(ctx, tx, galaxyNum int) error {
  // Для каждой системы (500 в галактике):
  //   - 0-2 ботов с профилем и позицией
  //   - распределение профилей: 40% farmer, 20% pirate, 15% miner,
  //     10% explorer, 10% turtle, 5% warlord (редкие)
  //   - tier = 1 при посеве, растёт сам через тики
  // Итого ~500-800 ботов на галактику
}
```

Критерии баланса:
- На 1 живого игрока — 10-20 ботов (чтобы галактика не казалась пустой)
- 5% ботов — warlord/raider (для эндгейма)
- 40% farmer/miner — pasture для живых игроков

### Ф.5: Воркер `ai-tick`

Новый event-kind `KindAITick` в event-системе (план 09). В работе:

```go
func aiTickHandler(ctx, tx, e Event) error {
  botIDs := loadBotsDue(tx, now, limit=50)  // batch
  for _, botID := range botIDs {
    if err := aiplayer.Tick(ctx, tx, botID); err != nil {
      log.Warn("ai tick failed", "bot", botID, "err", err)
      continue                                // не валим весь batch
    }
  }
}
```

**Шардирование**: если ботов много (10k+) — разбить по `bot_id % N`
воркеров. План 09 Фаза 3 (shard_key) — prerequisite для масштаба.
На старте — один воркер, тик раз в 30 сек, batch 50.

### Ф.6: Tier-прогрессия и старение

**Рост tier**:
- Каждые 10 тиков сверять `score_total` бота → обновить `tier`
- Tier влияет на `next_tick_at` interval (выше tier = чаще думает) и на
  бонусы в `scoring.go`

**Старение**:
- Если бот проигрывает 3 атаки подряд с потерями >50% флота → tier может
  упасть (но не ниже 1)
- Если бот 30 дней не взаимодействует с живыми (никто не атакует, он
  никого не атакует) → помечается как `stale`, tick interval увеличивается
  в 10× (бот «заснул» — не жрёт ресурсы воркера)
- Если бот умер (потерял все планеты и флот) — `disabled_at=now()`,
  respawn в другой системе через 7 дней с tier-1

### Ф.7: Взаимодействие с живыми

- **Сообщения**: боты шлют предзаготовленные реплики («Атакуем!», «Сдавайся!»,
  «Хочешь торговать?») из пула для профиля. Никакого LLM — random из array.
- **Альянсы**: на старте боты **не** вступают в альянсы. Отложено.
- **Рынок**: боты выставляют лоты по цене `market_avg × (0.9…1.1)` —
  создают органичные колебания курсов.
- **Дипломатия**: если живой игрок атакует бота-farmer → бот повышает
  defense weight и просит помощь у ближайших «союзных» ботов (если два
  бота с одинаковым профилем рядом — мягкий альянс, делятся силами).

### Ф.8: Админка и наблюдаемость

- `/admin/ai-players` — список ботов, фильтр по профилю/tier/галактике,
  кнопка «отключить» (`disabled_at=now()`)
- `/admin/ai-decisions/:bot_id` — последние 100 решений бота с score
  и payload (для дебага: «почему бот решил атаковать?»)
- Метрики: `ai_ticks_per_sec`, `ai_tick_duration_p95`, `ai_errors_per_min`,
  `ai_active_count`, `ai_disabled_count`
- Логи slog: `bot_id`, `profile`, `tier`, `action`, `score`

## Ограничения ИИ (инварианты баланса)

Обязательные инварианты — проверяются в движке бота **до** каждого
действия. Нарушение → действие отклоняется и логируется в `ai_decisions`
с `score=-1`. Дизайн-цель: боты создают активность, но **никогда** не
доминируют над живыми игроками и не ломают экономику.

### И-1. Потолок силы по tier

Ресурсы и флот бота жёстко ограничены его tier-ом:

| Tier | Max score | Max planets | Max fleet power | Max resources | Max tech level |
|---|---|---|---|---|---|
| 1 (новичок) | 50k | 2 | 5k | 500k металл-экв | 5 |
| 2 | 200k | 3 | 30k | 2M | 10 |
| 3 | 500k | 5 | 150k | 10M | 15 |
| 4 | 2M | 7 | 500k | 40M | 20 |
| 5 (warlord) | 5M | 10 | 1.5M | 100M | 25 |

Как применяется:
- **Build** — если пост-стройка выведет ресурсы/силу за потолок → действие
  отвергнуто, выбираем следующий кандидат из списка
- **Research** — то же для tech-уровней
- **Fleet build** — то же для fleet_power
- **Resource accumulation** — если на планетах бота лежит ресурсов больше
  потолка → бот **обязан** их потратить/слить на рынок (иначе цикл
  scoring принудит к dump'у)

**Почему жёстко**: без этого «удачный» бот накопит чит-флот за месяц и
будет убивать новичков одним ударом.

### И-2. Запрет на атаку слабых новичков

Бот **не может** атаковать игрока, у которого:
- `score_total < 0.5 × bot.score_total` — пятикратная разница оценок
- **ИЛИ** `user.created_at > now() - 7d` — новичок первой недели
- **ИЛИ** `user.vacation_mode = true` (защищено планом 20 Ф.1)
- **ИЛИ** уже атакован ботом за последние 5 часов (антибашинг, план 17 A1
  применяется и к ботам)

Проверка в `policies/attack.go::selectTarget` **до** scoring'а —
кандидат-игрок даже не попадает в список.

**Почему**: боты — «пища» для новичков, не наоборот. Если живой
атакован ботом на старте — он бросит игру.

### И-3. Ограничение фарма одного игрока

Бот не может атаковать одну и ту же **цель** (игрок или бот) более
**2 раз в сутки** и более **6 раз в неделю**. Если цель — живой игрок,
лимит ужесточается до **1/сутки, 3/неделя**.

Memory бота хранит лог атак (`state.memory.recent_attacks`) — bounded
queue на 50 записей. После истечения окна запись удаляется.

**Почему**: защита от «бот-зерг раша» на одного игрока. В legacy есть
антибашинг 4/5ч (план 17 A1), но это **per-attacker**. Здесь — **per-target
per-bot** для мягкого ограничения именно ботов.

### И-4. Пониженная эффективность атаки vs живых

При атаке **живого** игрока бот применяет штрафы к своему флоту:
- Атака снижается на **20%** (через multiplier в battle engine, только
  для бот-атакующего)
- Скорость полёта +30% (бот «летит медленнее» — живой успевает
  среагировать, отозвать флот, организовать defense)
- Шпионаж от бота менее информативен (отчёт округлён до порядка:
  «~10k металла» вместо точного числа)

**Почему**: живой игрок должен иметь **ощутимое преимущество** в PvP.
Бот — полигон для тренировки, не полноценный противник.

**Это не применяется** при бот-vs-бот — там боевой движок работает
без штрафов, боты воюют «на равных» между собой.

### И-5. Cap добычи debris

Когда бот уничтожает флот (чей угодно) — он может собрать **максимум
30%** debris. Остальные 70% лежат на поле до прилёта другого флота
(живого или бота).

**Почему**: живым должна быть доступна экономика debris-фарма после
больших боёв. Если бот-пират подчищает всё — живым ловить нечего.

### И-6. Нет читов, чит-ресурсов, спавна-из-воздуха

- Бот **не** получает стартовые ресурсы сверх того, что есть у живого
  игрока при создании аккаунта (см. spec §10 — стартовые ресурсы)
- Бот строит здания/корабли по **тем же формулам** `configs/` и **тем же
  event-handlers**, что и живой
- Бот **не** получает бонусы к производству, скорости, bat-speed
- Tier определяет только **частоту думания** и **softmax temperature** —
  не статы

**Почему**: если хоть один элемент читерский — игроки жалуются,
доверие к игре падает.

### И-7. Рынок — cap на объём и отклонение цены

Бот-merchant торгует по правилам:
- Цена лота: `market_avg × (0.9…1.1)` — не более ±10% от среднерыночной
- Объём одного лота: ≤ **5%** от суточного объёма рынка
- Не более **3 активных** лотов на бота одновременно
- Не может выставить **противоречивые** лоты (продаёт и покупает один
  ресурс одновременно) — защита от wash-trading

**Почему**: бот-merchant должен создавать **органичные** колебания
курсов, а не манипулировать рынком.

### И-8. Expedition-ограничения (особо важно)

Учитывая что экспедиции — эксплойт (см. [balance/audit.md BA-003](../balance/audit.md#ba-003)):
- Бот может отправить экспедицию **не чаще 1 раза в 6 часов** (живой —
  ограничен только `expedition_slots` из плана 20 Ф.7)
- Минимальный флот экспедиции бота — **30%** от его текущего fleet_power,
  не меньше (живому достаточно 10% из плана 21 B1)
- Награды экспедиции бота уходят в «потолок И-1» — если бот уже у max
  resources, награда списывается в 0 (чёрная дыра, для balance)

**Почему**: иначе боты быстро накачают экономику через фарм экспедиций и
сломают И-1.

### И-9. Защита от эксплуатации ботов живыми

Боты не должны быть «бесплатным банкоматом»:
- Бот помнит, кто его фармит. Если живой игрок успешно атаковал бота
  **3 раза за 7 дней** → бот помечает его `state.memory.threat = true`
- Бот с `threat=true` для данного игрока: эвакуирует флот/ресурсы на
  дальнюю планету, перестаёт оставлять ресурсы на орбите, увеличивает
  defense weight
- После 5 успешных атак подряд — бот **respawn**ится в другой галактике
  (живой теряет возможность фармить)

**Почему**: без этого живой-волк найдёт слабого бота и будет его доить
бесконечно — деньги-из-воздуха, ломающие экономику.

### И-10. Глобальные cap-ы на популяцию

- **Total bots per galaxy** ≤ 800 (чтобы не перегружать БД)
- **Bot:player ratio** ≤ 20:1 — если живых мало, часть ботов уходит в
  `disabled_at` (засыпает), не все активны одновременно
- **Max bot actions per hour** ≤ 10 × (tier) — даже tier-5 делает
  максимум 50 ходов/час, как реальный активный игрок

**Почему**: ботов не должно быть «слишком много» или «слишком шумных».

### И-11. Дыры, которые боты обязаны соблюдать как живые

Все эксплойты из [balance/audit.md](../balance/audit.md) действуют на
ботов как на живых:
- BA-003 (экспедиции) — минимум флота + cap наград (И-8 ужесточает ещё)
- BA-006 (антибашинг 4/5ч) — применяется к ботам-атакующим
- И любые будущие — через единый `validateAttack`/`validateExpedition`
- Боты **не** имеют bypass в коде — один и тот же handler обрабатывает
  живых и ботов

**Почему**: если бот может обходить правила — это чит, игрок увидит.

### И-12. Экстренный kill-switch

В админке: кнопка «Отключить всех ботов в галактике X» и «Отключить
всех ботов игры». При срабатывании:
- Все `ai_players.disabled_at = now()` для подмножества
- Активные флоты ботов летят на ближайшую свою планету (graceful shutdown)
- Выставленные лоты на рынке отменяются
- Событие в audit log: `who`, `when`, `reason`

**Почему**: если у бота баг ломает экономику в реальном времени — надо
уметь потушить за секунду, не ждать деплоя.

## Ограничения на поведение (soft, не инварианты)

Это не жёсткие пределы, а политики: они могут быть нарушены рандомом,
но в среднем поведение следует им.

### П-1. «Разумность» действий

Softmax temperature = 0.3–0.7 (в зависимости от tier). Высокий tier —
низкая температура (= почти всегда выбирает лучший ход). Низкий — высокая
(= чаще делает глупости). Новичку-боту простительны ошибки.

### П-2. Время реакции

Бот **не** реагирует на атаку мгновенно (как если бы «был онлайн»).
- Tier 1: реагирует через 2-6 часов (спит, ест)
- Tier 5: 15-60 минут

**Почему**: иначе бот выглядит как постоянно-онлайн-игрок — нереалистично.

### П-3. Суточный ритм

Tier 1-3 боты «спят» ночью UTC (02:00-08:00) — interval тиков ×5.
Tier 4-5 активны круглосуточно.

**Почему**: живые не в сети круглосуточно. Боты имитируют.

## ADR-требования (обновлено)

- **24-A**: принять концепт правило-ориентированного ИИ без внешних LLM
- **24-B**: процент ботов в галактике (ratio ≤ 20:1 vs живых) и
  распределение профилей
- **24-C**: конкретные числа инвариантов И-1 (потолок ресурсов/флота по tier)
- **24-D**: этика — должны ли боты помечаться как AI в рейтинге
- **24-E**: принять штрафы И-4 (−20% attack / +30% flight) при
  бот→живой атаке
- **24-F**: принять cap И-5 (30% debris ботам) — может быть спорно
  (живые тоже получают «чит», обирая ботов)

## Риски и открытые вопросы

1. **Нагрузка на БД и event-систему**. 500 ботов × тик раз в 30с =
   16 tick/s. Каждый тик — 5-20 SQL. Итого 50-300 qps только от ботов.
   Надо тестировать на dev-стенде до полного посева. Возможно, увеличить
   интервалы тиков или шардировать.

2. **Дисбаланс**. Если бот-warlord случайно накопит чит-флот — одним
   ударом убьёт всех новичков. Нужны **инварианты**: бот не может иметь
   ресурсов/флота больше чем `tier × threshold`. Превышение → проверка,
   не эксплуатирует ли бот баг.

3. **Предсказуемость**. Если игроки поймут паттерны ботов — будут их
   фармить тривиально. **Противодействие**:
   - Температура softmax → боты иногда делают странные ходы
   - Рандомная ротация профилей раз в N дней (10%)
   - Memory бота помнит, кто часто атакует → повышает defense / сбегает

4. **Baiting**. Живой игрок может устроить «ловушку»: поставить флот,
   ждать когда бот-pirate клюнет. Решение: scoring учитывает expected
   loss, боты высокого tier умеют считать «опасно — не иду».

5. **Ethic — должны ли боты помечаться явно**. Если бот не отличим от
   живого → игрок обижается когда узнает. Решение: в рейтинге и профиле
   — метка «AI» (серая иконка). Но UI карточки при наведении не пишет
   «это бот» в сообщениях / бою — иммерсия важна.

6. **Тестируемость**. ИИ без seed → флакающие тесты. Решение: каждый
   бот имеет свой `rand.Source` с seed из `bot.ID` → детерминированные
   тики для тестов с фикс-БД.

7. **Golden-stratery exploit**. Если policy-веса идеально прозрачны —
   игроки напишут «оптимальную стратегию против ботов». Решение: веса
   в конфиге (`configs/ai/profiles.yml`), можно менять без деплоя.
   Раз в квартал — ребаланс профилей.

## Как мерить успех

**KPI после посева**:
- Retention новичков +20% (есть с кем играть)
- Активность на рынке +50% (боты торгуют)
- Средняя длина сессии +10% (больше контента)
- Жалобы на мёртвую галактику — исчезают

**Anti-KPI** (если достигнуто — сворачиваем):
- Живые жалуются что «боты везде и не интересно»
- Боты фармят живых новичков → массовый уход
- Нагрузка на БД ≥30% от total qps
