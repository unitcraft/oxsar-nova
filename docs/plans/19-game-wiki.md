# План 19: Вики игры (документация для игроков)

**Дата**: 2026-04-24
**Статус**: ЧЕРНОВИК
**Стек затрагивает**: docs/ (Markdown-генерация), frontend (React, страница /wiki), backend (возможно — эндпоинт выгрузки каталогов)

---

## Цель

Создать игровую вики — публичную документацию по механикам, зданиям, юнитам,
исследованиям, экспедициям, бою, экономике. Аналог [ogame.fandom.com](https://ogame.fandom.com/wiki/OGame_Wiki):
структурированная, со статьями на каждую сущность, навигацией по категориям,
поиском.

**Отличие от CLAUDE.md / oxsar-spec.txt**: те — для разработчиков, этот — для
игроков. Без SQL, без кода, без внутренних имён полей.

---

## Референс: структура OGame Wiki + наши улучшения

За основу берём структуру OGame Wiki (по памяти, fandom возвращает 403),
но дорабатываем под oxsar-nova-специфику и устраняем слабости OGame-вики
(фрагментация, устаревшие числа, отсутствие навигации по контр-юнитам).

### Базовые категории (из OGame)

1. **Getting started** — старт игры, первые шаги, туториал
2. **Resources** — металл, кремний, водород (OGame: металл/кристалл/дейтерий)
3. **Buildings** — здания: описание, формулы, что даёт, зависимости
4. **Research** — технологии и дерево исследований
5. **Ships** — юниты: характеристики, стоимость, роль
6. **Defense** — оборонительные сооружения
7. **Combat** — раунды, щиты, rapidfire, debris
8. **Missions** — типы миссий флота
9. **Expedition** — исходы, вероятности
10. **Galaxy** — координаты, планеты, луны
11. **Alliance** — альянсы
12. **Economy** — рынок, курсы
13. **Account** — регистрация, vacation, premium
14. **Glossary / FAQ** — термины

### Наши улучшения поверх OGame-вики

- **Контр-юниты как граф связей**. Каждая карточка корабля показывает
  «кого контрит» и «кого боится» — рендерится из rapidfire-матрицы в
  `configs/ships.yml`. У OGame это размазано по абзацам.
- **Интерактивный калькулятор стоимости**. В статье здания — поле
  «уровень» → показывает металл/кремний/водород/время до любого уровня
  (чистый JS, без бэка).
- **Живые ссылки между статьями**. Синтаксис `[[metal_mine]]` в md →
  transformer превращает в ссылку с preview-тултипом. У OGame — обычные
  markdown-ссылки, без превью.
- **Версионирование статьи**. В футере: «генерировано из configs @ git-sha».
  Игрок видит актуальность. У OGame — ничего.
- **Калькулятор боя** (ссылка на наш `/battle-simulator`) прямо в статье
  конкретного юнита — готовый пресет с этим юнитом.
- **Симулятор экспедиции** (ссылка на `/expedition-sim` если сделаем) —
  в статье экспедиции.
- **«Что изменилось в v.N»** — changelog баланса. Генерируется из git log
  по `configs/*.yml`. Игрок всегда знает, что поменялось в последнем
  патче. У OGame этого нет в принципе.
- **Прогрессия vs конечные числа**. Таблица первых 10 уровней +
  формула + калькулятор (см. раздел «Числа vs формулы» ниже).
- **Единый поиск по всей вики** (fuse.js) — у OGame поиск хилый.
- **Режим «для новичка» vs «для опытного»**. Тумблер в шапке вики:
  новичку скрываются формулы, показываются только таблицы и туториалы.
  У OGame такого нет, продвинутые термины пугают новичков.

### Новые категории, которых нет у OGame

- **Artefacts** (у нас есть, у OGame нет) — каталог артефактов, эффекты,
  где выпадают.
- **Alien AI** (у нас есть, у OGame нет) — механика HOLDING, тиры
  пришельцев, стратегии обороны.
- **Events** (у нас будет, план 17 блок F) — расписание галактических
  событий.
- **Patch notes** — история изменений баланса и фич, отдельная категория.

## Подход: Markdown-first с генерацией из configs/

**Принцип**: числа живут в `configs/*.yml` (источник истины). Вики-статьи —
это шаблоны (frontmatter + Markdown), в которые подставляются значения из
YAML. Меняется баланс → вики обновляется одним `make wiki-gen`.

**Почему не рукописный Markdown**: будет устаревать. Любое изменение
`configs/ships.yml` должно автоматически отразиться в вики — иначе игроки
увидят неверные числа.

**Почему не vanilla Docusaurus/MkDocs**: хочется интегрировать в сам фронт
(React route `/wiki`), чтобы логин и общий layout работали. Markdown
рендерится через `react-markdown` + frontmatter → шапка статьи.

## Фазы реализации

### Ф.1. Структура и шаблоны (1 день)

- Создать `docs/wiki/ru/` с подпапками: `getting-started/`, `resources/`,
  `buildings/`, `research/`, `ships/`, `defense/`, `combat/`, `missions/`,
  `expedition/`, `galaxy/`, `alliance/`, `economy/`, `account/`,
  `glossary/`, `artefacts/`, `alien/`, `events/`, `patch-notes/`
- `docs/wiki/en/` — каталог создаётся пустой, переводы позже
- В каждой — `index.md` (обзор категории) + заглушки статей
- Шаблон frontmatter:
  ```yaml
  ---
  title: Металлический завод
  category: buildings
  entity_id: metal_mine   # для join с configs/
  order: 10
  ---
  ```
- Шаблон для генерируемых статей: Markdown с placeholders `{{cost(1)}}`,
  `{{prod(10)}}`, `{{deps}}` — подставляются генератором

### Ф.2. Генератор `cmd/wiki-gen` (1-2 дня)

Новая Go-CLI-утилита в [backend/cmd/tools/wiki-gen/](../../backend/cmd/tools/):
- Читает `configs/*.yml`
- Читает шаблоны из `docs/wiki/_templates/`
- Для каждого entity (ship, building, research) — генерирует
  `docs/wiki/<category>/<entity_id>.md` с подставленными числами:
  стоимости на уровнях 1/5/10/15, время постройки, предпосылки
- В `Makefile`: `make wiki-gen` — запускает утилиту
- В CI: проверка что `make wiki-gen && git diff --exit-code` (не забыли
  закоммитить обновлённую вики)

### Ф.3. Статьи ручного контента (3-5 дней, параллельно Ф.2)

Не генерируются — пишутся руками:
- `getting-started/` — 3-5 туториал-статей (первая постройка, первая атака,
  как играть в экспедиции)
- `combat/` — формулы боя, rapidfire-таблица, debris механика (аналитика)
- `missions/` — 8 типов миссий с примерами
- `expedition/` — 12 исходов + вероятности (частично генерируется из
  `fleet/expedition.go::expWeights`)
- `galaxy/`, `alliance/`, `account/`, `glossary/`, `faq/`

### Ф.4. Frontend: route /wiki (2-3 дня)

- `frontend/src/features/wiki/` — новый feature-модуль
- Route `/wiki/:lang(ru|en)?`, `/wiki/:lang/:category`,
  `/wiki/:lang/:category/:slug`. Без `:lang` → redirect на `/wiki/ru`
- Читает `docs/wiki/**/*.md` на build-time через Vite plugin → JSON-индекс
- Layout: left sidebar (категории), content (react-markdown), right TOC
- **Калькулятор стоимости**: React-компонент `<CostCalculator>`, MDX-блок
  или кастомный transformer `::: calculator building="metal_mine" :::`
- **Контр-юниты**: генератор Ф.2 кладёт в frontmatter `counters: [...]`
  и `countered_by: [...]`, компонент `<CounterGraph>` рендерит
- **Переключатель языка** в шапке (ru/en), en — fallback на ru с баннером
- **Режим «для новичка»** — toggle в localStorage, `<details>`-блоки с
  формулами сворачиваются
- Стилизация — Space Deco Minimal
- Поиск: клиентский (fuse.js), по заголовкам, тегам и телу

### Ф.5. Подключение к игре (1 день)

- Кнопка «?» рядом с названием здания/юнита/технологии → открывает
  `/wiki/<category>/<entity_id>` в модалке или новой вкладке
- Ссылки между статьями работают через `[[entity_id]]` → transformer в md

### Ф.6. SEO и публичный доступ (опционально, 1 день)

- `/wiki` доступен без логина (сейчас все роуты под auth)
- Sitemap.xml, meta-теги (title, description) из frontmatter
- Статическая сборка вики для Google (опционально, позже)

## Решённые вопросы

### 1. Числа vs формулы — **Вариант C: таблица + формула + калькулятор**

Каждая статья здания/технологии/юнита содержит три блока:

- **Таблица первых 10 уровней** — металл/кремний/водород/время. Удобно
  новичку, сразу видит прогрессию. Генерируется из `configs/`.
- **Формула** (`cost = base × factor^level`) в разделе «Как считается» —
  для опытных. Объясняет, почему 11-й уровень такой дорогой.
- **Интерактивный калькулятор** — `<input type="number">` для уровня →
  JS подставляет в формулу, показывает стоимость любого уровня до 100.
  Чистый frontend, без API.

Почему так: новичок смотрит таблицу, опытный — формулу и калькулятор.
MVP не тянет тяжёлых зависимостей (chart.js — отложено до Ф.6+).

### 2. Локализация — **ru + en структурно, реализуем только ru**

- Frontmatter: `lang: ru` в каждом файле
- Пути: `docs/wiki/ru/<category>/<slug>.md` — каталог `ru/` готов сразу
- Роут: `/wiki/:lang(ru|en)/:category/:slug` — `en` пока даёт 404 или
  fallback на ru с баннером «English version coming soon»
- Язык-переключатель в шапке вики уже в MVP (переключает только локаль
  UI, контент тот же)
- Когда придёт переводчик — просто копирует `ru/` → `en/` и переводит

### 3. Скрытая информация — **показываем всё, кроме seed-механик**

Решение: oxsar-nova — малая игра, прозрачность важнее сакральной тайны.
Игрок, понимающий механику, играет с удовольствием.

**Показываем**:
- Все веса исходов экспедиции (из `fleet/expedition.go::expWeights`) —
  «60% ресурсы, 30% астероиды, 10% что-то ещё»
- Шансы артефактов по тирам
- Формулы боя, rapidfire-матрицу
- Механику HOLDING, как платить пришельцам
- Все коэффициенты стоимости и производства

**Скрываем**:
- Конкретный `seed` выпавшего лута (очевидно)
- Внутренние id событий, UUID — нечитаемо
- Детали anti-cheat (rate-limit, fraud-detection) — для безопасности

Если окажется что прозрачность портит геймплей — добавим тумблер
«Показывать вероятности» по умолчанию выключенный. Но сейчас default on.

### 4. У легаси нет собственной доки

Проверено: в `d:\Sources\oxsar2\www\` нет ни `doc/`, ни `wiki/`,
ни таблицы `na_wiki`. Писать с нуля, без базы для адаптации.

### 5. Версионирование статей

В футере каждой статьи: «Сгенерировано из `configs/` @ `<git-sha-short>`,
дата сборки `<YYYY-MM-DD>`». При смене баланса — видно, актуальна ли
статья. Дополнительно `docs/wiki/patch-notes/` — ручная история
крупных изменений баланса.

## Что НЕ делаем в этом плане

- Не делаем пользовательское редактирование (вики на Fandom позволяет
  редактировать — у нас read-only, контент от разработчиков)
- Не делаем комментарии/обсуждения под статьями
- Не портируем all-in-one — сначала MVP (buildings + ships)
- Не интегрируем AI-ответы на вопросы (это план 06, AI-советник)
