## 1. Контекст

Аудит 72.1.2 §«Battlestats» зафиксировал критическое расхождение:

Legacy [`Battlestats.class.php::showBattles`](../../projects/game-legacy-php/src/game/page/Battlestats.class.php#L52)
принимает 12 параметров фильтрации:

```
date_min, date_max, user_filter, alliance_filter,
show_no_destroyed, show_aliens, new_moon, show_drawn,
sort_field, sort_order, page, moon_battle
```

В origin **5 фильтров отображаются в UI** (`dateFirst, dateLast,
userFilter, allianceFilter, showDrawn, showAliens`), но **в API
не передаются**. Backend [`ListMine`](../../projects/game-nova/backend/internal/battlereport/handler.go#L55)
принимает только `cursor + limit`. Все WHERE-условия hardcoded.

**Это полностью фейковый UI** — юзер вводит userFilter, ничего не
происходит.

## 2. Цель

Привести фильтрацию к семантике legacy + связать UI с API.

## 3. Декомпозиция

### A. Backend: расширить `GET /api/users/me/battles`

Файл: [`internal/battlereport/handler.go::ListMine`](../../projects/game-nova/backend/internal/battlereport/handler.go).

Параметры запроса:
- `date_min, date_max` (RFC3339) — фильтр по `at`.
- `user_filter` (uuid) — показать только бои с указанным
  оппонентом (`attacker_user_id == X OR defender_user_id == X`,
  при условии что юзер тоже участник).
- `alliance_filter` (uuid) — оппонент в указанном альянсе. Требует
  JOIN с users по `attacker_user_id`/`defender_user_id` →
  alliance_id.
- `show_drawn` (bool, default true) — показать ничьи (winner='draw').
  При false — `winner != 'draw'`.
- `show_aliens` (bool, default false) — показать бои с aliens.
  Aliens — это бои с `IsAliens` стороной. У нас в `battle_reports`
  нет явной отметки; нужна либо новая колонка, либо поле в JSON
  report. **Решение**: добавить колонку `battle_reports.has_aliens
  boolean` с миграцией; заполняется в `ApplyBattleResult`-time
  (если report.Attackers/Defenders содержит SideResult с
  IsAliens=true).
- `show_no_destroyed` (bool, default true) — скрыть бои без потерь
  (`(loot_metal + loot_silicon + loot_hydrogen) == 0 AND
  (debris_metal + debris_silicon) == 0`).
- `new_moon` (bool, default false) — только бои, в которых
  появилась луна. Поле есть в `Report.MoonCreated`. **Решение**:
  колонка `battle_reports.moon_created boolean`.
- `moon_battle` (bool, default false) — только бои на луне.
  Поле есть в `Report.IsMoon` (через первый round_trace или
  отдельно). **Решение**: колонка `battle_reports.is_moon boolean`.
- `sort_field` (`date|rounds|debris|loot`) — sort по соотв.
  колонке.
- `sort_order` (`asc|desc`) — направление.

Cursor pagination оставляем как есть (modernization vs legacy
page-based — backwards compat не нарушается, page-based просто
не используется).

**Миграция** (новая `0090_battle_reports_filter_columns.sql` —
номер уточнить tail'ом миграций):
```sql
ALTER TABLE battle_reports ADD COLUMN IF NOT EXISTS
    has_aliens boolean NOT NULL DEFAULT false;
ALTER TABLE battle_reports ADD COLUMN IF NOT EXISTS
    moon_created boolean NOT NULL DEFAULT false;
ALTER TABLE battle_reports ADD COLUMN IF NOT EXISTS
    is_moon boolean NOT NULL DEFAULT false;

CREATE INDEX IF NOT EXISTS ix_battle_reports_has_aliens
    ON battle_reports (has_aliens) WHERE has_aliens = true;
CREATE INDEX IF NOT EXISTS ix_battle_reports_moon_created
    ON battle_reports (moon_created) WHERE moon_created = true;
```

Backfill: для существующих записей колонки = false (default).
Это допустимо — старые записи будут «не в фильтре new_moon»,
что не блокер.

Заполнение колонок: в местах вставки `battle_reports`
(fleet/attack.go, acs_attack.go, expedition.go) — добавить
параметры в INSERT.

## 4. Ограничения

- pgx без sqlc.
- Без новых абстракций — фильтры передаются как параметры
  query-функции, не отдельный сервис.
- i18n строгий (без `??` fallback).
- Backwards-compat: cursor-based pagination остаётся, page-based
  не вводим.
- Производительность: индексы на `has_aliens`, `moon_created`
  (partial WHERE = true) дешёвые. На `at DESC` индекс уже есть.
