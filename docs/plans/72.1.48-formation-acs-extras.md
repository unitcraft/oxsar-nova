# План 72.1.48 — Formation: ACS-группы post-factum + invite + naming

**Дата**: 2026-05-02
**Статус**: ✅ Закрыт (без AutoMsg — в follow-up)

**Подплан** [плана 72.1](72.1-post-remaster-stabilization.md) §20.12,
закрывает Mission `formation` action (legacy
`Mission.class.php:1750-1798`).

## Контекст

После 72.1.47 в /mission реализованы: holding, load/unload, recall,
StarGate, resource-carry, ACS dispatch. Остаётся **legacy `formation`
action** — конверсия single-атаки в ACS post-factum + именование +
invitation через Relation.

Legacy схема:
- Игрок отправил `EVENT_ATTACK_SINGLE` → жмёт «formation»
- Backend меняет `events.mode = EVENT_ATTACK_ALLIANCE` (UPDATE на
  лету). Создаёт `attack_formation (eventid, name, time)` и
  `formation_invitation (eventid, userid)`.
- На отдельной странице — invite по username с проверкой Relation
  (NAP/Ally/Diplomatic).

**Архитектурный конфликт с nova**: legacy мутирует `events.kind`,
у нас events immutable. Решение — гибрид:
- НЕ мутируем существующее событие.
- Создаём `acs_groups` запись с именем + invitation-таблицу.
- При **новом** `Send` с `acs_group_id` — присоединение синхронизирует
  `arrive_at` под существующий event группы (это уже работает в
  `resolveACSGroup`).
- Конверсия single→ACS = создание ACS-группы для уже летящего флота
  + отправка приглашений (новые флоты через стандартный Send).
