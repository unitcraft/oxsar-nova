# План 72.1.48 — Formation + control-fleet rate-limit + комиссия + back_consumption

**Дата**: 2026-05-02
**Статус**: ✅ Закрыт

**Подплан** [плана 72.1](72.1-post-remaster-stabilization.md) §20.12,
закрывает Mission `formation` action (legacy
`Mission.class.php:1750-1798`).

## Контекст

После 72.1.47 в /mission реализованы: holding, load/unload, recall,
StarGate, resource-carry, ACS dispatch. Остаётся **legacy `formation`
action** — конверсия single-атаки в ACS post-factum + именование +
invitation через Relation.

Legacy схема:
- Игрок отправил `EVENT_ATTACK_SINGLE` → жмёт «formation»
- Backend меняет `events.mode = EVENT_ATTACK_ALLIANCE` (UPDATE на
  лету). Создаёт `attack_formation (eventid, name, time)` и
  `formation_invitation (eventid, userid)`.
- На отдельной странице — invite по username с проверкой Relation
  (NAP/Ally/Diplomatic).

**Архитектурный конфликт с nova**: legacy мутирует `events.kind`,
у нас events immutable. Решение — гибрид:
- НЕ мутируем существующее событие.
- Создаём `acs_groups` запись с именем + invitation-таблицу.
- При **новом** `Send` с `acs_group_id` — присоединение синхронизирует
  `arrive_at` под существующий event группы (это уже работает в
  `resolveACSGroup`).
- Конверсия single→ACS = создание ACS-группы для уже летящего флота
  + отправка приглашений (новые флоты через стандартный Send).

## Дополнение: control-fleet (доделка от 2026-05-02 поздно)

Первоначальное закрытие 72.1.47 ввело `/load` и `/unload`, но
без legacy-семантики `controlFleet`. Re-audit показал, что в legacy
эта страница не «mutation курса» (моё ошибочное предположение), а
**rate-limit + back_consumption check + комиссия** на load/unload.

### Definition of Done (доп.)

- [ ] **Rate-limit `control_times`** (legacy `holding_event.data.
   control_times` + `getRemainFleetControls`):
   - Каждый holding-флот хранит `control_times` (jsonb-поле в event
     payload). Каждый успешный load/unload инкрементит.
   - Лимит: `MAX_FLEET_CONTROLS` (legacy в `config/consts.php` —
     уточнить точное значение, fallback = 10).
   - При превышении — `409 ErrControlsExhausted`.
- [ ] **Back-consumption check** (legacy `data.back_consumption =
   ceil(consumption / 2)` + проверка `hydrogen >= back_consumption`):
   - В payload holding-event'а при отправке — рассчитываем `consumption`
     (gas total flight) и `back_consumption = ceil(consumption / 2)`.
   - При unload H проверяем что после операции `carried_hydrogen >=
     back_consumption` иначе отказ — `409 ErrInsufficientReturnFuel`.
   - Это защищает от выгрузки H, после которой флот не вернётся.
- [ ] **Комиссия (legacy `getControlComis`)**:
   - 5% если флот на **своей** планете (planets.user_id = fleet.owner).
   - `(user.exchange_rate - 1) × 100%` если на чужой (с холдинг-доступом).
   - При load/unload — реальный transfer = X, debited на планете
     (load): X + ceil(X × comis/100); credited на флот: X. На unload
     наоборот: debited флот X; credited планета X − ceil(X × comis/100).
- [ ] UI:
   - badge «осталось контролей: N» в FleetRow для hold-флотов.
   - tooltip на load/unload-кнопках с показом будущей комиссии
     (`X × (1 + comis/100)`).

### Прогресс (доп.)

- ✅ Миграция 0095_fleet_control_times добавляет 3 поля в fleets:
  control_times (счётчик), max_control_times (лимит при отправке),
  back_consumption (резерв H).
- ✅ `transport.go::Send` для KindHolding вычисляет:
  - `back_consumption = ceil(totalConsume × dist / 35000)` —
    минимальный H на одно плечо (упрощение — nova не списывает fuel
    с планет вообще; формула используется только для unload-проверки).
  - `max_control_times = 1 + comp_tech_owner / 6` (legacy
    `NS::getMaxFleetControls` для своего планета-владельца;
    дополнение от location-владельца **не учтено** — на момент Send
    мы не знаем, кто владеет dst, на ту же позицию могло переехать).
- ✅ `LoadResources`:
  - Чтение `control_times`/`max_control_times` под FOR UPDATE.
  - Pre-check `controlTimes >= max_controls` → `ErrControlsExhausted`.
  - Расчёт комиссии через `getControlCommission` (default 1.2 →
    20%). `maxFromStock(s) = s × 100 / (comis + 100)` clamp planet stock.
  - Реальное списание = requested + ceil(requested × comis / 100).
  - Инкремент `control_times` атомарно с зачислением во flot.
- ✅ `UnloadResources`:
  - Те же check'и + **back_consumption check**: `(carried_h - unload_h)
    >= back_consumption` → иначе `ErrInsufficientReturnFuel`.
  - Реальное зачисление на планету = requested - ceil(requested × comis / 100).
  - Инкремент `control_times`.
- ✅ HTTP errors: `ErrControlsExhausted`, `ErrInsufficientReturnFuel`
  → 409 Conflict.
- ✅ Fleet DTO: новые поля `control_times`, `max_control_times`,
  `back_consumption` (omitempty); `List` запрос их выбирает.
- ✅ FE: badge в state-колонке для hold-флотов с показом
  «Контролей: N/M · Резерв H: X».
- ✅ go vet + go test (все пакеты ok) + tsc + vitest 133/133 ✅.

**Известные упрощения vs legacy 1:1**:
- back_consumption формула проще legacy (не учитывает speed_percent
  и tech-modifiers). Поскольку nova вообще не списывает fuel — это
  «нижняя оценка», достаточная для unload-проверки.
- max_control_times не пересчитывается в runtime при изменении
  comp_tech (legacy тоже не пересчитывает — это значение время
  отправки в legacy фиксируется в payload event'a).
- Комиссия применяется **всегда** к своему holding (legacy комментарий
  показывает что 5%-ветка для своего была закомментирована, активна
  только exchange_rate). Совпадает с production legacy.
