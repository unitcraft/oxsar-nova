# План 72.3 (ремастер): legacy → game-nova импорт игрового профиля

**Дата**: 2026-04-29
**Статус**: ✅ Завершён
**Зависимости**: план 72.1 (uni03 Classic — legacy-баланс), 72.2 (handoff
для входа в game-фронт), план 87 (utility `import-legacy-user`,
разработана в `main`-ветке).

## Контекст

После плана 72.1 у нас есть **uni03 «Classic»** — отдельная dev-вселенная
с legacy-PHP override-балансом (`configs/balance/uni03.yaml`). После
плана 72.2 — рабочий handoff с портала. Чтобы проверить **визуальное
соответствие** origin-фронта (на uni03) с legacy-PHP UI, нужны
семантически эквивалентные данные у юзера в обеих БД.

В `main`-ветке (планы 87-92) для этого разработана утилита
`import-legacy-user` — копирует юзера `test` (legacy MySQL, userid=1)
в identity-db и game-nova-db. Наша ветка `remaster-finalization`
ответвилась **до** этих коммитов, утилиты в ней не было.

## Решение

Cherry-pick утилиты из последнего стабильного коммита main
(`c79817ecc5` — после P89.0.1 hotfix про global_credits) — без
переноса плана 87/92 (они описывают разработку, нам нужен только
готовый код).

Запуск с переопределением dev-DSN под наш стек:
- `--legacy-db='root:root@tcp(127.0.0.1:3306)/oxsar_db'` —
  legacy MySQL (контейнер `oxsar2-mysql-1`, не `game-legacy-php`).
- `--game-db='postgres://oxsar:oxsar@127.0.0.1:5438/oxsar?sslmode=disable'`
  — uni03-db (host-port 5438), не uni01-db (5433).
- `--target-universe=uni03`.
- `--target-username=test`, `--target-password=test1234`.

## Что делает утилита

`projects/game-nova/backend/cmd/tools/import-legacy-user`:

1. **Подключение** к legacy MySQL + identity-db + game-nova-db.
2. **Загрузка legacy-юзера** по `na_user.userid` (default 1):
   user, planets, galaxy cells, buildings, researches, ships
   (включая defense+rockets — сплитятся по unit_id), artefacts,
   alliance membership, notepad, messages.
3. **Cleanup**: удаляет существующего юзера с `username=target-username`
   в обеих БД (по id, cascade).
4. **Identity**: INSERT в identity.users с argon2id-хешем пароля.
   Email = `<username>@test.local`. Universe membership добавляется в
   `identity.universe_memberships` для `target-universe`.
5. **Game-nova**: новый UUIDv7 user, перенос всех 7 разделов с
   маппингом legacy unit_id → nova unit_id (`mapping.go`,
   ~150 entries).

**Идемпотентность**: повторный запуск удалит ранее импортированного и
зальёт заново.

**Маппинг** (`mapping.go`):
- buildings: 18 типов (legacy id 1-18) → nova names (metal_mine, etc.)
- moon-buildings: 7 типов (id 41-47).
- research: 21 тип.
- ships: 24 типа.
- defense: 8 типов.
- rockets: 2 типа.
- planet shields: 2 типа.
- artefacts: ~14 (часть legacy типов **не в nova catalog** — пропускаются
  с WARN'ом).
- professions: 5.

`mapping_test.go` — 11 тестов проверяют что каждый target nova unit_id
существует в `configs/units.yml`/`artefacts.yml`/`professions.yml`.

## Дополнение: эталонный legacy-php стенд для pixel-perfect сравнения

Чтобы сверять origin-фронт с эталоном (план 72 — pixel-perfect клон
legacy), нужно поднять наш `projects/game-legacy-php` стенд с
test-юзером, чьи данные импортируются параллельно в uni03 через
`import-legacy-user`. Стенд работает на http://localhost:8092 и должен
показывать **тот же UI**, что origin-фронт стремится воспроизвести.

### Что делает стенд

`projects/game-legacy-php/docker/docker-compose.yml`:
- mysql 5.7 (host-port 3307) — изолированная БД, не пересекается с
  `oxsar2-mysql-1` (внешний oxsar2).
- php-fpm (наш PHP-порт legacy oxsar2 в clean-room rewrite, миграции
  001-005, своя схема).
- nginx (host-port 8092).
- memcached.
- event-monitor (CLI worker для legacy-events).

### Запуск

```bash
docker compose -f projects/game-legacy-php/docker/docker-compose.yml \
  -p legacy-php up -d --build
```

После старта миграции 001-005 применяются автоматом
(volume `migrations/:/docker-entrypoint-initdb.d:ro`).

### Засеять test-юзера

Снэпшот legacy-юзера лежит в
`projects/game-legacy-php/migrations/fixtures/test-user-snapshot.sql`
(36М очков, 7 планет, 64 постройки — **те же данные** что и в
`oxsar2-mysql-1` userid=1, источник для импортера). Применяется
скриптом:

```bash
bash projects/game-legacy-php/tools/apply-test-user-fixture.sh
```

### Известные проблемы (TODO)

При попытке поднять стенд 2026-04-29 столкнулись с:

1. **Скрипт `apply-test-user-fixture.sh` хардкодит имя контейнера**
   `docker-mysql-1` (старое имя из проекта `-p docker`). При запуске
   с `-p legacy-php` контейнер `legacy-php-mysql-1` и скрипт падает.
   Нужно: добавить `${CONT:-docker-mysql-1}` в скрипт, чтобы можно было
   через env переопределить.

2. **Snapshot-файл имеет ссылку на удалённую таблицу `na_referral`**
   (миграция 005 удалила её). Apply падает на line 333. Workaround —
   ручной apply через `docker exec mysql < snapshot.sql` (применяется
   до точки ошибки, ключевые таблицы `na_user`/`na_planet`/
   `na_buildings` залиты).

3. **PHP-стенд redirect-loop `/login → /login?error=NO_ACCESS`**.
   Корневая причина — backend ожидает identity-handoff (план 36
   IDENTITY_JWKS_URL), но запущен без него (env пустой). Нужно либо:
   - подключить наш identity-service в legacy-php (проброс
     IDENTITY_JWKS_URL через docker network),
   - либо настроить **standalone-режим** legacy-php с локальным auth
     (без identity), как было до плана 36.

   Это **отдельная задача**, не блокирует план 72.3 — для целей
   импортера достаточно прямого SQL-доступа к `oxsar2-mysql-1`.
   Pixel-perfect сравнение origin vs legacy-PHP отложено до того
   момента, когда будет починен auth-flow legacy-php стенда.
