# 72.1.58 — ProfessionScreen 1:1 с legacy

**Дата:** 2026-05-03
**Контекст:** Smoke origin :5176 vs legacy :8092 показал расхождения на
`/profession` (S-041). Делаем полный 1:1 с legacy
`templates/standard/profession.tpl` + `Profession.class.php`.

## Расхождения, выявленные на скриншотах

1. **Заголовок секции эффектов** — origin `«Текущая профессия»`,
   legacy `«Специализация»` (`PROFESSION_SPECIALISATION`).
2. **Описание профессии** отсутствует в origin — в legacy перед
   списком эффектов «Шахтер специализируется на добыче ресурсов»,
   «Универсал не имеет специализации, все постройки и исследования
   работают в нормальном режиме».
3. **Названия эффектов** — origin показывает упрощённые ярлыки
   («Скорость постройки» дважды), legacy показывает имена из catalog
   (`metalmine` → «Рудник по добыче металла», `gun` → «Оружейная
   технология», `shipyard` → «Верфь», `defense_factory` → «Оборонный
   завод»). Из-за упрощения у Атакующего получается дубль «Скорость
   постройки +1 / -3» без различия (один = `shipyard`, другой =
   `defense_factory`).
4. **Универсал** — legacy включает опцию `Универсал` (без
   специализации). У origin её нет.
5. **Порядок** — legacy: Универсал → Шахтёр → Атакёр → Защитник → Танк.
   Origin: Шахтёр → Атакующий → Защитник → Танк (random/yaml-order).
6. **i18n-ключи** `[profession.changeFreeInfo]` отображается как
   raw-placeholder (ключ есть, но не находится).
7. **Английский error-текст** `profession: cannot change profession yet
   (14 day cooldown)` приходит от backend (sentinel `ErrChangeTooSoon`),
   хотя в текущем коде он dead — возможно у пользователя запущен
   старый бинарь `server.exe` от 2 мая. Удалим dead code + переведём
   все sentinel-ошибки.

## Tasks

### T1. Backend (configs + service)

- `configs/professions.yml`: добавить `universal` как первую опцию:
  ```yaml
  universal:
    label: "Универсал"
    description: "Универсал не имеет специализации, все постройки и исследования работают в нормальном режиме."
    bonus: {}
    malus: {}
  ```
- Менять порядок ключей в YAML — нужен явный sort на стороне
  service.go или handler'а (Go map iteration не упорядочена). Решение:
  добавить поле `sort_order` в `ProfessionSpec` (или захардкодить
  порядок в handler через slice).
- Удалить dead `ErrChangeTooSoon` в `service.go` + случай в
  `handler.go`.
- `internal/profession/service.go::List` (если есть) или
  `handler.go::ListProfessions` сортирует по `sort_order` перед
  возвратом.

### T2. i18n + sentinel ошибки

- Проверить наличие `profession.changeFreeInfo` в ru/en — если
  отсутствует, добавить:
  ```yaml
  changeFreeInfo: "Сейчас можно изменить профессию бесплатно. В следующий раз изменить можно будет через {{days}} дней или за {{cost}} кредитов в любое время."
  changeCostInfo: "Стоимость смены профессии: {{cost}} кредитов (бесплатно через {{days}} дней)."
  ```
- Backend handler: вместо `err.Error()` (английский raw текст)
  возвращать `code` (machine-readable), фронт переводит:
  - `unknown_profession` / `not_enough_credit` / `vacation_mode`.
  - HTTP body: `{"code": "vacation_mode", "message": "..."}` (текст
    оставить для обратной совместимости / логов).
- Frontend `ProfessionScreen` маппит `code` → i18n key через locale.

Альтернатива (проще): handler пропускает err.Error() как fallback,
но **i18n bundle в backend** (уже есть в officer.go) переводит на язык
юзера и пишет в response `message`. По образу `officer.go` —
выбрать `i18n.LangRu` если нет language-detection.

Решение: **первый вариант** (codes), чище для CI и e2e-тестов.

### T3. Имена эффектов 1:1

- Удалить `BONUS_LABEL` map (захардкоженный сокращённый словарь).
- Использовать **catalog-имена** напрямую: `cat.Buildings[id].Name`
  / `cat.Research[id].Name` через i18n.
- Backend подход: расширить `Profession` DTO так, чтобы каждый эффект
  был объектом `{ key, label_i18n_key, value }` где `label_i18n_key`
  указывает на ключ в i18n (например `buildings.metalmine.name`).
- Frontend получает готовые `label` (уже на языке юзера).
- Backend service.go при сборке profession DTO делает lookup по
  catalog по ключу и копирует label.

Альтернатива: frontend имеет уже свой catalog (Zustand или
TanStack Query на /api/catalog), берёт оттуда — отрицательная
сторона: дубль catalog и backend, и frontend. Лучше: единый source —
backend.

Решение: **backend составляет label**, отдаёт готовый текст в DTO.

### T4. UI рефакторинг

- `ProfessionScreen.tsx`:
  - Заголовок секции `currentLabel` → `specialization`
    (`PROFESSION_SPECIALISATION` в legacy).
  - Render `description` рядом с лейблом профессии (legacy кладёт его
    в `<label for=profession_id>`).
  - Универсал — render обычно, но если `bonus`/`malus` пустые →
    скрыть `<table>` со специализацией (как в legacy).
