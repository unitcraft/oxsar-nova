## Цель

Дать **обоснованный ответ** на вопрос: «можно ли переиспользовать
backend game-nova как движок для legacy-вселенной oxsar2-classic
(вместо переписывания PHP-кода game-origin на отдельный Go-сервис)?»

**Что план НЕ делает:**

- Не переписывает game-origin.
- Не меняет game-nova.
- Не проектирует legacy-режим.
- Не пишет код вообще.

**Что план делает:**

- Картирует расхождения механик между game-origin (PHP, ~43 500 строк
  игровой логики) и game-nova (Go).
- Оценивает объём изменений в game-nova, нужных для поддержки
  legacy-режима.
- Даёт grade-карту по 5 категориям и итоговый вердикт: feasible /
  infeasible / точечная замена.
- **Составляет полный детальный журнал расхождений** (`divergence-log.md`)
  как актив для будущей работы — не одноразовое решение, а
  справочник, к которому можно вернуться.
- Готовит исходник для последующего плана (полное переписывание ИЛИ
  legacy-режим в game-nova ИЛИ оставить PHP как есть).

### Два deliverable

План производит **два разных артефакта**, оба нужны:

1. **`feasibility-report.md`** — короткая (1-2 страницы) сводка для
   принятия решения. Вердикт A/B/C, оценка недель/месяцев.
   Используется один раз — для выбора следующего шага.

2. **`divergence-log.md`** — длинный (десятки–сотни строк) технический
   справочник всех расхождений: каждая строка с указанием file:line
   в обеих кодовых базах + конкретным предложением «как сделать
   nova-конфигурируемой для эмуляции origin». Используется
   многократно — как чек-лист при реализации legacy-режима, как
   контекст для будущих агентов, как источник тикетов.

Без второго артефакта план превратится в выводное «да/нет» без
сохранённого знания, и при возврате к задаче через месяц придётся
повторять анализ. Журнал расхождений — то, ради чего стоит вообще
делать исследование.

## Контекст

### Текущая ситуация

- **game-origin** — PHP-проект, ~43 500 строк игровой логики (без core/
  переписанного в плане 43). Готовый продукт, работает, юридически
  чист (план 50 почти закрыт). 0 живых игроков.
- **game-nova** — Go-движок, переписан с нуля на современный стек.
  Бой портирован (план 01), event-loop с надёжностью/наблюдаемостью
  (план 09), мультивселенная (план 36). По словам автора — «механика
  существенно переработана и перебалансирована».
- **Цель пользователя** — запустить **classic-режим** на современном
  стеке (Go + React + event-loop в стиле nova), сохранив legacy-баланс
  и legacy-UI.

### Архитектурные вводные (зафиксированы заранее)

Эти решения **уже приняты** и не пересматриваются в плане 62 —
они формируют контекст, в котором ведётся исследование:

1. **Cross-universe сервисы единые.** Identity (51, 52), billing
   (38, 42, 54), portal, reports (56), mail (57), admin-bff (53),
   реферальная программа (59), moderation (46, 48) — общие для
   всех вселенных, включая origin. Legacy-PHP-аналоги этих сервисов
   **не портируются**, заменяются вызовами существующих API.
2. **Origin-фронт = новый React-проект на nova-конвенциях
   (стек), но с pixel-perfect клоном legacy-UI (визуал).**
   - **Стек:** TanStack Query, Zustand, OpenAPI-клиент, TS strict,
     JWT, WebSocket, TipTap (MIT) для редакторов.
   - **Визуал:** воспроизводим legacy-вёрстку **попиксельно** на
     старте — те же layout'ы, позиции элементов, шрифты, цвета,
     иконки, табличные структуры. Smarty .tpl — это **референс
     для воспроизведения**, не для кодового переноса.
   - **Nova-frontend компоненты НЕ переиспользуются** для origin.
     У origin свой набор React-компонентов, визуально клонированных
     с .tpl. Общими между origin и nova остаются только:
     инфра-обёртки (auth/billing/reports/mail-клиенты), редактор
     TipTap, OpenAPI-сгенерированный клиент.
   - **Новшества (адаптив, тёмная тема, улучшенный UX,
     перепроектированные экраны) — после старта**, не в первой
     итерации.
3. **Origin-фронт сразу использует nova-API**, без backend-адаптеров.
   Расхождения только в семантике endpoint'ов (новые DTO-поля или
   отсутствующие endpoint'ы), не в именах путей.
4. **PolyForm Noncommercial 1.0.0 + clean-room** (план 43) — лицензия
   допускает любое переписывание origin-кода.
5. **Балансовый и механический паритет — per-universe.**
   - **Для modern-вселенных** (uni01, uni02 и будущие): истина =
     текущая game-nova (YAML, Go-код, ребалансы планов 17/18/20/21).
     **Не пересматривается.** Если в origin Bomber RF=20, а в
     nova=12 (план 18) — nova остаётся 12 для uni01/uni02.
   - **Для legacy-вселенной** (legacy01): истина = game-origin
     (`na_construction` + `config/consts.php` + поведение legacy-кода).
   - **Расхождения origin↔nova** — это **не «nova ошиблась, надо
     вернуть»**, а **«параметризовать в nova то, что сейчас
     захардкожено, чтобы legacy.yaml/legacy-режим мог переопределить»**.
     Каждая запись D-NNN в журнале формулируется как «какой ключ
     конфигурации/код-путь нужен для поддержки legacy-числа», а не
     «исправить число в nova».
6. **«Общий знаменатель» — только новые механики, не числа.**
   - Фича отсутствует в nova (биржа артефактов, турниры, телепорт
     планеты, дипломатия с полным набором статусов и т.п.) → **добавить
     в nova-backend**, чтобы legacy01 её получила. Эта же фича может
     быть включена/выключена флагом для других вселенных.
   - Число/формула отличается между origin и nova → **не правим
     nova**, выносим в конфиг/код-путь под профилем balance.
7. **Pixel-perfect клон UI — от game-origin (тема `standard`),
   первая итерация — только русский язык.** Источник для replication:
   `projects/game-origin/src/templates/standard/*.tpl` +
   `public/css/*.css` + `public/images/`. JS-логика legacy не
   переносится — поведение переписывается React-средствами; шрифты
   и иконки переиспользуем (с проверкой лицензий).
8. **Тестирование паритета — screenshot-diff** (Playwright +
   visual regression). Эталоны снимаются с запущенного game-origin
   (тема standard) до начала переписывания и фиксируются в репо.
9. **legacy01 — третья вселенная**, поднимается на отдельном
   поддомене (имя по ADR-0010 — открытый вопрос). Свой Vite-bundle
   (origin-фронт), не темизация nova-frontend. CORS / ALLOWED_ORIGINS
   расширяются.
10. **Mail-service (план 57) — до старта origin-фронта.** TipTap
    из 57 переиспользуется как редактор чата (минимальный режим) и
    почты альянса (полный режим) в обеих вселенных.
11. **game-origin запущен в Docker и доступен по http://localhost:8092/**
    — clean-room PHP-порт legacy. Dev-логин — `/dev-login.php`
    (мгновенный, как `test`/userid=1, 9 планет, ~36M очков).
    БД на `localhost:3307`, контейнер `docker-mysql-1`, БД `oxsar_db`,
    пользователь `oxsar_user`/`oxsar_pass`. Параметры доступа и
    проверенные экраны — в [docs/legacy/game-origin-access.md](../legacy/game-origin-access.md).
    Это **основной** источник для сверки UI и поведения; legacy
    oxsar2 на :8080 (`docs/legacy/game-reference.md`) — вторичный
    (для сверки оригинального PHP).

### Стратегическое решение принято: ремастер на nova-backend

**Не feasibility-вопрос «делать ли», а аудит «что нужно сделать».**
Решение зафиксировано пользователем 2026-04-28:

- Origin-фронт переписывается на современный стек (React/TS/TanStack
  Query/Zustand/TipTap) с **pixel-perfect клоном legacy-визуала**.
- Backend = game-nova, расширяется параметризуемостью под legacy-режим.
- **Это ремастер с новыми возможностями** (после старта pixel-perfect
  идут постепенные новшества).
- legacy01 — третья вселенная рядом с uni01/uni02.

**Цель плана 62 переформулирована:** не «выбрать стратегию A/B/C»,
а **получить полный список работ**, которые нужны в nova-backend и
nova-frontend для запуска legacy-режима. Этот список превратится в
серию последующих планов 63+.

Вердикт «infeasible» из плана исключён: ничего не infeasible, есть
только список «вынести в конфиг X», «добавить механику Y», «реализовать
endpoint Z».

## Что исследуем

### Категория 1: домен-модель

Сравниваем сущности и их атрибуты в обеих кодовых базах.

**Источники:**
- game-nova: `projects/game-nova/backend/internal/<domain>/` — Go-структуры,
  sqlc-генерированные модели, migrations.
- game-origin: `projects/game-origin/src/game/*.class.php`,
  `projects/game-origin/src/game/models/`, миграции в
  `projects/game-origin/migrations/`.

**Что фиксируем:**

| Сущность | Поля в nova | Поля в origin | Расхождения |
|---|---|---|---|
| User / Player |  |  |  |
| Planet |  |  |  |
| Fleet / Mission |  |  |  |
| Building |  |  |  |
| Tech / Research |  |  |  |
| Artefact |  |  |  |
| Alliance |  |  |  |
| Combat / Battle Report |  |  |  |
| Espionage Report |  |  |  |
| Expedition |  |  |  |
| Achievement |  |  |  |
| Event (event-loop) |  |  |  |

**Категории расхождений:**

- 🟢 **«В nova поле есть, в origin нет, оно опционально»** — отключаемо
  флагом, лёгкое.
- 🟡 **«В nova поле есть, в origin нет, оно обязательное»** — нужен
  default или nullable, средняя сложность.
- 🔴 **«В origin поле есть, в nova нет»** — миграция nova-схемы под
  legacy, тяжёлая.
- 🟣 **«Поле есть в обеих, но семантика разная»** — самое опасное:
  тихие баги, нужен отдельный legacy-кодпатч.

### Категория 2: формулы и константы

Сравниваем балансные формулы.

**⚠️ Архитектурное расхождение источников истины (важно):**

Origin и nova хранят балансные данные **в принципиально разных
местах**, и это расхождение само по себе требует решения.

- **game-nova: формулы и числа в коде/конфигах**
  - YAML: `configs/units.yml`, `configs/buildings.yml`,
    `configs/defense.yml` и др. — стоимости, время, RF.
  - Go-код: формулы боя, экономики, ресурсов в
    `projects/game-nova/backend/internal/{combat,economy,
    research,building}/...`.
  - Источник истины — **репозиторий** (балансовые правки идут
    через PR + миграцию деплоя).

- **game-origin: формулы и числа в БД**
  - Таблица `na_construction` (`projects/game-origin/migrations/
    001_schema.sql:727-762`) — **главная таблица баланса**:
    содержит `basic_metal/silicon/hydrogen/energy/credit/points`
    (стоимости 1 уровня), `prod_*`, `cons_*`, `charge_*`
    (формулы производства/потребления/удорожания **как строки**
    `varbinary(255)`, которые код интерпретирует), `demolish`,
    `front`, `ballistics`, `masking`.
  - Таблица `na_attack_formation` — формации атаки
    (`projects/game-origin/migrations/001_schema.sql:497`).
  - Таблица `na_artefact_datasheet` — балансовые параметры
    артефактов (`projects/game-origin/migrations/001_schema.sql:246`).
  - Таблица `na_engine`, `na_config`, `na_options` — связанные.
  - PHP-константы в `projects/game-origin/config/consts.php` —
    глобальные коэффициенты (например, время боя, тики).
  - Источник истины — **продакшн-БД**. Балансовые правки
    исторически шли через UPDATE на живой базе.

**Что это значит для исследования:**

- При сравнении формул нельзя просто сопоставить YAML nova и
  PHP-константы origin. Нужно извлечь данные **из таблиц origin**
  (или из `migrations/002_data.sql`, где есть seed) и нормализовать
  под YAML-структуру для сравнения.
- Формулы `prod_metal` и т.п. в origin — это не числа, а
  **строки-выражения**, которые интерпретирует код origin. Нужно
  понять, какой парсер их разбирает (искать в Functions.inc.php /
  EventHandler.class.php / производственных классах) и какие
  переменные/операторы используются. Это отдельный артефакт.
- При выборе вердикта B (legacy-режим в game-nova) предстоит
  **архитектурное решение**:
  - **B1.** Извлечь формулы origin в новый YAML
    `configs/balance/legacy.yaml` и переключаться по
    `universe.balance_profile` (стандарт nova).
  - **B2.** Расширить game-nova поддержкой формул-в-БД (новые
    таблицы `legacy_construction`, парсер строк-выражений).
    Дублирование двух механик хранения баланса — anti-pattern,
    скорее всего НЕ выбираем.
  - **B3.** Гибрид — статика (числа) в YAML, динамика
    (формулы строками) интерпретируется в Go.
- При вердикте A (PHP остаётся) — расхождение не блокирует.
- При вердикте C (отдельный сервис) — origin может остаться с
  formulae-in-db, но тогда инструменты редактирования баланса
  будут разные у двух сервисов.

**Источники для извлечения:**
- game-nova: `configs/*.yaml`, `projects/game-nova/backend/internal/
  {economy,combat,research,building}/...`
- game-origin (БД-таблицы):
  - `na_construction` — здания, исследования, юниты, защита
    (одна таблица на всё, различаются `mode`).
  - `na_attack_formation` — формации.
  - `na_artefact_datasheet` — артефакты.
  - `na_engine`, `na_config`, `na_options`.
- game-origin (PHP-константы): `config/consts.php`.
- Сид-данные: `projects/game-origin/migrations/002_data.sql` —
  это `mysqldump` с реальными значениями, который агент может
  парсить как фактический баланс.

**Что фиксируем:**

| Формула | nova | origin | Совпадает? |
|---|---|---|---|
| Стоимость зданий по уровням |  |  |  |
| Время постройки |  |  |  |
| Производство ресурсов |  |  |  |
| Стоимость технологий |  |  |  |
| Время исследований |  |  |  |
| Стоимость юнитов |  |  |  |
| Время постройки флота |  |  |  |
| RF-таблица (rapid fire) |  |  |  |
| Атака/щит/HP юнитов |  |  |  |
| Формула боя (раунды, потери) |  |  |  |
| Ущерб защите |  |  |  |
| Экспедиции — диапазоны находок |  |  |  |
| Энергопотребление зданий |  |  |  |
| Лимит полей на планете |  |  |  |
| Расход топлива |  |  |  |

**Заключение по каждой строке:**

- ✅ **Идентично** — переиспользуется.
- 🟡 **Числа отличаются, формула та же** — закрывается legacy-конфигом
  (легко).

### Категория 4: event-loop и асинхронные обработчики

**Это одна из самых критичных категорий.** Расхождения в формулах
видны в YAML/БД и легко проверяются числом, расхождения в API ловятся
типами; а **расхождения в событиях** долго остаются незамеченными:
событие срабатывает «не так» (или вообще не срабатывает / срабатывает
лишний раз), а пользователь это видит как баг механики через дни.
Особенно опасны **AI-движения инопланетян** — они происходят без
участия игрока и с богатой логикой состояний (HALT → HOLDING →
HOLDING_AI → CHANGE_MISSION_AI).

Ключевые поля для каждого события:

1. **Когда событие создаётся.** Игрок инициирует (отправил флот),
   автоматически (cron, alien AI), как реакция на другое событие
   (chain).
2. **Когда событие срабатывает.** Время рассчитано как X, формула X.
3. **Что событие делает при срабатывании.** Бизнес-эффект — выписать
   списком действий с таблицами/полями.
4. **Цепочки событий.** Какие новые события порождает (например,
   `EVENT_ALIEN_HALT` → через 12-24ч → `EVENT_ALIEN_HOLDING`).
5. **Идемпотентность / дедуп.** Origin использует `NS::isFirstRun`
   memcached TTL=2s; nova — Postgres advisory locks (план 32).
   Совместимы концептуально, но возможны тонкие различия в окнах.
6. **Race-conditions.** Какие защищены, какие нет (см. memory
   `reference_isfirstrun_dedup.md`).

**Особый блок: AI инопланетян.**

крадут кредиты, меняют цели в полёте, агрегируются в ходлинги.
Это **самая богатая логика во всём origin event-loop'е**, и
перенос её в nova ≠ просто маппинг событий.

Для AI-инопланетян отдельный подсравнительный артефакт:

[docs/research/origin-vs-nova/alien-ai-comparison.md](../research/origin-vs-nova/alien-ai-comparison.md):

- Состояния AI: какие есть в origin, какие в nova.
- Переходы между состояниями (state machine).
- Триггеры начала AI-активности (четверг? случайно? на основе
  очков?).
- Параметры: сколько кораблей, какие классы, длительность
  состояний, формула «сколько ресурсов крадут».
- Эффекты на игрока: HOLD планеты, мут производства, кража, что-то
  ещё.
- **План 15 в game-nova** реализован, но АВТОР ОТМЕТИЛ что часть
  (Этап 3: ChangeMission/GrabCredit/FlyUnknown) пропущена/упрощена
  ссылка: `docs/plans/15-...` или memory. Это означает что AI в
  nova может быть **не полным** относительно origin. **Подробно
  проверить.**

**Категории по событиям:**

- 🟢 **Совпадает по семантике, payload может отличаться структурно
  (JSON vs позиционные)** — переиспользуем nova-handler, при
  необходимости адаптируем payload-схему.
- 🟡 **Совпадает имя/триггер, но действие отличается** (например,
  обоих случаях `BUILD_FLEET`, но в legacy дополнительно даёт
  бонус за артефакт А) — добавить ветку в nova-handler за флагом.
- 🟠 **Только в origin, нужно реализовать в nova** — отдельный
  модуль/handler. Запись в журнал с оценкой.
- 🔴 **Только в origin, концептуально несовместимо** (например,
  legacy `TEMP_PLANET` требует поля `temporary`+`expires_at` в
  планете, которых в nova нет) — нужна миграция схемы + handler.
- 🔵 **Только в nova** — отключить флагом для legacy-вселенной.
- 🟣 **Совпадает имя, но СЕМАНТИКА разная** — самый опасный класс,
  тихие баги. Пример проверки: оба события называются
  `EVENT_RECYCLING`, но в origin перерабатываются обломки оборонки,
  а в nova — только флота.

**Заключение:** покрывает ли nova event-loop все типы событий origin
напрямую (🟢), с правкой за флагом (🟡), с расширением (🟠), с
миграцией схемы (🔴); сколько нужно отключить (🔵); сколько тихих
расхождений (🟣 — самые опасные, выделяются в отдельный список).

### Категория 5: API-поверхность

Сравниваем endpoint'ы, которые потребуются origin-фронту.

**Категории по API:**

- ⚪ **Имя/путь отличается, семантика идентична** — НЕ расхождение.
  Origin-фронт пишется с нуля и **сразу вызывает nova-имена**, без
  собственных. Адаптеры на backend не делаем — это мёртвый груз.
  В журнал не идёт.
- 🟢 **Endpoint есть в обоих с одинаковой семантикой** — origin-фронт
  использует nova-API напрямую (через тот же сгенерированный
  OpenAPI-клиент, что nova-фронт).
- 🟡 **Endpoint есть в обоих, но возвращает разный набор полей или
  формат** — расхождение. Решение: расширить nova-DTO опциональными
  полями (origin-фронт читает свои поля, nova-фронт игнорирует) —
  обратно совместимо. Запись в журнал.
- 🟠 **Endpoint нужен origin-фронту, в nova его нет** — расхождение.
  Решение: добавить endpoint в nova ИЛИ обойтись комбинацией
  существующих на фронте. Запись в журнал.
- 🔴 **Endpoint требует параметра/контекста, которого в nova нет
  концептуально** (например, legacy-механика, отсутствующая в nova) —
  нужна доработка backend'а: либо новый endpoint, либо параметризация
  существующего. Запись в журнал.

**Заключение:** покрывает ли nova-API всё, что нужно origin-фронту,
напрямую (🟢), через расширение DTO (🟡), через новые endpoint'ы
(🟠), или есть концептуальные пробелы (🔴).

**Важная вводная:** «адаптер-роутер на backend» как стратегия
**отвергнут**. Origin-фронт корректирует свои вызовы под nova-API,
не наоборот. Это дешевле в разработке (нет лишнего Go-кода) и в
поддержке (один источник истины — OpenAPI nova).

### Категория 6: UI-экраны и UI-логика

**Двойной фокус** (два самостоятельных deliverable, не путать):

1. **`origin-ui-replication.md`** — полный визуальный инвентарь
   origin для последующего pixel-perfect-клонирования на React.

Эти два артефакта **частично пересекаются по источникам**, но
имеют разную аудиторию:
- Replication читает агент, реализующий origin-фронт (план N+1).

**Источники:**
- game-origin: 125 .tpl-файлов в `src/templates/standard/`.

  каждый соответствует одному экрану + его действиям.
- game-origin: `public/css/`, `public/images/`, `public/js/` —
  визуальные ассеты, шрифты, иконки. **Critical для replication.**
- game-nova: компоненты `projects/game-nova/frontend/src/features/`,

**Вводная:** legacy-фронт = React + pixel-perfect-клон визуала.
.tpl служат **референсом для воспроизведения** (не для построчного
переноса в React, а для воссоздания того же визуального результата
React-средствами).

**UX-возможности**:

```markdown
### U-NNN. <Название функции>

- **Где в origin:** `src/templates/standard/<file>.tpl` +
  `src/game/page/<Controller>.class.php:NN`
- **Что делает игрок:** <одно предложение>
- **Что происходит на backend:** <если функция требует API,
  которого нет в nova — это ещё и запись D-NNN в Категории 5>
- **Аналог в nova:** нет / есть но урезанный / не применимо
- **Кандидат на добавление в game-nova?** да / нет / зависит
- **Связь с origin-проектом:** обязательно / желательно / только
  для legacy-режима
- **Объём:** строк frontend + backend (грубо, часы/дни)
```

Пример возможных записей (агент верифицирует и расширит):

- **U-001. Симулятор боя.** Игрок вводит составы флотов,
  получает прогноз исхода. В origin — `Simulator.class.php` (749 стр).
  В nova — есть battle-sim CLI (`backend/cmd/tools/battle-sim/`),
  но не как UI-фича. Кандидат на добавление в nova-frontend.
- **U-002. Биржа артефактов.** Игроки выставляют артефакты на
  обмен. В origin — `Exchange.class.php` (1220 стр). В nova нет.
- **U-003. Турниры.** В origin есть `EVENT_TOURNAMENT_*` + UI.
  В nova нет.
- **U-004. Buddy-list.** Список друзей.
  `na_buddylist`-таблица, UI в шапке. В nova проверить.
- **U-005. Изменение профессии за кредиты** (judging by
  legacy seed — `PROFESSION_CHANGE_COST_INFO`). Проверить в nova.
- **U-006. Симулятор / калькулятор постройки** (на странице
  Constructions показывается прогноз окупаемости).
- **U-007. Теги/иконки альянса** (custom logo upload).
- **U-008. История атак на планету** (длинный лог).
- … (агент дополняет до конца)

**Категории по UI:**

- 🟢 **Экран есть в nova с тем же набором функций** — переиспользуем
  (с темизацией для legacy-skin).
- 🟡 **Экран есть, но функции урезаны** — список недостающих
  функций → записи U-NNN, кандидаты на добавление.
- 🟠 **Экрана в nova нет** — нужно создать (frontend + при
  необходимости backend). Запись U-NNN.
- 🔵 **Экран в nova есть, в origin нет** — отметить, в журнал не идёт
  (это про развитие nova, не про feasibility origin).

**Заключение:**

- Сколько экранов origin не имеет аналога в nova → объём legacy-skin'а
  (если вердикт ≥ B).
- Сколько UI-функций origin отсутствуют в nova и **должны быть
  добавлены в game-nova независимо** (бэклог nova).
- Сколько UI-функций origin специфичны только для legacy-механик
  и не нужны в современной вселенной.

**`nova-ui-backlog.md` — самостоятельный актив.** Используется не
только в плане 62, но и при общем планировании nova-roadmap. После
закрытия плана 62 — каждая U-NNN-запись становится либо тикетом в
nova-backlog, либо обоснованием отказа.

#### Особый блок: UX-микрологика (цвета, тексты, состояния)

Помимо «больших» функций (целые экраны и кнопки), в origin есть
богатый слой **визуальной обратной связи**, который часто более
проработан, чем в современных SPA. Примеры:

- Стоимость ресурса в форме «Заказать постройку» **окрашивается
  красным**, если у игрока не хватает.
- Кнопка «Начать исследование» **отключена** и подписана конкретно
  («не хватает уровня Энергетической технологии») — не просто
  «нельзя».
- В списке возможных миссий неактивные варианты **подсвечены
  серым с tooltip'ом «причина»**, активные — нормальным цветом.
- В обзоре империи планеты **с проблемой** (нет энергии, перепол.
  склад) подсвечены индикатором.
- Тексты сообщений в боевом отчёте меняются по контексту
  (победа/ничья/поражение, потери союзника).
- В чате собственные сообщения / союзники / враги имеют разные
  цвета.
- Отрицательное производство ресурсов отображается красным.
- Время до завершения постройки/исследования меняет формат при
  пересечении пороговых значений.

**Эта логика часто:**

1. Зашита в Smarty .tpl как inline-условия `{if $cost > $have}...{/if}`.
2. Не задокументирована отдельно.
3. Воспроизводит **накопленный за годы UX-опыт игроков** —
   что-то, что нельзя получить, спроектировав игру с нуля.

**Что фиксируем (в `nova-ui-backlog.md`, отдельный раздел
"UX-микрологика"):**

```markdown
### X-NNN. <Где и какой сигнал>

- **Где в origin:** `templates/standard/<file>.tpl:NN`
  (с цитатой Smarty-условия)
- **Триггер:** что должно быть в данных, чтобы сигнал сработал
- **Визуальное воплощение:** цвет / иконка / disabled-состояние /
  tooltip / change of label
- **Аналог в nova:** есть / отсутствует / частично
- **Кандидат на добавление:** да / нет / зависит
- **Связь с фичей:** какая страница / экран / контекст
```

**Минимальный набор поверхностей для проверки:**

- Constructions / Shipyard / Research — индикация «не хватает X»
  / «требуется уровень Y»
- Стоимости в любых формах — цветовая индикация дефицита
- Mission / Galaxy — почему миссия недоступна (расстояние,
  тип цели, лимит флота)
- Empire — индикаторы проблем на планетах
- Battle reports — формулировки исхода
- Chat — типы участников
- Resource bars — отрицательная балансировка / переполнение
- Building queues — статус (в очереди, активен, завершён)
- Officers / Achievements — прогресс / "осталось X до получения"

**Объём:** `nova-ui-backlog.md` ожидает раздел «UX-микрологика» с
**не менее 20 записями** X-NNN. Это часть, которую агенту проще
всего недосмотреть (нет вызовов API, нет отдельных файлов — всё
в шаблонах). Если меньше 20 — агент схалтурил.

**Зачем это в плане 62:** даже если origin не переписываем, это
**самый ценный UX-актив legacy-вселенной**, который хочется
собрать в систему до того, как PHP-проект будет архивирован.
Каждая запись X-NNN — кандидат на улучшение nova-UI.

#### Особый блок: Альянсы

Альянсная подсистема в origin — одна из самых развитых частей
игры. Стоит проверить детально, чтобы не пропустить функции при
сравнении.

**Источники для origin:**
- `projects/game-origin/src/game/page/Alliance.class.php` (1413 стр)
  — главный контроллер, ~30 protected-методов = ~30 действий.
- `projects/game-origin/src/game/AllianceList.class.php` (294 стр).
- Шаблоны: `templates/standard/ally*.tpl` (ally_page, ally_search,
  manage_alliance, member_list, diplomacy, …).
- Таблицы БД: `na_alliance`, `na_allyrank`, `na_allyapplication`,
  `na_ally_relationships`, `na_ally_relationships_application`.

**Источники для nova:**
- `projects/game-nova/backend/internal/alliance/` (1314 стр Go,
  16 endpoint'ов).

**Уже видимые расхождения** (стартовая точка, агент верифицирует):

| Функция origin | Где origin | В nova? | Кат. |
|---|---|---|---|

| Подача заявки с текстом | `apply` / `writeApplication` | ✅ Join | проверить семантику |
| Принять/отклонить заявку | `manageCadidates` | ✅ Approve / Reject | 🟢 |
| Покинуть | `leaveAlliance` | ✅ Leave | 🟢 |
| **Забросить альянс / передать лидерство** | `abandonAlly` / `referFounderStatus` | ❓ нет явного | 🟠 проверить |
| Обновить тег | `updateAllyTag` | ❓ через Update? | проверить |
| Обновить имя | `updateAllyName` | ❓ через Update? | проверить |
| **Три отдельных описания (внешнее, внутреннее, для заявки)** | `updateAllyPrefs` | ❓ только `description` | 🟠 расширить |
| **Кастомные ранги с гранулярными правами** | `manageRanks` / `getRankSelect` | ✅ SetMemberRank | проверить — enum или права? |
| **Дипломатия: NAP / союз / война** | `diplomacy` / `applyRelationship` | ✅ Relations API | проверить полный набор статусов |

| BBCode-парсер | `bbcode` | ❌ → TipTap | не расхождение, переписывается |

**Что фиксируем в журнале (D-NNN с категорией `домен` или `api`):**

- 🟠 **Три описания альянса (external/internal/application).** В
  origin — три текстовых поля разной видимости. В nova — одно
  `description`. Решение: добавить поля в `alliances` + DTO.
  Объём: миграция БД + handler + frontend (1-2 дня).
- 🟠 **Передача лидерства (abandonAlly).** Если нет — добавить
  endpoint `POST /api/alliances/{id}/transfer-ownership/{userID}`.
- 🟠 **Гранулярные права рангов.** Если в nova роль = enum
  (`leader / officer / member`), а в origin у роли есть набор
  битов (`canManageApps, canSendGlobalMail, canManageDiplomacy,
  canKick`) — расширение модели рангов.
- 🟠 **Global mail членам альянса.** Зависит от mail-service
  (план 57). До его реализации — отметить как заблокированное.
- 🟡 **Дипломатические статусы.** Сравнить enum статусов:
  origin может иметь `NAP / союз / война / враждебный нейтралитет`,
  nova — `friend / neutral / war` (или подобное). Если расходится —
  расширить enum.

**Что относится к UI-backlog (U-NNN):**

- U-NNN: «Дипломатия» — полный экран управления отношениями
  альянсов (если в nova нет).
- U-NNN: «Управление рангами» — UI создания кастомных рангов с
  правами (если в nova нет).
- U-NNN: «Global mail редактор» — TipTap-форма с превью получателей
  (после mail-service).
- U-NNN: «Поиск альянсов» — полнотекстовый поиск с фильтрами
  (открытый/закрытый, размер, тип).
- U-NNN: «Альянсный лог» — кто вступил/вышел/изгнан/повышен
  (если есть в origin и нет в nova).

#### Особый блок: чат и текстовые редакторы

**Архитектурное решение принято заранее:**

- **Чат и почта используют один компонент TipTap (MIT).**
  План 57 (mail-service) явно фиксирует TipTap как редактор.
- Чат origin (`chat.tpl` + bbcode-парсер в `bbcode` функции
  Alliance.class.php / Functions.inc.php) — **полностью
  выкидывается**, не портируется.
- В обоих фронтах (nova и origin) чат пишется заново на одном
  React-компоненте `<RichTextEditor mode="chat">`, использующем
  TipTap внутри. Toolbar редактора параметризуется по режиму:
  - `chat` — минимум: bold, italic, link, quote.
  - `mail` — полный набор + custom-nodes (battle-report-card,
    spy-report-card, expedition-summary, и т.д.).
  - `alliance-mail` — между chat и mail.
- Рассылки альянса (`globalMail`) идут через mail-service после
  его реализации (план 57).

**Что это значит для исследования:**

- В журнал расхождений по чату детально **не лезем** — обе
  кодовые базы получают единый новый компонент.
- BBCode-парсер origin не учитывается как расхождение — он не
  переносится никуда.
- Достаточно отметить факт: `chat.tpl` + bbcode → один React-компонент
  TipTap.
- Если в origin есть какие-то **специфические фичи чата**
  (приватные вкладки, шифрование, история по альянсам), которых
  нет в nova-чате — это идёт в `nova-ui-backlog.md` как U-NNN
  (UI-функции чата).

### Категория 7: runtime-генерируемые ассеты

Часть ассетов в origin **не статические файлы**, а генерируются
кодом на лету. Это легко пропустить при сравнении сущностей и
формул, но без них фронт origin будет показывать пустоты.

**Известный пример** (от автора):
- `projects/game-origin/public/artefact-image.php` (153 строки) —
  PHP-GD endpoint, отдающий композитную PNG-картинку артефакта:
  фон постройки/исследования + иконка типа артефакта + уровень +
  кеш в `$CACHE_DIR/{name}{level}{art_name}.jpg`.
- URL: `/artefact-image.php?cid=N&level=N&typeid=N`.
- Используется в шаблонах артефактов (`Artefact.class.php` →
  `setViewParams`, поле `artefact["image"]`).
- **В nova аналога нет** — артефакты в game-nova показываются по
  статическим иконкам.

**Что фиксируем:**

| Ассет | Источник в origin | Источник в nova | Статус |
|---|---|---|---|
| Изображение артефакта (composite) | `public/artefact-image.php` | — | 🟠 нужен Go-генератор |
| Аватар игрока |  |  |  |
| Карта галактики (вид) |  |  |  |
| Превью отчёта боя |  |  |  |
| Логотип альянса |  |  |  |
| Скриншот боя / экспедиции |  |  |  |
| Иконки построек по уровню |  |  |  |
| Прочие сгенерированные изображения |  |  |  |

**Поиск в game-origin:**

```bash
grep -rln "imagecreate\|imagecopy\|imagecolor\|imagepng\|imagejpeg\|imagettftext" \
  projects/game-origin/
```

→ найдены `public/artefact-image.php` (точно), плюс возможно другие
PHP-скрипты, отдающие image/* контент.

**Категории по ассетам:**

- 🟢 **Статические файлы (PNG/SVG в public/)** — переиспользуем как
  есть. Не расхождение.
- 🟠 **Генерируется кодом в origin, в nova аналога нет** — нужен
  Go-генератор. Запись в журнал. Для каждого:
  - Стек: Go-стандарт `image/draw` / `image/png` подходит для
    композитов (overlay, text). Для сложных — `golang.org/x/image`.
  - Кеш: те же правила, что в legacy — disk cache по
    параметрам, инвалидация по версии конфига.
  - Endpoint: новый handler в game-nova, либо отдельный
    media-сервис.
- 🟡 **Генерируется в обоих, но логика отличается** — выбрать
  целевую (nova-стиль) и допилить параметризацию. Запись в журнал.
- 🔴 **Использует шрифты/ресурсы с несовместимой лицензией** —
  блокер по лицензии (см. план 40), не по технике. Записать
  отдельно.

**Заключение:** какие генераторы нужно добавить в Go, оценка
объёма (Go GD-эквиваленты простые — 100-200 строк на генератор +
тесты).

### Категория 8: общая инфраструктура (cross-universe сервисы)

**Архитектурное решение принято заранее:** все cross-universe
сервисы — единые для всех вселенных, включая origin. Нет двух
billing'ов, двух identity, двух почт, двух реферальных программ.
Origin-фронт пишется на тех же API, что nova-фронт.

**Список общих сервисов:**

| Сервис | План | Используется в origin |
|---|---|---|
| Identity (JWT, OAuth, RBAC, users) | 51, 52 | ✅ как в nova |
| Billing (платежи, кошельки, лимиты, аудит) | 38, 42, 54 | ✅ как в nova |
| Portal (новости, предложения) | 36 | ✅ как в nova |
| Reports (жалобы, модерация) | 56 | ✅ как в nova |
| Mail-service (когда будет) | 57 | ✅ как в nova |
| Admin-bff + admin-frontend | 53 | ✅ единая админка на все |
| Реферальная программа | 59 | ✅ как в nova |
| Moderation blacklist | 46, 48 | ✅ общий YAML |
| Multi-instance / scheduler / chat fan-out | 32 | ✅ как в nova |

**Что это значит для исследования:**

- Эта категория **не должна порождать расхождений** — если происходит,
  это сигнал что архитектурная договорённость была нарушена.
- Если в legacy-PHP origin была своя логика биллинга / своя таблица
  пользователей / свой механизм рассылок — она **выкидывается**, не
  портируется. Замена — вызов соответствующего nova/portal/identity
  API.
- В журнал расхождений по этой категории идут только **подводные
  камни**: например, «origin использует поле X из identity, которого
  ещё нет в текущей версии identity-service» — это не игровая
  механика, а доработка инфраструктурного сервиса.
- БД-схема `users` / `planets` / `fleets` сравнивается только в
  части **игровых полей** (Категория 1). Системные поля (`id`,
  `created_at`, JWT-claims) уже унифицированы планом 52.

**Что фиксируем:**

- Использует ли legacy-PHP origin какие-либо сервисы, которых **нет**
  в nova-стеке (или есть, но без нужного endpoint'а)?
- Есть ли поверхности, где legacy-логика глубоко переплетена с
  игровой и не поддаётся простой замене на API-вызов? (Например,
  legacy `na_payment` напрямую модифицирует `users.credit` в той же
  транзакции, что игровой эффект — в nova это разнесено по сервисам,
  нужно решить как.)

**Заключение:** «лёгкая» категория по умолчанию (всё уже сделано —
переиспользуется), кроме точечных мест, где legacy-PHP делает
кросс-сервисные операции в одной транзакции БД. Эти места — в
журнал, с предложением «как разделить на API-вызовы».

### Ф.3.5. Составление журнала расхождений

Параллельно с заполнением сравнительных таблиц (Ф.3) — каждое
найденное расхождение конвертируется в **запись в журнале** по
следующему шаблону.

**Файл:**
[docs/research/origin-vs-nova/divergence-log.md](../research/origin-vs-nova/divergence-log.md)

**Шаблон записи:**

```markdown
### D-NNN. <Название расхождения, одна строка>

- **Категория:** домен / формула / механика / event-loop / api / ui / assets / инфра
- **Цвет:** 🟢 / 🟡 / 🟠 / 🔴 / 🟣
- **Origin (legacy):**
  - Файл: `projects/game-origin/src/game/<...>.class.php:NN`
  - Что делает: <2-3 строки описания>
  - Конкретные числа/формула/поведение: <выписать как есть>
- **Nova (current):**
  - Файл: `projects/game-nova/backend/internal/<...>.go:NN`
  - Что делает: <2-3 строки>
  - Конкретные числа/формула/поведение: <выписать как есть>
- **Разница:** <один абзац точно про различие — не «отличается»,
  а «origin использует X, nova использует Y, потому что Z»>
- **Как сделать nova-конфигурируемой:** <конкретное предложение —
  «вынести константу в configs/balance/legacy.yaml», «добавить флаг
  universe.feature_X», «extract-method и добавить ветку», «нужен
  отдельный модуль internal/legacy/...» и т.п.>
- **Объём правки nova (оценка):** <строк кода, часы или дни>
- **Риски:** <тихие баги, breaking changes, тесты которые сломаются>
```

**Правила:**

- Каждое расхождение получает уникальный ID **D-001**, **D-002**, ….
  Это позволяет ссылаться на них из будущих планов и коммитов.
- Записи группируются по категории внутри файла (заголовки H2 на
  категорию, H3 на запись).
- Если расхождение **массовое** (например, отличаются 200 констант
  баланса) — одна запись D-NNN с примером, плюс ссылка на полный
  список в приложении или CSV.
- Цвет 🟢 (полное совпадение) **в журнал не записывается** — журнал
  только для расхождений. Совпадения учитываются в `comparison.md`.
- Для каждого 🟣 (тихая семантика) — обязательно конкретный пример
  поведения, при котором seman'тика расходится. Не «возможно
  отличается», а «при action X origin делает Y, nova делает Z».

**Минимальный целевой объём журнала:**

- Не менее **30 записей** при любом вердикте — даже если игра
  «почти совпадает», расхождений в формулах баланса достаточно для
  десятков записей.
- Если записей < 15 — это сигнал, что агент схалтурил, а не что
  игры идентичны.

**Что журнал НЕ содержит:**

- Тривиальные расхождения именования полей (`user_id` vs `userid`) —
  это технический мусор, в журнал не идёт.
- Различия в стеке (PHP vs Go) — это очевидно, не расхождение.
- Расхождения внутри только одной из кодовых баз (рефакторинг,
  TODO в origin) — журнал только про nova↔origin diff.

### Ф.4. Оценка объёма работ и группировка в будущие планы

На основе сравнения суммируем (числовые выкладки для каждой
категории расхождений):

- 🟡 **Числа отличаются, формула та же** → объём legacy.yaml +
  оценка работ по конфиг-loader'у (часы–дни). **В nova не правим**,
  выносим в конфиг.
- 🔵 **Фичи только в nova, отключаемые в legacy** → сколько
  feature-флагов на уровне `universe.profile = legacy` (часы).
- 🟠 **Фичи только в origin** (биржа, турниры, телепорт планеты,
  расширенная дипломатия и т.д.) → сколько новых модулей в
  nova-backend (дни–недели каждый). **Реализуем в nova**, общий
  знаменатель.
- 🔴 **Формулы расходятся** → сколько код-путей нужно
  параметризовать в Go (`if profile == "legacy" { ... }` ИЛИ
  изоляция в `internal/<domain>/legacy/`). **Nova-формулы остаются
  для современных вселенных**, legacy получает свой код-путь.
- 🟣 **Тихая семантика** → сколько unit/golden-тестов на паритет
  нужно написать (дни).
- 🟠 (UI) **U-NNN-функции, отсутствующие в nova** → сколько
  фронтенд-фич добавить на стороне nova-frontend (общий
  знаменатель).
- 🟠 (UI) **X-NNN UX-микрологика** → опциональное улучшение
  nova-фронта, не блокирует origin (origin получит X-логику в своём
  pixel-perfect клоне).

[docs/research/origin-vs-nova/roadmap-report.md](../research/origin-vs-nova/roadmap-report.md)

планов**. Стратегическое решение принято ранее (см. «Стратегическое
решение принято» выше): origin-фронт — pixel-perfect клон на
nova-backend, ремастер.

рабочие пакеты и предлагаем их как темы для последующих планов
(нумерация ориентировочная — на момент финализации 62 будут
заняты другие номера, агент проверит и сдвинет).

Предполагаемая декомпозиция (агент уточняет на основе фактических
объёмов):

- **План «legacy.yaml + per-universe loading».** Параметризация

  отключение achievements / dark-matter / oxsariт / AI-советника /
  daily-quests в legacy01. Уровень: per-universe-конфиг.
- **План «Новые модули в nova-backend».** Все 🟠-фичи из origin:
  биржа артефактов, турниры, телепорт планеты, расширенная
  дипломатия, global-mail альянса (после 57), три описания
  альянса, передача лидерства, гранулярные права рангов, … Может

- **План «Расширение event-loop».** EVENT_TOURNAMENT_*,
  EVENT_EXCH_*, EVENT_TELEPORT_PLANET, EVENT_TEMP_PLANET_DISAPEAR,

  EVENT_ATTACK_DESTROY_BUILDING, EVENT_ALIEN_ATTACK_CUSTOM,
  расхождения в alien-AI (если останутся после плана 15 этап 3).

- **План «UX-микрологика» (X-NNN).** Опционально для
  nova-frontend. Не блокирует legacy01.
- **План «Origin-фронт — pixel-perfect клон».** Главный план

## Чего НЕ делаем

- Не пишем код (никакой).
- Не меняем balance/конфиги/миграции.
- Не проектируем legacy-режим (это отдельный план, если вердикт ≥ B).
- Не оцениваем legacy-UI глубоко — только подсчёт экранов
  (детальная оценка UI — отдельный план, если вердикт ≥ B).
- Не сравниваем построчно код — сравниваем на уровне сущностей,
  формул, фич, событий, API.
- **Не записываем в журнал тривиальные расхождения именования** —
  ни полей БД (`user_id` vs `userid`), ни путей API (`/Mission` vs
  `/api/v1/fleet/missions`), ни параметров (`go=Page` vs REST-ресурсы).
  Семантически идентичные вещи с разными именами решаются на этапе
  реализации в новом origin-фронте — он сразу пишется на nova-API
  без собственных имён, без backend-адаптеров. В журнал идёт только
  то, что меняет **смысл**.
- Не оцениваем legacy-вёрстку Smarty (.tpl) построчно — фронт
  origin будет новый React-проект, .tpl служат только источником
  UX/layout-референса, не подлежат портированию.
- **Не сравниваем cross-universe сервисы** (identity, billing,
  portal, mail, reports, реферальная программа, moderation,
  admin-bff). Архитектурное решение зафиксировано: все они **единые
  для всех вселенных**, включая origin. Legacy-PHP-логика этих
  сервисов в origin **выкидывается**, заменяется на вызовы
  nova-API. В журнал идут только точечные подводные камни (см.
  Категорию 8), не общая «совместимость».

  с нуля, а **дополняется/верифицируется** (отметить, что устарело).

**2. `nova-ui-backlog.md` (записи U-NNN + X-NNN)** — UI-фичи и
UX-микрологика, которых нет в nova-frontend.

- Состоит из двух разделов:
  - **U-NNN — функции/экраны:** «биржа артефактов», «турниры»,
    «симулятор боя в UI».
  - **X-NNN — UX-микрологика:** красная подсветка дефицита,
    disabled-кнопка с конкретной причиной, индикаторы проблемных
    планет в обзоре, цветовые маркеры в боевом отчёте.
- **Не зависит от вердикта по origin.** Даже если origin останется
  на PHP — каждая запись становится тикетом nova-roadmap'а (или
  явным «не делать, потому что Z»).
- Используется при планировании любых nova-итераций.
- Связан с `divergence-log.md` ссылками: U-NNN/X-NNN, требующая
  нового endpoint'а или поля DTO, имеет ссылку на D-NNN.
- **Самая ценная часть для UX-эволюции nova** — каждая X-NNN
  отражает накопленный за годы UX-опыт legacy-проекта, который
  невозможно получить, проектируя игру с нуля.

**3. `alien-ai-comparison.md`** — паритет AI инопланетян.

- Если решим расширять alien-AI в nova (план 15 этап 3) — этот
  файл становится спецификацией.
- Если решим оставить упрощённый — файл документирует, что мы
  **намеренно** не реализовали и почему.

**4. `formula-dsl.md`** — описание parser-DSL legacy-формул.

- Используется только если вердикт ≥ B (legacy-режим). Иначе —
  справочный архив.

**5. `origin-ui-replication.md` (записи S-NNN)** — репродукционный
инвентарь origin-UI.

- **Прямая зависимость от вердикта:** если решено реализовывать
  origin-фронт на современном стеке (что соответствует целевой
  стратегии — план N+1) — этот файл становится **техническим
  заданием для агента-реализатора**: каждая запись S-NNN = один
  React-экран для воспроизведения.
- Используется в связке с `nova-ui-backlog.md`: при разработке
  S-NNN экрана агент проверяет, какие функции из U-NNN/X-NNN
  относятся к этому экрану, и **в первой итерации pixel-perfect
  не реализует** их (это «новшества потом»), но фиксирует как
  TODO.
- При появлении желания «добавить новшества» (после старта) —
  каждая S-NNN-запись становится точкой входа для последующих
  улучшений.
