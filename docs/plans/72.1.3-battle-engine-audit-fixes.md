## 2. Что сделано в re-audit 2026-05-02

### BA-009 — Moon chance: legacy Java composite formula

**Было**: `chance = min(20, debris/100000)` — упрощённая OGame-формула
без учёта experience и lostUnits.

**Стало** (`attack.go::tryCreateMoon`): legacy Java формула из
`Assault.java:1281-1306`:

```text
if min(atkExp, defExp) >= MOON_EXP_START_CHANCE=5:
    expChance     = round(min(atkExp, defExp) ^ 0.8)
    debrisChance  = (debrisM + debrisS) / MOON_PERCENT_PER_RES=200000
    chance        = min(expChance, debrisChance, min(atkLost, defLost))
guaranteedChance = (debrisM + debrisS) / MOON_GUARANTEED_PERCENT_PER_RES=10M
chance = max(chance, guaranteedChance)
chance = clamp(0, MOON_MAX_CHANCE=20)
if chance < MOON_START_CHANCE=5: skip
```

`tryCreateMoon` сигнатура расширена: принимает `debrisM, debrisS,
atkExp, defExp, atkLostUnits, defLostUnits` отдельно. Все 2 call-sites
(`attack.go`, `acs_attack.go`) обновлены — суммируют `report.Attackers
[*].LostUnits` и `report.Defenders[*].LostUnits`, передают
`report.AttackerExp/DefenderExp`.

### BA-013 — Tech.Laser/Ion/Plasma убраны

**Было**: `Tech` struct содержал `Laser/Ion/Plasma`, helpers
заполняли их с **ошибочным mapping** (`Ion = IDTechSilicon`,
`Plasma = IDTechHydrogen` — это совершенно другие технологии). В
`engine.go` поля никогда не читались.

**Стало**: поля удалены из `Tech` (`battle/types.go:74-87`).
`fleet/attack.go::collectTech` и `alien/helpers.go::collectAlienTech`
больше не присваивают эти поля. validateTech и engine_test обновлены.

Если 3-channel combat (laser/ion/plasma vs физический/магнитный/силовой
щит — см. `docs/balance/3channel-combat-idea.md`) будет реализован
в будущем — поля вернутся вместе с применяемой формулой.

### BA-019 — int64 overflow guard в Cost

**Было**: `validateUnit` проверял только `Cost.* >= 0`, без верхнего
лимита. При `Quantity ≤ 1e10` и `Cost ≤ ∞` `summarize` мог
переполнить int64 в `LostMetal = lost * cost.Metal`.

**Стало** (`engine.go::validateUnit`): добавлен
`Cost.Metal/Silicon/Hydrogen ≤ maxUnitCost = 1e8`. Defense:

```text
quantity × cost ≤ 1e10 × 1e8 = 1e18 < int64.max ≈ 9.2e18
```

С запасом 9× для int64. Realistic: legacy deathstar = 5M+8M+1M = 14M
суммарно, 1e8 — запас 7× от самого дорогого юнита.

## 3. Закрытые ранее (подтверждено re-audit'ом)

### BA-007 — Experience halving (HasPlanet vs IsMoon) ✅

`engine.go:139` корректно использует `in.HasPlanet`. Все 5 call-sites:
- `fleet/attack.go:237` — HasPlanet=true.
- `fleet/acs_attack.go:206` — HasPlanet=true.
- `alien/alien.go:259` — HasPlanet=true.
- `fleet/expedition.go::expPirates/expBattlefield` — HasPlanet=false (бой
  в открытом космосе).
- `simulator/handler.go:64` — HasPlanet=true.

Java equivalent — `Assault.java:838-840` (`if planetid == 0: bpc *= 0.5`).

### BA-008 — Debris formula 50%/1% ✅

`attack.go::calcDebris` использует `debrisShipsFactor=50` для флота и
`debrisDefenseFactor=1` для обороны (`attack.go:341-348`). Java
`Assault.java::getBulkIntoDebris(mode)` возвращает 0.5 для FLEET, 0.01
для DEFENSE.

### BA-010 — Destroy mechanics ✅

- `moon_destruction.go` (235 LoC): `tryDestroyMoon` для kind=25,
  `tryDestroyMoonACS` для kind=27. Формула 1:1 с Java
  (`P_destroy = clamp((100 - sqrt(diameter)) × sqrt(rip_count), 0, 100)`).
- `destroy_building.go` (218 LoC): kind=26 + `KindAttackAllianceDestroyBuilding`
  (kind=29). Формула `5×DS^0.3 clamp [0,25]` (план 72.1.45 §6).

Rocket attack (IPM) реализован в `internal/rocket/`, не использует
`battle.Calculate`.

### BA-011 — Experience for rocket attack ✅

`engine.go:139` имеет `&& !in.IsRocketAttack` guard. Defence-in-depth
на случай если в будущем IPM-flow пойдёт через `Calculate`.

### BA-012 — SimStats.AttackerExp ✅

`simstats.go:80-81` суммирует реальный `rep.AttackerExp/DefenderExp`
из atan-formulas, не proxy `lost_resources_sum`.

### BA-014 — IsAliens в Calculate ✅

`Calculate` не использует `IsAliens` — это правильно, флаг чисто для
post-battle обработки. `ApplyBattleResult` (battlestats) фильтрует
alien-стороны при applyExperience/Points чтобы не пытаться апдейтить
несуществующего юзера. Not a bug.

### BA-015 — validate() malicious input ✅

`engine.go:992-1076`: validate проверяет 8 типов лимитов (Quantity,
Damaged, ShellPercent, Front, Attack, Shield, Shell, Cost.*) +
Tech-уровни + Rapidfire + Rounds + NumSim. Все ошибки через
`ErrInvalidInput`.

### BA-018 — Front semantics ✅

`engine.go::unitWeight` = `2^front × quantity`. Java
`Units.java:243,249`: `weight = pow(2, front); startTurnWeight = weight × quantity`.
Совпадает.

### BA-021 — Multi-sim seed=0 RNG character ✅

`simstats.go:42-46`: при `seed0=0` ставится `seedOffset=1`, чтобы
первая итерация не попадала на guard `rng.New(0) → golden_ratio`.

## 4. Осознанные расхождения с Java (документированы)

### BA-016 — rawShots=1 fallback

`engine.go:763-765`:
```go
rawShots := int64(math.Round(float64(shooter.quantity) * portion))
if rawShots <= 0 { rawShots = 1 }
```

Гарантия: каждый shooter делает минимум 1 выстрел в каждую активную
цель. Java per-shooter loop этой проблемы не имеет (каждый shooter
сам выбирает 1 цель). Наш per-target distribution (см. BA-017) может
дать `rawShots=0` для слабых юнитов с большим количеством целей.

**Решение**: оставить fallback. Альтернатива — пропускать выстрел
вовсе — даёт пользователю counterintuitive `0 shots` с большим
числом целей. Игровой эффект: average DPS близок к Java; для
single-unit edge-cases (когда quantity=1, целей много) — слабый юнит
«размазывает» 1 выстрел между всеми, в Java бы стрелял в одну.
**Не баг — design choice** для предсказуемости UI.

### BA-017 — Rapidfire per-target distribution vs per-shooter

`engine.go::shootAtSides`: одна `rawShots = round(qty × portion)` per
(shooter, target) пара, потом `× rapidfire(shooter, target)`.

Java loop: для каждого `shooter unit` выбирает 1 цель пропорционально
weight, делает 1 выстрел, затем `canShootAgain` (rapidfire), затем
снова 1 выстрел в (возможно) другую цель.

**Эффект**: average DPS одинаков (linearity expectations). Variance
отличается: наш — детерминированное распределение, Java —
стохастический walking. Влияет на golden-snapshot tests — мы не
можем 1:1 повторить Java seed-результаты, но баланс одинаков.

**Решение**: оставить per-target. Performance — O(side_pair) против
O(shooter × shot) у Java, в 5-10× быстрее на large fleets. §14.4
cross-verification — отдельная задача.

### BA-020 — RNG Java-compat (xorshift vs LCG)

`pkg/rng` использует xorshift+ (быстрый, modern), Java
`java.util.Random` — Linear Congruential. Cross-verification по §14.4
блокирована.

**Решение**: §14.4 — non-blocking для прода (golden tests внутри Go
работают, баланс одинаков). При желании вернуть compat — добавить
`pkg/rng/javacompat` отдельным режимом. **Низкий приоритет**.

### BA-NEW-1 — Rapidfire deterministic vs stochastic (новая, найдена в re-audit)

Go `rapidfireMult` возвращает **константу** (`shots × rf`). Java
`canShootAgain`:

```java
if (rf > 0) {
    chance = 100.0 * (rf - 1) / rf;
    return randDouble(1, 100) <= chance;
}
```

Каждый выстрел с вероятностью `(rf-1)/rf` стрелять снова. Average
число выстрелов = `rf` (geometric series), но variance высокий.

**Решение**: оставить deterministic. Average одинаковый, в реальном
бою с тысячами выстрелов variance не виден на агрегированном уровне.
Расхождение — только в early-fight scenarios с малым количеством
shooters. **Не баг — performance optimization**.
