## Цель

Реализовать реферальную программу: игрок-«реферер» получает игровые
награды (оксариты) за привлечение новых игроков и их покупки. Работает
как дополнительный канал органического роста + лояльности.

**Архитектурно**: награды только в **оксаритах** (soft, игровой
ресурс). Не в hard-оксарах, не в рублях — это критично для
юридической чистоты.

---

## Юридическая основа

### Почему оксариты, а не рубли / hard-оксары

| Способ награды | Юридический статус | Решение |
|---|---|---|
| Прямая выплата рублями | Агентский договор (ст. 1005 ГК), НДФЛ (ст. 226 НК), 161-ФЗ при выплате через ЭДС. Большая бухгалтерия, договоры с каждым реферером. | ❌ Не используем |
| Hard-оксары (награда на кошелёк) | Возможна квалификация как доход в натуральной форме. Через рефанд теоретически выводится в рубли. Налоговая может квалифицировать как доход физлица. | ❌ Не используем |
| **Оксариты (игровой ресурс)** | Не имеет рублёвой стоимости вне игры (выводу не подлежит). Эмитируется оператором как награда (ст. 1062 ГК — игровая механика). Не доход, не платёж. | ✅ **Используем** |
| Игровые скидки/бонусы (премиум-офицер бесплатно за N рефералов) | Скидка/льгота, не доход. Чисто. | ✅ Используем как дополнение |

Это **стандартная практика** F2P-игр (World of Tanks Bonus Code,
Genshin Impact Friendship, EVE Online Buddy Program) — все
реферальные награды только в soft-валюте или игровых преимуществах.

### Индустриальная практика 2024–2026

Сравнение нашего плана с крупными F2P-проектами:

| Параметр | План 59 | World of Tanks | EVE Online | Genshin Impact | Roblox Stars |
|---|---|---|---|---|---|
| Welcome для реферера | 50 оксаритов | золото за каждый бой реферала | Skill Extractor + ISK | Welkin Moon (премиум-подписка) | — |
| Welcome для привлечённого | 100 оксаритов | стартовый танк + золото | 1 месяц Omega | стартовые предметы + примогемы | — |
| % от покупок реферала | 5% оксаритов | 7-15 единиц золота за бой | — (фикс. награда) | материалы | до 5% Robux |
| Activation cap | 7 дней + 3 логина + ≥100₽ | N боёв в M дней | подписка на Omega | достичь уровня 25 | 30+ дней + платежи |
| Max уровней (MLM) | 1 | 1 | 1 | 1 | 1 |
| Антифрод | IP/device/payment | капча + email-верификация | manual review | капча | платёжный метод + activity |
| Награды только в soft | да (оксариты) | да (только золото и время) | да (Omega-time) | да (примогемы и Welkin) | Robux (hard, но микросуммы) |

**Закономерности индустрии (по этим и аналогичным проектам):**

- **% от покупок** редко превышает 10%, типично 5-7%.
- **Welcome привлечённому > Welcome рефереру** — стандарт, причина:
  привлечь нового игрока ценнее, чем поощрить старого.
- **Activity-cap есть везде** — от 7 дней до 30 дней + специфичные
  условия (уровень / подписка / N боёв).
- **Многоуровневая (MLM) структура запрещена везде.** Только 1
  уровень: А пригласил Б, А получает за активность Б. Не «А пригласил
  Б, Б пригласил В, А получает за В».
- **Лимиты по IP/устройству/платёжному методу** — стандарт.
- **Награды только в soft-валюте или премиум-time** — после 2020
  индустрия отказалась от рублёвых выплат рефереру (агентский
  договор, налоговые проблемы).

#### Тренды 2024-2026

1. **Activity-cap стал жёстче.** Раньше платили сразу при регистрации,
   сейчас минимум 7 дней + проверка реальной активности.
2. **Fingerprinting через Canvas/WebGL.** Не только IP — поведенческие
   сигнатуры устройства.
3. **Геймификация рефералов.** «Привлеки 5 друзей → получи редкий
   скин/корабль/титул». Долгосрочно работает лучше микро-выплат.
4. **Push/email уведомления о выплатах** — повышает retention реферера.
5. **Ужесточение anti-MLM регулирования.** В РФ — Указ Президента
   № 309 (2007) о финансовых пирамидах + ст. 172.2 УК. Многоуровневые
   рефералы могут квалифицироваться как пирамида.

Наш план 59 находится **в норме индустриальной практики 2024-2026** —
не консервативен (не ниже минимума), не агрессивен (не выше потолка).
Можно корректировать числа после статистики первых месяцев работы.

## Модель данных

### Таблица `referral_codes`

Реферальный код — личный для каждого игрока.

```sql
CREATE TABLE referral_codes (
    user_id     UUID PRIMARY KEY REFERENCES users(id),
    code        TEXT UNIQUE NOT NULL,        -- буквенно-цифровой, 8 знаков, без визуально похожих (0/O, 1/I)
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    is_active   BOOLEAN NOT NULL DEFAULT true
);

CREATE INDEX referral_codes_code_idx ON referral_codes (code) WHERE is_active = true;
```

Код генерируется автоматически при первой активации страницы рефералов
(или при регистрации). Не меняется. Деактивируется только при бане
владельца.

### Таблица `referrals`

Связь «кто кого привёл».

```sql
CREATE TABLE referrals (
    id                  UUID PRIMARY KEY,
    referrer_user_id    UUID NOT NULL REFERENCES users(id),    -- кто привёл
    referee_user_id     UUID UNIQUE NOT NULL REFERENCES users(id),  -- кого привели (1 раз)
    code_used           TEXT NOT NULL,
    created_at          TIMESTAMPTZ NOT NULL DEFAULT now(),
    activated_at        TIMESTAMPTZ,                            -- когда reach activity threshold
    status              TEXT NOT NULL DEFAULT 'pending',        -- pending / activated / rejected (антифрод)
    rejected_reason     TEXT,                                   -- если status='rejected'
    fingerprint_hash    TEXT,                                   -- IP+device для антифрода
    metadata            JSONB
);

CREATE INDEX referrals_referrer_idx ON referrals (referrer_user_id, status);
CREATE INDEX referrals_status_pending_idx ON referrals (status, created_at) WHERE status = 'pending';
```

`referee_user_id` уникальный — пользователя нельзя «перепривлечь».

### Таблица `referral_payouts`

Журнал выплат реферальных наград.

```sql
CREATE TABLE referral_payouts (
    id                  UUID PRIMARY KEY,
    referrer_user_id    UUID NOT NULL,
    referee_user_id     UUID NOT NULL,
    referral_id         UUID NOT NULL REFERENCES referrals(id),
    payout_kind         TEXT NOT NULL,    -- 'welcome' / 'purchase_percentage' / 'milestone'
    oxsarit             BIGINT NOT NULL,    -- сумма выплаты в оксаритах
    universe_id         TEXT NOT NULL,    -- куда зачислены
    source_metadata     JSONB,            -- {payment_order_id, amount_rub, ...}
    paid_at             TIMESTAMPTZ NOT NULL DEFAULT now(),
    idempotency_key     TEXT UNIQUE NOT NULL
);
```

Полная история начислений — для аудита и отладки споров.

## Механика

### Способы получения реферера

1. **Реферальная ссылка**: `https://oxsar-nova.ru/?ref=ABCD1234`
   (опционально с подсказкой вселенной: `&hint=uni01`).
2. **Поле «Реферальный код»** на странице регистрации.
3. **Ввод кода после регистрации** — в первые 7 дней (для тех,
   кто узнал о программе уже играя).

При регистрации:
- если в URL/форме указан валидный `code`, создаётся запись в
  `referrals` со `status='pending'`;
- referee получает welcome-бонус **сразу** (см. ниже).

### Реферальная ссылка — куда ведёт

Базовая ссылка ведёт на **главную страницу portal'а**, не во
конкретную вселенную. Это соответствует индустриальной практике
multi-world игр (Wargaming, EVE Online, Roblox) и решает несколько
задач:

- **Гибкость для реферера.** Если он переедет играть в другую
  вселенную, его старые ссылки продолжают работать.
- **Информированность нового игрока.** Он видит весь проект —
  список вселенных, новости, особенности. Не «вваливается» сразу
  в игру.
- **Простота реализации.** Один маршрут на portal'е, не нужны
  per-universe handler'ы.
- **Нет привязки к статусу вселенной.** Если uni01 переполнена
  или закрыта для регистрации — ссылка всё равно работает,
  пользователь выберет другую.

#### Формат URL

```
Базовая:    https://oxsar-nova.ru/?ref=ABCD1234
С подсказкой: https://oxsar-nova.ru/?ref=ABCD1234&hint=uni01
```

Параметры:
- `ref` — обязательный, реферальный код (8 знаков буквенно-цифровой).
- `hint` — опциональный ID вселенной, в которой играет реферер.
  Используется для UX (см. ниже), не влияет на правила выплаты.

#### Поведение portal'а при визите со ссылкой

**Не залогинен:**
- Парсится `ref` из URL, валидность проверяется через
  `GET /api/referrals/code/{code}/validate` (без раскрытия данных
  реферера, только status: ok / not_found / inactive).
- Код **сохраняется в localStorage** на 7 дней (на случай если
  игрок не зарегистрируется сразу).
- Если код валиден, на главной показывается **банер**:
  «Вас пригласил игрок (если есть `hint`) во вселенной uni01 →
  получите 100 оксаритов в подарок при регистрации! [Зарегистрироваться]».
- В форме регистрации поле «Реферальный код» **автозаполнено** и
  заблокировано на изменение (можно только убрать через крестик).
- При успешной регистрации — referee автоматически получает
  welcome-бонус, реферер получает свой welcome-бонус.
- Если есть `hint=uni01` — после регистрации portal **предлагает
  одним кликом** перейти в эту же вселенную: «AdmiralVeles играет
  в uni01. Зайти туда?». Игрок может согласиться или выбрать
  другую вселенную из общего списка.

**Уже залогинен:**
- Если в `referrals` для текущего пользователя нет записи (он
  ещё никем не приведён) **и** прошло ≤ 7 дней с регистрации
  (см. правило «ввод кода после регистрации»):
  - Toast-уведомление: «Ваш друг приглашает вас. Применить код
    `ABCD1234`? [Применить] [Игнорировать]».
- Если запись уже есть (либо ≤ 7 дней истекли) → код игнорируется,
  без сообщений.

**Невалидный/inactive код:**
- Никаких сообщений пользователю — портал работает как обычная
  главная страница.

#### Куда ведёт ссылка после изменений

Ни в коем случае **не на профиль реферера** (вариант C из
обсуждения), потому что:
- Требует разработки публичной страницы профиля игрока
  (отдельная задача).
- Может содержать ПДн (никнейм + статистика) — нужно отдельное
  согласие реферера на публикацию.
- Не масштабируется на не-игровых рефереров (если реклама от
  партнёров).

И **не во вселенную напрямую** (вариант B), потому что
- Игрок не видит общий проект.
- Привязка к конкретной вселенной хрупкая.

Только **portal с опциональной подсказкой** — это правильный
паттерн для multi-world игры.

### Welcome-бонус

**При регистрации по реферальному коду:**

| Кому | Что |
|---|---|
| **Реферер** | + 50 оксаритов в **активную вселенную** (см. ниже про выбор вселенной). Сразу при регистрации reрефала. Стимул реферить активно. |
| **Referee** | + 100 оксаритов в первую вселенную, в которую он войдёт. Стимул использовать чужой код вместо игнорирования. |

Числа — стартовые. Балансировка — в `configs/economy.yaml`, корректируется по статистике.

### Процент от покупки

**При каждой покупке оксаров приведённым (через webhook YooKassa):**

- Реферер получает **5% оксаритов** от суммы покупки в оксарах.
- Пример: referee купил пакет 1000 оксаров за 100 ₽ → реферер
  получает 50 оксаритов.
- Зачисляется только если `referrals.status='activated'` (см.
  антифрод). Если ещё `pending` — выплата запрашивается, но
  встаёт в очередь до активации.

### Активация рефала (антифрод)

`referrals.status` переходит из `pending` в `activated` при
выполнении условий:

1. **Activity-cap**: referee провёл в игре ≥ 7 дней с момента
   регистрации (calendar days, не connected hours).
2. **Минимальная активность**: ≥ 1 successful login в каждый из
   3 разных дней внутри этого периода.
3. **Минимальная покупка**: referee совершил хотя бы одну покупку
   оксаров на сумму ≥ 100 ₽ ИЛИ выполнил N достижений (количество
   определяется в balance-конфиге; на старте — 5 ачивок).

Эти условия **И** — все три должны быть выполнены. Это **резко
снижает** мотивацию создавать фейк-реферрерства.

### Активная вселенная для зачисления оксаритов

Реферер играет в нескольких вселенных. Куда зачислить?

**Правило:** в **последнюю активную** вселенную реферера на момент
выплаты (последний login в game-nova). Если он играет только в
одной — туда. Если ни в одной не играл за 30 дней — выплата
встаёт в очередь до возвращения.

Альтернатива: позволить реферера **выбирать вселенную** в настройках
рефералов. Это сложнее в UI, но даёт контроль.

Решение для старта: **последняя активная**. Опция выбора —
расширение в будущем.

### Cap на количество рефералов

| Cap | Значение | Зачем |
|---|---|---|
| Рефералов в день | 10 / IP / реферер | Антифрод — VPN-фермы |
| Рефералов в месяц | 100 / реферер | Защита экономики от взрывного роста выплат |
| Welcome-бонусов на одного реферера за всё время | 1000 | Защита от долгосрочного фарма |

Лимиты конфигурируемые. Превышение → запись `status='rejected'`,
причина в `rejected_reason`.

## Antifraud — что проверяем

### Самореферал

Игрок регистрирует второй аккаунт по своему коду.

**Митигации:**

1. **IP-fingerprint**: при регистрации сохраняется hash(IP+UA).
   Если совпадает с реферером → `status='rejected'`,
   `reason='same_fingerprint'`.
2. **Платёжный fingerprint**: если referee позже оплачивает с
   карты, ранее использованной реферером → `status='rejected'`,
   `reason='same_payment_method'`.
3. **Email/phone-домен**: проверка confidential — не используем
   (privacy).

### Купил-ушёл

Referee регистрируется, кладёт минимальную сумму, реферер получает
бонус, referee не возвращается.

**Митигация — activity-cap** (7 дней + N логинов + 1 покупка).

### Боты

Автоматическая регистрация фейковых аккаунтов через скрипт.

**Митигации:**

1. Капча Yandex SmartCaptcha (план 46 §UGC) на форме регистрации.
2. Email-верификация (отправка кода на email — план 57 mail-service
   когда будет готов).
3. Activity-cap (бот не сможет «играть» 7 дней без сложной
   симуляции).

### VPN-фермы

Разные IP, но тот же владелец.

**Митигации:**

1. **Device fingerprint** через privacy-friendly технологии
   (Canvas-fingerprint, WebGL-renderer-hash). Не absolute, но даёт
   сигнал.
2. **Behavioral patterns**: новый аккаунт сразу клик-в-клик повторяет
   pattern реферера → флаг подозрения.
3. **Платёжная карта**: совпадение → жёсткая блокировка.

### Логика обработки антифрода

1. При регистрации referee → запись `referrals.status='pending'`.
2. Через 24 часа → background-job проверяет critical fingerprint
   matches. Если совпадение → `status='rejected'`, никакие выплаты
   не идут.
3. После activity-cap (7 дней) → проверка остальных условий →
   `status='activated'` или `status='rejected'`.
4. Все pending-выплаты по этому рефералу → выполняются (если
   activated) или отменяются (если rejected).

## API

### Публичные (без авторизации)

```
GET    /api/referrals/code/{code}/validate   валидация кода для UX
                                              → 200 {valid: true, hint_universe_id?: "uni01"}
                                              → 404 / 410 (inactive)
                                              Не раскрывает данные реферера, только статус.
                                              Используется portal'ом при визите по ?ref=...
                                              Rate-limited: 60 запросов / IP / минута.
```

### Игроку (авторизован)

```
GET    /api/referrals/me                     мой код, статистика, история
                                              {code, total_referrals, activated, pending, total_paid_oxsarit}
POST   /api/referrals/code/regenerate        перегенерировать код (опционально, не часто)
POST   /api/referrals/apply                  применить чужой код (после регистрации, в первые 7 дней)
                                              {code: "ABCD1234"} → 200 / 409 (уже есть реферер) / 404 / 410 (expired)
GET    /api/referrals/list                   мои рефералы (постраничный список)
                                              [{username, status, joined_at, total_purchases_count, last_seen_at}, ...]
GET    /api/referrals/payouts                история начислений
                                              [{kind, oxsarit, universe_id, paid_at, source}, ...]
```

### Internal (вызывается от billing webhook)

```
POST   /internal/referrals/on-purchase       срабатывает при successful payment
                                              {user_id, payment_order_id, amount_oxsars, idempotency_key}
                                              → 200 если есть реферер и payout произошёл
                                              → 200 если реферера нет (no-op)
                                              → 200 если status='pending' (выплата отложена)
```

### Админу (через admin-bff)

```
GET    /api/admin/referrals/users/{id}       рефералы пользователя
GET    /api/admin/referrals/payouts          все выплаты с фильтрами
POST   /api/admin/referrals/{id}/reject      ручной reject (модерация фрода)
POST   /api/admin/referrals/{id}/activate    ручная активация (по запросу пользователя)
GET    /api/admin/referrals/stats            агрегированная статистика по программе
```

RBAC: `referrals:moderate` — для рефанд-операций.

## Где живёт сервис

**Вариант A**: внутри **billing-service** (он уже видит покупки).
- Плюс: webhook-логика естественна, никакого лишнего сервиса.
- Минус: billing становится «жирнее».

**Вариант B**: отдельный микросервис `referrals-service`.
- Плюс: чёткое разделение.
- Минус: лишний сервис, лишний deploy.

**Вариант C**: в **portal-backend** (как и `reports` по плану 56).
- Плюс: portal — публичный фасад, рефералы естественно видны там
  (UI «Мои рефералы»).
- billing-webhook вызывает portal-backend через internal token.

**Решение**: **B (отдельный сервис)** на этапе зрелости проекта;
**C (portal-backend)** — для MVP. На старте проще встроить в portal:

- Меньше инфраструктуры.
- Когда программа разрастётся (50+ тыс. рефералов, сложные
  антифрод-правила) — выделить в `projects/referrals/`.

В этом плане идём по **варианту C — portal-backend**.

## Этапы

### Ф.1. Backend — модель данных и базовое API

- Миграции в portal-БД: `referral_codes`, `referrals`,
  `referral_payouts`.
- Endpoint'ы `GET /api/referrals/me`, `POST /api/referrals/apply`,
  `GET /api/referrals/list`, `GET /api/referrals/payouts`.
- Генератор кода (8 знаков, без 0/O/1/I, уникальный).

### Ф.2. Welcome-бонус

- При регистрации с кодом → создание записи `referrals`,
  начисление welcome-бонуса referee (через game-nova oxsarite API).
- Welcome-бонус для реферера сразу (50 оксаритов в активную вселенную).
- Запись в `referral_payouts`.

### Ф.3. Activity-cap и активация

- Background-job (через scheduler — план 32) каждый час:
  - проверяет `referrals.status='pending'` старше 24 часов;
  - применяет fingerprint-checks;
  - применяет activity-checks (вызывает game-nova API
    `GET /api/users/{id}/activity`);
  - переводит в `activated` или `rejected`.

### Ф.4. Процент от покупки

- В billing webhook добавить вызов `POST /internal/referrals/on-purchase`.
- Логика выплаты: если referer найден и activated → 5% оксаритов
  в активную вселенную реферера.
- Если pending → запись в очередь, выплата при активации.

### Ф.5. Antifraud

- IP/device fingerprint при регистрации (через identity-сервис,
  если он уже собирает; иначе — добавить).
- Проверка совпадений в background-job.
- Cap на количество рефералов в день/месяц.

### Ф.6. Frontend

- Страница `/profile/referrals` на portal.
- Поле «Реферальный код» в `AuthPage.tsx` с autofill из URL.
- Toast-уведомления при welcome-бонусе и purchase-payout.
- QR-код через библиотеку `qrcode.react` (MIT).

### Ф.7. Admin UI

- В `admin-frontend` (план 53) через `admin-bff` (план 53) —
  админ-страница рефералов: список, фильтры, ручной reject/activate,
  статистика.

### Ф.8. Юр-документы

- В `docs/legal/offer.md` добавить раздел «Реферальная программа»
  (формулировка из секции «Юридическая основа» этого плана).
- В `docs/legal/game-rules.md` (план 47) — пункт «Запрет
  фрода реферальной программы» (самореферал, боты).
- Версия оферты `1.0.X` → `1.1.0` (значимое изменение).
- При апдейте оферты — пользователи дают новый акцепт (план 47
  Ф.4 чекбокс при первом входе после изменения).

### Ф.9. Финализация

- Smoke-тест: новый пользователь регистрируется по коду, реферер
  получает welcome → активность → покупка → 5% оксаритов
  капают.
- Антифрод-тест: попытка саморегистрации с того же IP →
  rejected.
- Запись в `docs/project-creation.txt` — итерация 59.
- Серия коммитов:
  - `feat(portal): referrals model + base API`;
  - `feat(portal): activity-cap + antifraud`;
  - `feat(billing): on-purchase referral hook`;
  - `feat(frontend): referral profile page + register form`;
  - `feat(admin-bff): referral moderation`;
  - `docs(legal): offer v1.1.0 + game-rules § антифрод`.

## Тестирование

- Smoke-тест end-to-end (см. Ф.9).
- Unit-тесты на:
  - генератор кода (нет коллизий);
  - идемпотентность on-purchase (двойной webhook не приведёт к
    двойной выплате);
  - activity-cap (точные числа).
- Антифрод-кейсы:
  - один IP → second registration через тот же код → rejected;
  - один payment method → rejected;
  - VPN-bypass simulation (разные IP, но тот же device fingerprint) →
    отслеживается через behavioral signals.

## Что НЕ делаем

- **Не выплачиваем рублями** или hard-оксарами.
- **Не делаем многоуровневую (MLM) реферальную программу** —
  только 1 уровень: А пригласил Б, А получает за покупки Б.
  Многоуровневая попадает под определение пирамиды (Указ
  Президента РФ № 309 / законодательство о финансовых
  пирамидах).
- **Не покупаются «реферальные коды» за оксары** — каждый игрок
  получает свой код бесплатно.
- **Не выплачиваем за активность реферера** (только за активность
  referee). Иначе мотивация — приглашать ботов.
- **Не делаем рефанд реферальных оксаритов** — они «эмитированы»
  и дальше живут в game-nova по своим правилам.

## Известные риски

| Риск | Митигация |
|---|---|
| Массовый фрод через VPN-фермы | Activity-cap + capping + ручная модерация подозрительных кейсов |
| Юридическая квалификация как пирамиды | Только 1 уровень + награды только в soft-валюте + чёткая формулировка в оферте |
| Реферер играет и зарабатывает оксариты, его аккаунт банится → что с pending выплатами? | Pending-payouts отменяются при бане реферера. Activated — остаются (уже зачислены). |
| Конфликт интересов реферер-referee (один человек) | Fingerprint-checks + manual review |
| Нагрузка на billing webhook | Идемпотентность, retry-cron на referral-side |
| Пользователь не может найти страницу рефералов | Нативная промо-кампания: уведомления, banner, FAQ |
