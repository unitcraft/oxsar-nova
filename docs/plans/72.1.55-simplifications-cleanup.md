### Task B — [P72.S2.A] own applications (~2-3ч)

**Что**: legacy `ally.tpl` в блоке «Текущие заявки» показывает
applicant'у его pending applications (на какие альянсы он подал).

**Verify-handsweep до реализации**: проверить что endpoint
`GET /api/users/me/applications` действительно отсутствует, что
схема `alliance_applications` содержит нужные поля для join'a с
alliance.

**Реализация**:

- Frontend: api/alliance.ts `fetchMyApplications()` + блок в

  endpoint'а `DELETE /api/alliances/applications/{id}` и

**Verify-handsweep**: проверить что memberlist endpoint реально не

- Frontend: убрать клиентскую сортировку (если есть), полагаться на
  серверный порядок.

### Task D — [P72.S2.D] granular permissions UI (~2-3ч)

**Что**: backend проверяет `can_manage_ranks/can_change_description/
can_invite/etc`, но frontend всем не-owner'ам **не показывает**
management кнопки (manage/ranks/diplomacy). Backend защищает 403'ами,
но игроку с правом `can_manage_ranks` (но не owner) — кнопка
"Ranks" не видна.

**Verify-handsweep**: проверить как сейчас устроен Member DTO
(plan 67 говорит «нет rank_id»). Проверить `hasPerm()` в frontend —
действительно ли он fallback'ает только на owner.

**Реализация**:
- Backend Member DTO: `+rank_id?` (uuid), `+effective_perms?` (map).
  effective_perms — резолвенный набор permissions для текущего юзера
  (если он смотрит свои данные).
- Альтернатива: `GET /api/alliances/{id}/me-permissions` отдельно
  возвращает `{rank_id, effective_perms}` для текущего юзера.
- Frontend `hasPerm()`: использовать effective_perms если есть,
  иначе fallback на isOwner.
- Frontend AllianceMyScreen / AllianceManageScreen: показать
  manage/ranks/etc когда есть соответствующее permission.

### Task H — [P72.1.1.NO_BATTLE_REPORT_FOR_RAIDS] (~5-7ч)

**Что**: alien-рейды и экспедиционные бои вызывают
`battlestats.ApplyBattleResult(ctx, tx, report, "")` с пустым
battleID. user_experience.battle_id = NULL → UNIQUE constraint
не блокирует дубликат при event re-process.

**Решение для 1:1**: legacy oxsar2-java хранит ВСЕ бои в `assault`
(один сторадж). У nova split на `battle_reports` (PvP) +
`messages` (raids) + `expedition_reports`. Унификация = большая
работа.

**Verify-handsweep**:
- Прочитать `internal/alien/alien.go` и `internal/fleet/expedition.go`
  чтобы увидеть точные call sites `ApplyBattleResult`.
- Прочитать `internal/battlestats/apply.go` — что делает функция,
  и можно ли просто **писать в battle_reports** для всех типов боёв.
- Проверить, есть ли какие-то battle_reports схема-ограничения,
  которые мешают записи raids (например, NOT NULL planet_id который
  при экспедиции может быть NULL).

**Реализация**:
- Schema: проверить что `battle_reports.planet_id` nullable
  (для экспедиции planet_id отсутствует).
- alien.go: вызывать `WriteBattleReport` для каждого raid и передавать
  его id в `ApplyBattleResult`.
- expedition.go: то же самое.
- Тест: event re-process не даёт double-credit (UNIQUE сработает).

### Task I — [P72.S4.SETTINGS] show_all_*/planetorder/esps/ipcheck (~3-5ч)

**Что**: 7 полей legacy preferences:
- `show_all_constructions/research/shipyard/defense` (4 bool)
- `planetorder` (smallint: 0=по дате, 1=по имени, 2=по координатам)
- `esps` (bool: уровень детализации espionage info)
- `ipcheck` (bool: предупреждать о логине с другого IP)

**Что не делается**: 5 полей `templatepackage/skin_type/user_bg_style/
user_table_style/imagepackage` — архитектурно невозможны (legacy-PHP
темы; React+CSS modules не позволяют менять стиль через user
preferences без полного reskinning кодовой базы). Эти 5 остаются
в simplifications как **архитектурное упрощение**.

**Verify-handsweep**: проверить что в `users` schema нет этих
колонок; что legacy `preferences.tpl` действительно отображает их.

**Реализация**:
- Migration: +7 колонок в `users` (4 bool defaults true, smallint
  default 0, 2 bool defaults true).
- Backend Settings DTO + Update endpoint расширить.
- Frontend SettingsScreen: 7 чекбоксов/select.
- i18n.
- **Применение** preferences:
  - `show_all_*` — ConstructionsScreen/ResearchScreen/ShipyardScreen
    учитывают settings и фильтруют unavailable items.
  - `planetorder` — PlanetSidebar / planet-switcher используют
    серверный sort.
  - `esps` — espionage report viewer показывает разный уровень
    детализации.
  - `ipcheck` — login flow проверяет IP при auth (или auto-msg
    при разнице с last_login_ip).

  Эти 4 «применения» — отдельные подзадачи. Реализовать **только
  storage + update endpoint + UI form** в этом плане; реальные effects
  (filtering, sort, espionage detail, ipcheck) — в подпланах
  72.1.55.* по мере необходимости.
