## 1. Контекст

Legacy [`Ranking.class.php`](../../projects/game-legacy-php/src/game/page/Ranking.class.php)
поддерживает 11 типов сортировки + 4 mode'а + avg + пагинацию по 25.
72.1.12 закрыл базовые типы (b/r/u/a/e/dm/max/total) и 3 mode'а
(player/alliance/vacation). Осталось закрыть 1:1:

1. **avg-режим** — `(user.points / ((now - regtime)/86400))` —
   очки в день. POST-параметр `avg=1`.
2. **b_count / r_count / u_count** — агрегаты по таблицам:
   - `b_count = COUNT(*) FROM buildings JOIN planets WHERE planets.user_id=u.id`
   - `r_count = COUNT(*) FROM research WHERE user_id=u.id`
   - `u_count = SUM(count) FROM ships+defense JOIN planets`
3. **battles** — `users.battles` колонка (есть в схеме).
4. **player_observer / player_old_vacation** — два недостающих mode'а.
5. **Пагинация по 25** (legacy `USER_PER_PAGE`) — параметр `?page=N`.

## 2. Что нужно

### Backend (`internal/score/`)

#### `service.go::columnFor` расширить
Для `b_count`/`r_count`/`u_count` нужны не колонки `users`, а
суб-запросы. Менять `columnFor` бессмысленно — нужен отдельный
`Top`-метод с CTE/JOIN для агрегатов, либо ORDER BY суб-запросом.

Простой путь: расширить SQL `Top()` чтобы для соответствующих
типов делать LEFT JOIN с агрегатами. Тип-маппинг:
- `b` → `users.b_points` (есть).
- `r` → `users.r_points`.
- `u` → `users.u_points`.
- `a` → `users.a_points`.
- `e` → `users.e_points`.
- `dm` → `users.dm_points`.
- `max` → `users.max_points`.
- `total` → `users.points`.
- **новое** `b_count` → COUNT по buildings.
- **новое** `r_count` → COUNT по research.
- **новое** `u_count` → SUM по ships+defense.
- **новое** `battles` → users.battles.

Для `*_count` нужно добавить computed columns в SELECT, чтобы они
возвращались в Entry (даже когда сортировка по другому полю —
для UI показа всех значений, как в legacy).

#### avg-режим
Параметр `?avg=true`: финальное значение = stored / days_since_reg
(где days = greatest(1, (now - regtime).days)).

В backend: extra column `score_avg = users.points / GREATEST(1,
EXTRACT(EPOCH FROM (now() - users.regtime)) / 86400.0)` в SELECT.
ORDER BY этой column когда avg=true.

#### Mode расширить
`player_observer`: `WHERE u.is_observer = true`.
`player_old_vacation`: `WHERE u.umode = true AND u.last_seen <
now() - interval '11 days'` (legacy `time - (VACATION_DISABLE_TIME -
3*86400)` — VACATION=14d, итого 11d с last_seen).

#### Пагинация
- Параметр `?page=N` (1-indexed).
- LIMIT 25 OFFSET (page-1)*25.
- Возвращать `total_count` для UI.

### Frontend

- `api/highscore.ts`:
  - `ScoreType` расширить: `'b_count' | 'r_count' | 'u_count' | 'battles'`.
  - `Mode` расширить: `'player_observer' | 'player_old_vacation'`.
  - `fetchHighscore(type, options)`: `{ avg?, mode?, page? }`.
- `RankingScreen`:
  - 4 новых тип-селекта.
  - 2 новых mode-таба (observer, old_vacation).
  - Чекбокс «Среднее за день» (avg).
  - Пагинатор (page-input + ◀ ▶ кнопки).
  - В таблице игрока — колонки b_count, r_count, u_count, battles
    (как в legacy `playerstats.tpl`).
