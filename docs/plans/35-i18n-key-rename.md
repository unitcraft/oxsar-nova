# План 35 — i18n: убрать суффикс `Ui` из названий групп

## Проблема

Группы в `configs/i18n/ru.yml` / `en.yml` делятся на два класса:

- **С суффиксом `Ui`** (`fleetUi`, `galaxyUi`, `messagesUi`, …) — исторически
  добавлялись как frontend-ключи (`useTranslation('fleetUi')`).
- **Без суффикса** (`mission`, `assaultReport`, `alien`, `achievement`, …) —
  backend inbox-сообщения через `Bundle.Tr`.

Суффикс не несёт смысловой нагрузки и выглядит несогласованно. Цель —
единый стиль: все группы без суффикса.

## Принцип выполнения

**Ничего не должно сломаться.** Переименование чисто механическое — смысл ключей
не меняется, только имя группы. Каждая фаза должна оставлять проект в рабочем
состоянии: тесты зелёные, frontend собирается, UI показывает текст (не пустоты).

Порядок фаз критичен: сначала полная инвентаризация, потом YAML и frontend
**атомарно в одном коммите** — чтобы не было промежуточного состояния когда
YAML уже переименован, а frontend ещё читает старые имена (или наоборот).

## Фазы

### Ф.1 — инвентаризация и маппинг

1. Grep всех `*Ui:` групп в ru.yml → полный список.
2. Grep всех `useTranslation('*Ui')` в `frontend/src/` → список call sites.
3. Составить таблицу: `старое → новое` (просто отрезать `Ui`).
4. Проверить конфликты: нет ли уже группы без суффикса с тем же именем
   (например `achievements` vs `achievementsUi`). При конфликте — слить ключи
   вручную, проверить что нет дублирующихся ключей внутри объединённой группы.
5. Параллельно с п.4 — проверить все группы на предмет смыслового пересечения
   и дублирования текстов:
   - Найти буквально одинаковые строки в разных группах (grep по значению).
     Дубли — кандидаты на вынос в общую группу `common` или на объединение групп.
   - Проверить смысловую близость: группы одного домена (например `mission`
     и `fleetUi` — оба про флот) объединить в одну.
   - Критерий объединения: один домен + нет конфликта ключей → объединить.
     Разные домены или много ключей → оставить раздельно, не объединять насильно.
   - **Учитывать нагрузку на frontend**: каждый экран сейчас читает одну группу
     через `useTranslation('fooUi')` — это дешево. Если объединить несвязанные
     группы в одну большую, frontend будет загружать лишние ключи на каждом экране.
     Объединять только группы которые используются на одном и том же экране —
     тогда слияние не увеличивает объём загружаемых ключей.
   - Результат п.5 — итоговая таблица: `старое → новое (после возможных слияний)`.
5. Убедиться что список call sites полный: grep по всем вариантам
   (`useTranslation`, `i18next.t`, `Trans` с namespace) — ни один не пропущен.

### Ф.2 — переименование в YAML + frontend (один коммит)

Выполняется за один шаг чтобы не создавать сломанного промежуточного состояния:

1. В `ru.yml` и `en.yml`: переименовать все `fooUi:` → `foo:`.
2. Во всех `*.tsx` / `*.ts`: `useTranslation('fooUi')` → `useTranslation('foo')`.
3. После замен — grep по старым именам (`*Ui`) чтобы убедиться что ни один
   call site не пропущен. Если найден — исправить до коммита.

### Ф.3 — проверка перед коммитом

1. `go test ./backend/internal/i18n/...` — `TestI18nConsistency` PASS.
2. `npm run build` — без ошибок TypeScript и без warnings про missing keys.
3. `npm run lint` — без новых ошибок.
4. Запустить игру локально, пройтись по ключевым экранам (флот, галактика,
   постройки, сообщения) — весь текст отображается, пустот нет.
5. Только после всех проверок — коммит.

## Риски и страховки

- **Конфликт имён** (`foo` + `fooUi` оба существуют): слить вручную, проверить
  на дубли внутри группы — дубль ключа в YAML парсится молча (берётся последний).
- **Пропущенный call site**: страховка — grep по `*Ui` после замен. `TestI18nConsistency`
  это не поймает (он проверяет только ru↔en симметрию), но пустой текст в UI
  заметен при ручной проверке в браузере.
- **Атомарность**: YAML и frontend меняются в одном коммите. Нельзя пушить
  только YAML без frontend — это сломает прод.

## Коммиты

Каждая логически завершённая единица работы — отдельный коммит:

- **Ф.1**: `docs: план 35 Ф.1 — инвентаризация i18n групп, таблица маппинга`
- **Ф.2**: один или несколько коммитов по доменам если слияний много,
  например отдельно fleet, отдельно galaxy и т.д.; если замены механические
  без слияний — один коммит `refactor(i18n): убрать суффикс Ui из групп`

Не объединять Ф.1 и Ф.2 в один коммит — инвентаризация и фактические изменения
должны быть разделены для читаемости истории.
