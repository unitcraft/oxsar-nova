## 1. Контекст

Legacy `oxsar2-java/Assault.jar` после **реального** боя
(`!isBattleSimulation`, table-prefix без `sim_`) автоматически
инкрементит у каждого участника:

| Колонка       | Изменение                                        |
| ------------- | ------------------------------------------------ |
| `e_points`    | `+= battleExperience` (суммарный опыт игрока)    |
| `be_points`   | `+= battleExperience` (резервуар для апгрейда тех) |
| `battles`    | `+= 1` (счётчик боёв)                            |
| `points`     | `-= lostPoints` (общий рейтинг)                  |
| `u_points`   | `-= lostPoints` (рейтинг по юнитам)              |
| `u_count`    | `-= lostUnits` (общее число юнитов игрока)       |

И пишет лог-запись в `na_user_experience (time, isatter, userid,
experience, assaultid)`. В режиме симулятора (prefix `sim_`) — пропускает.

В `game-nova` (Go-движок, `projects/game-nova/backend/`) бой считает
`internal/battle.Calculate`, но инкременты пользователю **НЕ** применяются —
эта логика отсутствует. `battle_reports` пишутся в БД через
`internal/battlereport`, но `users.*` остаётся неизменным.

Подтверждение поведения — live-smoke в legacy: после
`java -cp Assault.jar:connector assault.Assault $DB ... na_ <assaultid>`
наблюдалось:
- `users[1172]` (атакующий): `e_points +4`, `be_points +4`, `battles +1`,
  `u_count -47`, `u_points` / `points` уменьшились на стоимость потерь.
- `users[1]` (защитник): `e_points +3`, `be_points +3`, `battles +1`,
  `u_count -23`.
- `na_user_experience`: 2 строки с `experience = 4 / 3`.

**Цель**: воспроизвести 1:1 в `game-nova`.

## 2. Источники истины

1. [`oxsar2-java/Participant.java:924-987`](../../d:\Sources\oxsar2-java\assault\src\assault\Participant.java)
   — главный кусок:
   - guard `!isBattleSimulation && userid > 0 && !useridReported.contains(userid)`,
   - `INSERT INTO user_experience`,
   - `UPDATE user += e_points/be_points/battles`,
   - и отдельно `UPDATE user GREATEST(0, points/u_points/u_count - lost*)`.

2. [`oxsar2-java/Assault.java:819-847`](../../d:\Sources\oxsar2-java\assault\src\assault\Assault.java)
   — **ФОРМУЛА опыта** (atan-based, **именно она действующая** в legacy):

   ```text
   atterExperience    = (atan(defStartPower/atkStartPower * 1.5 - 1.5) + 1)
                        * 0.4 * 3 * battleTurnsCoefficient + 1
   defenderExperience = (atan(atkStartPower/defStartPower * 1.5 - 1.5) + 1)
                        * 0.4 * 3 * battleTurnsCoefficient + 1
   ```

   Множители при моральных условиях (Assault.java:822-833):
   - если defender победил без потерь  → `atterExperience *= 3`
   - если attacker победил без потерь  → `defenderExperience *= 3`
   - и т. п.

   Множитель `battlePowerCoefficient` (Assault.java:836-843) —
   суммарная мощность × нормировка.

   Финальный round: `atterBattleExperience = (int) Math.round(atterExperience)`.

3. [`oxsar2-java/Units.java:113-115`](../../d:\Sources\oxsar2-java\assault\src\assault\Units.java)
   — стоимость юнита в очках:

   ```text
   points_per_unit = (cost.metal + cost.silicon + cost.hydrogen) / 1000 * 2
   ```

   Это и есть множитель в `lostPoints += quantity_lost × points_per_unit`
   (Participant.java:766).

4. [`oxsar2-java/Participant.java:759-771`](../../d:\Sources\oxsar2-java\assault\src\assault\Participant.java)
   — накопление `lostPoints / lostUnits`:

   ```text
   lostUnits  += curLostUnits          (число потерянных штук)
   lostPoints += curLostUnits × units.getPoints()  (сумма очков)
   ```

## 4. Что нужно сделать

### B. Миграция: users.u_count + таблица user_experience

- Файл: `migrations/0084_battle_user_stats.sql` (следующая свободная;
  проверить tail перед именованием).
- Стиль: см. [`0082_users_legacy_extra_points.sql`](../../projects/game-nova/migrations/0082_users_legacy_extra_points.sql)
  (комментарии why/why-not, `+goose Up/Down`).

```sql
-- +goose Up

ALTER TABLE users ADD COLUMN IF NOT EXISTS u_count integer
    NOT NULL DEFAULT 0 CHECK (u_count >= 0);

CREATE TABLE user_experience (
    id          bigserial   PRIMARY KEY,
    time        timestamptz NOT NULL DEFAULT now(),
    is_atter    boolean     NOT NULL,
    user_id     uuid        NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    experience  integer     NOT NULL,
    battle_id   uuid        REFERENCES battle_reports(id) ON DELETE SET NULL,
    -- idempotency: один и тот же user не получает опыт дважды за тот же
    -- бой даже если событие переиграется (см. §6 plan).
    UNIQUE (battle_id, user_id, is_atter)
);
CREATE INDEX ix_user_experience_user_id  ON user_experience(user_id, time DESC);
CREATE INDEX ix_user_experience_battle_id ON user_experience(battle_id);

-- +goose Down

DROP TABLE IF EXISTS user_experience;
ALTER TABLE users DROP COLUMN IF EXISTS u_count;
```

### C. Расширить `Report` / `SideResult`

  LostPoints float64 `json:"lost_points"` // Σ qty × (cost/1000*2)

### D. Новый слой: применить результат боя к юзерам

- Пакет: `internal/battlestats/apply.go` (выбрать имя без круговых
  импортов; рассмотрены варианты `battlestats`, `userstats`, или
  отдельный файл внутри `internal/battlereport/` — на твой выбор).
- Сигнатура (Go-стиль):

  ```go
  // ApplyBattleResult инкрементирует users.e_points/be_points/battles,
  // декрементит users.points/u_points/u_count, пишет в user_experience
  // по каждому участнику реального боя.
  // Идемпотентность гарантируется UNIQUE (battle_id, user_id, is_atter)
  // в user_experience — повторный вызов вернёт ErrAlreadyApplied.
  func ApplyBattleResult(
      ctx context.Context,
      tx pgx.Tx,
      report battle.Report,
      battleID uuid.UUID,
  ) error
  ```

- Логика — порт Participant.java:924-987:
  1. Один проход по `report.Attackers` и `report.Defenders`.
  2. Для каждого `SideResult`:
     - skip если `IsAliens` или `UserID` невалиден (нет в `users`);
     - skip если уже видели этот `UserID` в этом бою (legacy:
       `useridReported` set; для Go — локальная `map[uuid.UUID]bool`);
     - `INSERT INTO user_experience (is_atter, user_id, experience,
       battle_id) VALUES (...) ON CONFLICT (battle_id, user_id,
       is_atter) DO NOTHING RETURNING id` — если 0 строк затронуто,
       значит дубликат, дальше для этого юзера ничего не делаем
       (idempotency);
     - `UPDATE users SET e_points = e_points + $1, be_points =
       be_points + $1, battles = battles + 1 WHERE id = $2`;
     - `UPDATE users SET points = GREATEST(0, points - $1), u_points =
       GREATEST(0, u_points - $1), u_count = GREATEST(0, u_count - $2)
       WHERE id = $3`.
  3. Возврат `error`.

- pgx напрямую, без ORM/sqlc (CLAUDE.md правило для game-nova).

## 5. Ограничения

- **Паритет с legacy жёсткий**: формула опыта 1:1 atan-based, имена
  колонок 1:1 (`e_points, be_points, battles, u_count, u_points,
  points`). Не переименовывать.
- **Симулятор не трогаем** (`internal/simulator/handler.go`).
- **`NumSim > 1`** (multi-run в engine.go) — опыт начислять только за
  один реальный бой; multi-sim — это путь симулятора.
- **Idempotency**: если событие переиграется (event re-process), опыт
  не должен начислиться дважды. Гарантируется UNIQUE-индексом
  `(battle_id, user_id, is_atter)` в `user_experience`.
- **Без новых абстракций «на будущее»** (CLAUDE.md). `ApplyBattleResult`
  — одна функция, не интерфейс с моками.
- **i18n-правила родителя 72.1 §3** применяются: если в этой задаче
  потребуются user-facing строки (например, system-message «получено
  N очков опыта») — только через `t(group, key, vars)`, без `??
  'fallback'`, без хардкода рус/англ.
