================================================================================
0. НАЗНАЧЕНИЕ ДОКУМЕНТА
================================================================================

Документ описывает новую реализацию oxsar-nova: браузерной MMO-стратегии в
стиле OGame (метал/кристалл/дейтерий, флоты, бои, альянсы, экспедиции).
За основу берётся игровая логика проекта oxsar2; расчётные формулы,
ID юнитов, механики боя, цепочки событий, структура ресурсов и зависимости
сохраняются. Технологический стек переводится на Go (бэкенд) + TypeScript
(фронтенд). В ТЗ отдельно выделены механики OGame, которые в oxsar2
отсутствуют или не доведены до конца (см. раздел 12).


================================================================================
1. ЦЕЛИ И НЕ-ЦЕЛИ
================================================================================

1.1. Цели
---------
- Сохранить игровой баланс и механики oxsar2 (здания, исследования, юниты,
  бои, экспедиции, артефакты, альянсы).
- Устранить legacy-проблемы: Yii 1.1, прямое использование $_POST,
  отсутствие миграций, отсутствие тестов, sim_*-таблицы-песочницы рядом
  с продом.
- Реализовать недостающие механики OGame: маскировка (masking), баллистика
  (ballistics), полноценная система миссий флота, лунное уничтожение
  с формулой из OGame, бонусы офицеров, полный список достижений, Lifeforms
  опционально (см. 12.10).
- Ввести единую кодовую базу без «Sim*»-дублей: симуляция боя — та же
  функция, что и настоящий расчёт, просто без записи в БД.

1.2. Не-цели (вне scope первой версии)
--------------------------------------
- Миграция пользовательских данных из oxsar2 (старт с чистой БД).
- Поддержка старых соцсетевых авторизаций (ВК, Одноклассники, Mail.ru).
  В v1 — только e-mail + OAuth2 (Google). Остальное опционально.
- Мобильное нативное приложение (только адаптивный web).
- Платежный процессинг (credit-механика остаётся, но без реальных покупок
  в v1 — только админская выдача и внутренний обмен).

    и с PHP Assault.
  - Консты UNIT_* / EVENT_* берутся из game/Functions.inc.php
    (основной) и new_game/protected/config/consts.php (сводный
    индекс). При расхождении — игнорируем new_game/.

Правила для oxsar-nova:

  - Схема БД nova проектируется ЗАНОВО (PostgreSQL, goose-миграции),
    не переносится из oxsar2-schema.sql. Но при работе над
    соответствующим модулем разработчик ОБЯЗАН открыть соответст-
    вующий CREATE TABLE в oxsar2-schema.sql как референс: какие
    поля, какие типы, какие индексы были в легаси. Это не «копия»,
    а «проверка полноты».

  - Балансовые числа (стоимости, параметры кораблей, требования,
    достижения) НЕ берём «на глаз» и не копируем из OGame classic
    без сверки. Источник истины — table_dump/*.sql. На этапе M1.5
    пишется CLI cmd/tools/import-datasheets, который парсит эти
    dump'ы и генерит YAML в configs/. YAML ручной правки — только
    после этого, иначе уйдём в дрейф от legacy-баланса.

  - I18N-строки. na_phrases.sql (1489 ключей) — полный словарь
    игры на русском. Все UI-тексты в nova берутся ОТТУДА. Добавлять
    новый ключ можно, но «перепридумывать» существующий нельзя —
    это нарушит привычные игроку формулировки. См. новый §10.3.

  - Parametric formulas в na_construction (prod_metal, cons_energy,
    charge_metal и т.д.) — это мини-DSL. Мы НЕ портируем eval()
    PHP, а делаем в Go типизированный вычислитель с фиксированным
    набором переменных ({level}, {tech=N}, {temp}, {basic}). Это
    даёт паритет без уязвимости. См. новый §5.2.1.

  - Варианты schema-*.sql полезны для исторического контекста,
    но в nova не портируются. Разве что schema-change-2.0.sql
    можно почитать как пример «как делать большие миграции
    правильно» (этот опыт уходит в стандарты §17).

  - sql/useful-queries.sql — выписать в отдельный документ
    docs/admin-scenarios.md, превратить в admin-UI endpoints
    (см. §11).

2. ТЕХНОЛОГИЧЕСКИЙ СТЕК
================================================================================

2.1. Backend
------------
- Язык: Go 1.23+
- HTTP: стандартная библиотека net/http + chi router.
- Транспорт: REST (JSON) + WebSocket для real-time (чат, live-обновление
  очередей, уведомления о прилете флота).
- БД: PostgreSQL 16 (вместо MySQL — JSONB, range types, better locking).
- Кеш: Redis 7 (сессии, rate-limit, кеш галактики, pub/sub для WebSocket).
- ORM: sqlc (типобезопасные запросы из SQL) + ручной SQL для сложной
  агрегации. НИКАКОЙ рефлексивной ORM-магии, наследия WarperActiveRecord.
- Миграции: goose (up/down SQL-файлы) — в отличие от oxsar2, где миграций не
  было вообще.
- Планировщик: внутренний «tick» для event-loop, плюс cron-задачи внутри
  Go-процесса (без BAT-файлов).
- Аутентификация: JWT (access 15 мин) + refresh (httpOnly cookie, 30 дней).
- Логи: slog (structured) + OTel traces.
- Тесты: testify + dockertest (integration) + go-fuzz для боевой системы.

2.2. Frontend
-------------
- Язык: TypeScript 5.x (strict).
- Фреймворк: React 18 + Vite.
- Стейт: TanStack Query (server state) + Zustand (UI state). Redux не
  используется.
- Роутинг: TanStack Router (типобезопасный).
- Стили: CSS Modules + CSS variables (для тем/скинов).
- WebSocket-клиент: нативный WS + реконнект + heartbeat.
- Симулятор боя: чистый TS-модуль, переиспользуется сервером через
  перенос формул (бой — детерминированный, seed-based); дублирует
  серверную логику для офлайнового расчёта, как simulator.js в oxsar2.

2.2.1. Визуальный дизайн (полностью из oxsar2)
----------------------------------------------
Весь визуальный язык, UX-layout и графические ассеты берутся 1-в-1 из
legacy-проекта d:\Sources\oxsar2. Фронтенд переписывается на React/TS,
но «выглядеть» должен как старый oxsar2 — чтобы существующие игроки
не заметили смены движка.

Правила переноса:
- CSS переводится в CSS Modules, но классовая структура/цвета/размеры
  сохраняются. Тёмная скин-схема ui-darkness — как default.
- Темы пользователя (user_bg_style, user_table_style из users-таблицы)
  сохраняются: список us_bg/a-bg-*.css и us_table/table_td2_bg_*.css
  переезжает в configs/themes.yml, пользователь переключает их как
  раньше.
- Графика (иконки юнитов/зданий, preview-кадры, баннеры, ajax-лоадеры)
  копируется в assets/ фронта без переотрисовки. Ресайз и WebP/AVIF —
  на этапе сборки Vite, исходники не трогаем.
- Разметка экранов (Overview, Resources, Buildings, Shipyard, Galaxy,
  Fleet, Alliance, Messages, Highscore) воспроизводит табличные
  сетки и порядок колонок из соответствующих .tpl. Для каждого
  React-экрана в nova ставится комментарий-ссылка:
    // layout reference: oxsar2/www/templates/standard/<name>.tpl
  чтобы при доработке было ясно, откуда скопирована разметка.
- Smarty-директивы ({foreach}, {if}, {loop=...}, {@var}, {const=...})
  НЕ портируются один-в-один — React-компоненты используют .map(),
  тернарники, props. Но порядок элементов, table-сетки, классы
  сохраняются максимально близко.
- Клиентское поведение из www/js/ портируется как React-хуки:
    main.js   -> useAppShell, useMenu
    fleet.js  -> useFleetDispatcher
    simulator.js -> src/sim/battle.ts (единый движок с бэком)
    chat.js   -> useWebSocketChat
    stats.js  -> usePlayerStats
- Шрифты, курсор, стиль всплывающих подсказок (tooltip'ов), рамки
  кнопок, скин чата — без изменений.
- Всё, что добавляется из OGame-доработок (раздел 12), рисуется в том
  же стилевом ключе: новые иконки (masking, ballistics, офицеры,
  достижения) выполняются в стиле существующих ассетов oxsar2.
- Баннерные/платёжные .tpl (banner_*, appgrade-*, payment*) портировать
  в v1 не нужно — это legacy-интеграции соцсетей и платёжек (§1.2
  «не-цели» + §5.22 «платёжки в M9»).

2.3. Инфраструктура
-------------------
- Docker Compose для dev (postgres, redis, backend, frontend).
- GitHub Actions: линт (golangci-lint, eslint), тесты, сборка Docker-образа.
- Staging / production — разворачивается одной командой, конфиг через ENV
  (как env-vars, не как consts.local.php).

5.2. Ресурсы и экономика
------------------------
- Три ресурса: metal, silicon (кристалл в OGame), hydrogen (дейтерий).
  Хранятся как float64 на планете, целочисленно — в транзакциях.
- Производство в секунду вычисляется из уровня шахты, температуры,
  produce_factor, research_factor (plasma), бонусов артефактов/офицеров,
  и текущей доли энергии. Формулы — как в oxsar2 (metal/silicon/hydrogen
  rate = base * level * 1.1^level * factor). Точные коэффициенты взять
  из oxsar2 CronCommand.php::updatePlanetRes() и Planet_YII.php.
- Ёмкость хранилища растёт экспоненциально от уровня склада, модуль
  storage_factor. Переполнение — остановка добычи ресурса (как OGame).
- Энергия: сумма по солнечным станциям + спутникам + синтезатор. Недостача
  энергии масштабирует добычу всех шахт пропорционально (как OGame).
- GAMESPEED = 0.75 (берём как default, вынести в config; для серверов
  speed-x2/x5 менять одним параметром).

5.2.1. Parametric formulas из na_construction (ВАЖНО)
-----------------------------------------------------

В legacy-oxsar2 балансовые формулы хранятся как СТРОКИ в таблице
na_construction (см. §1.4). Пример для METALMINE (buildingid=1):

  prod_metal    = floor(30 * {level} * pow(1.1+{tech=23}*0.0006, {level}))
  cons_energy   = floor(10 * {level} * pow(1.1-{tech=18}*0.0005, {level}))
  charge_metal  = floor({basic} * pow(1.5, ({level} - 1)))

Для HYDROGEN_LAB (buildingid=3) — формула ещё зависит от температуры:

  prod_hydrogen = floor(10 * {level} * pow(1.1+{tech=25}*0.0008, {level})
                         * (-0.002 * {temp} + 1.28))

DSL-переменные:

  {level}         — уровень здания/исследования/количество единиц
  {basic}         — базовое значение поля basic_metal / basic_silicon /
                    basic_hydrogen / basic_energy / basic_credit
  {tech=N}        — уровень исследования с buildingid=N в research2user
                    (23 = laser_tech, 18 = energy_tech, 25 = plasma_tech)
  {temp}          — средняя температура планеты (для hydrogen_lab)

Реализация в nova:

  pkg/formula/                -- evaluator DSL
    parser.go                 -- разбор строки в AST
    eval.go                   -- исполнение с контекстом
    context.go                -- безопасный набор переменных
                                 (level int, basic int64, tech map[int]int,
                                 temp int)
    funcs.go                  -- whitelist функций: floor, ceil, round,
                                 pow, min, max, sqrt. НИЧЕГО БОЛЬШЕ.
                                 Никакого eval, никакого PHP-интерпретатора.

  configs/buildings.yml       -- содержит строки-формулы в том же
                                 синтаксисе, что oxsar2. Экспортируются
                                 конвертером import-datasheets из
                                 table_dump/na_construction.sql.

Почему не хардкодить формулы в Go: если геймдизайн хочет изменить
множитель с 1.5 на 1.48 — это правка YAML без редеплоя Go-кода.
Если формулы зашиты — каждая правка требует Go-ревью и билда.

Почему не использовать goja или другой JS/lua runtime: overkill
и attack surface. Наш DSL — 7 функций и 4 переменных, пишется
за выходные, проверяется fuzz-тестами.

Это ЗАКРЫВАЕТ ранее обозначенную задачу §5.2 «Точные коэффициенты
взять из oxsar2 CronCommand.php::updatePlanetRes() и Planet_YII.php».
На деле этих формул в PHP-коде нет — они в na_construction.

Прежнее утверждение (версии ТЗ 1.0–1.2) о том, что "настоящий бой
в oxsar2 считался через JAR" — ОШИБКА. JAR — это отдельный
симулятор-калькулятор, а не прод-путь.

PHP-BattleComponent.php в new_game/ — обрубок-прототип, не
использовать.

Задача портирования: получить Go-движок, который бьётся с ОБОИМИ
эталонами (А и Б). При расхождении между PHP и Java — сверяем с
расчётами OGame classic и принимаем решение в ADR. Вероятнее всего,
PHP-версия (А) ближе к фактическому поведению, которое видели
игроки, поэтому при неразрешимых конфликтах приоритет за ней.

Состав java-движка (d:\Sources\oxsar2-java\assault\src\assault\):
  Assault.java        1941 строк — ядро: раунды, распределение
                                   выстрелов, rapidfire, masking,
                                   ballistics, debris, формирование
                                   боевого отчёта.
  Participant.java     988 строк — сторона боя: технологии владельца
                                   (attack/shield/shell/laser/ion/
                                   plasma/ballistics/masking levels),
                                   ресурсы, флот, луна/планета,
                                   подсчёт потерь и очков.
  Party.java           778 строк — группировка участников одной
                                   стороны (ACS, инопланетяне).
  Units.java           650 строк — unit-stats, damaged/shell_percent,
                                   front, weight, rapidfire-таблицы.
  Database.java        126 строк — выгрузка входа/записи результата
                                   в MySQL (в nova НЕ портируем,
                                   см. ниже).

Задача — переписать всю боевую логику java-пакета assault на Go в
internal/battle/. Database.java не портируется: в Go-версии чтение
входа и запись отчёта идёт через sqlc-репозиторий Postgres рядом
с движком, а сам battle.Calculate(input BattleInput) остаётся
чистой функцией без I/O.

Высокоуровневый алгоритм (как в Assault.java):

  for turn := 0; turn < 6; turn++ {
      prepareTurn(turn)                 // пересчёт sum_weight, щитов
      applyArtefacts(turn)              // BATTLE-артефакты стороны
      for each participant {
          distributeShots(participant)  // c учётом ballistics/masking
      }
      resolveDamage()                   // щит -> броня -> уничтожение
      finishTurn()
      if aliveSides <= 1 { break }
  }
  buildReport()   // раунды, потери, lootable, debris, moon chance

Ключевые поля юнита (Units.java): unit_id, mode, quantity, damaged,
shell_percent, front (вес = 2^front), attack[0..2], shield[0..2],
shell. Три «канала» урона (index 0..2) — обычный/laser-mode/ion-mode
(extended battle, see Assault.java isExtended).

Обязательный паритет с Java-движком:
- Rapidfire-таблицы (rapid_fire_by_unit в Assault.java) переносятся
  один-в-один в configs/rapidfire.yml.
- Формулы applyBallistics() и applyMasking() из Assault.java
  портируются без изменений (это закрывает TODO, который остался
  в прототипе BattleComponent.php — см. 12.1, там теперь только
  ссылка на факт портирования).
- primaryTargetUnitid (Participant.java) — поддержать: возможность
  выбрать приоритетную цель для своего флота.
- Учёт технологий владельца: attack = 1 + 0.1 * gunTech,
  shield = 1 + 0.1 * shieldTech, shell = 1 + 0.1 * shellTech и т.д.
  Точные множители берутся из Participant.java (applyLevels).

Детерминированность: в Java-оригинале используется java.util.Random
с системным seed. В Go-порте боевая функция ОБЯЗАТЕЛЬНО принимает
seed (uint64), все random-вызовы идут через pkg/rng. Тот же seed
и тот же вход => бит-в-бит тот же отчёт. Это нужно для:
  - симулятора боя на фронте (src/sim/) — работает на том же seed,
    что и сервер, с кросс-компиляцией формул в TS;
  - репродьюса инцидентов (в репорт пишется seed, по тикету можно
    прогнать заново);
  - property-based тестов (раздел 14.2).

Верификация паритета:
- Прогоняем пачку боёв через исходный Assault.jar (d:\Sources\
  oxsar2-java\assault\dist\oxsar2-java.jar) и через новый Go-движок
  с одним и тем же входом.
- Допускается расхождение ТОЛЬКО в местах, где мы явно меняем
  формулу (например, формула луны в 12.2 или новые OGame-механики
  раздела 12). Всё остальное должно совпадать по урону и потерям
  с точностью до целых юнитов.

5.10.1. Персональные факторы игрока и планеты
---------------------------------------------
Ключевая архитектурная особенность, которую НЕЛЬЗЯ терять при порте:
эффекты ряда артефактов «материализованы» как поля в БД, а не как
список активных бонусов. Это проще для чтения (добыча = base *
produce_factor, никаких join'ов), но требует аккуратного учёта
при активации/деактивации.

Шесть персональных полей-множителей:
  users.exchange_rate         -- устанавливается 1.2 ↔ 1.03
                                 ARTEFACT_MERCHANTS_MARK (300)
  users.research_factor       -- изменяется ± 1
                                 ARTEFACT_SUPERCOMPUTER (305)
  planets.build_factor        -- изменяется ± 1 на КОНКРЕТНОЙ планете
                                 ARTEFACT_ROBOT_CONTROL_SYSTEM (315)
  planets.produce_factor      -- изменяется ± 0.1 на ВСЕХ планетах игрока
                                 ARTEFACT_CATALYST (301)
  planets.energy_factor       -- изменяется ± 0.15 на ВСЕХ планетах
                                 ARTEFACT_POWER_GENERATOR (302)
  planets.storage_factor      -- изменяется ± 0.15 на ВСЕХ планетах
                                 ARTEFACT_ATOMIC_DENSIFIER (303)

Дефолтные значения (Artefact::resyncUser, строки 1225–1229):
  users.exchange_rate          = 1.2
  users.research_factor        = 1
  planets.build_factor         = 1
  planets.produce_factor       = 1
  planets.energy_factor        = 1
  planets.storage_factor       = 1

Правила применения:
- Активация артефакта: UPDATE поля с оператором $op = '+' или '-'
  (см. switch в Artefact::activateArtefact / deactivateArtefact).
- Деактивация — зеркальная операция, чтобы поле вернулось в исходное
  значение ТОЛЬКО при отсутствии других активных артефактов того же
  класса. Если два CATALYST'а активны, produce_factor = 1 + 0.1 + 0.1
  = 1.2. Снять один — 1.1, снять оба — 1.0.
- MERCHANTS_MARK — исключение: он не additive, а set (1.2 → 1.03),
  поэтому активировать второй запрещено:
    if (getActiveCount(type_id, user_id, art_id) > 0) return false;
- Исследования и здания «добавленные артефактом» идут в отдельные
  поля research2user.added, building2planet.added (legacy-схема),
  чтобы resyncUser мог вернуть все back к базовому level.
- Cron-команда ResyncArtefacts (scripts/cron-win32/ResyncArtefacts
  .bat) периодически сбрасывает и переприменяет все активные
  артефакты — защита от накопленной рассинхронизации.

В nova:
- Поля сохраняем один-в-один (миграции 0001 и 0002 уже это делают,
  проверка в §4).
- Доменный пакет artefact публикует функции Apply(ctx, tx, userID,
  artefactInstance) и Revert(ctx, tx, userID, artefactInstance).
- Resync — отдельная CLI-команда в cmd/tools, плюс админский endpoint.
- Тесты: activate/revert на single-thread — factor возвращается в
  дефолт; activate/activate/revert — частично откатывается.

================================================================================
6. EVENT-LOOP
================================================================================

Единый источник игрового времени — таблица events.

6.1. Принципы
-------------
- Все действия, имеющие задержку (стройка, полёт, исследование, экспирация
  артефакта, экспедиция), создают запись в events с fire_at.
- Воркер каждые EVENT_BATCH_PROCESS_TIME=10 сек выбирает события,
  у которых fire_at <= now() AND state=WAIT, блокирует их FOR UPDATE SKIP
  LOCKED, обрабатывает по типу, переводит в OK / ERROR.
- При пользовательском запросе «дай мне текущее состояние планеты»
  бэкенд синхронно прогоняет pending-события этого игрока до now(),
  чтобы не было рассинхрона UI (так же как CronCommand перед запросом).
- Обработка события должна быть идемпотентной: повторный запуск с тем же
  event_id не должен дублировать эффект.

6.2. Типы событий — как EVENT_* в oxsar2 (раздел 5.6) + новые:
  EVENT_MOON_DESTROY_RESULT    — рекалькуляция после уничтожения
  EVENT_LIFEFORM_TICK          — если включены Lifeforms (12.10)

6.3. Ретрит (отзыв флота)
-------------------------
- Игрок может отозвать свой флот из полёта (EVENT_RETURN).
- Допустимые миссии — из RETREAT_FLEET_EVENTS в consts.php.
- RETREAT_EVENT_BLOCK_END_TIME = 5 сек — нельзя отозвать флот за
  5 сек до прибытия.

10.3. Локализация (i18n)
------------------------

Источник истины: d:\Sources\oxsar2\sql\table_dump\na_phrases.sql.

Факты из legacy:
  - 1489 русских строк (languageid=1) в 24 группах:
    global, error, info, mission, buildings, Resource, UnitInfo,
    UserAgreement, Registration, Statistics, и т.д.
  - Формат ключа: UPPERCASE_SNAKE_CASE (ACCOUNT_WILL_BE_DELETED,
    ALLIANCE_ATTACK, PLANET_UNDER_ATTACK, UNKOWN_MISSION, ...).
  - Ключ + группа уникальны в рамках языка.
  - Контент — строка, допустимо HTML (<b>, <br>) и %s/%d-
    подстановки.

Реализация в nova:

  configs/i18n/ru.yml          -- сгенерировано из na_phrases.sql
                                   конвертером import-phrases,
                                   разбито по группам на подразделы.
  configs/i18n/en.yml          -- перевод на английский, переведено
                                   вручную (в legacy английского не
                                   было — это gap, который мы
                                   закрываем).

  Формат YAML:
    global:
      ACCOUNT_WILL_BE_DELETED: "Аккаунт будет удалён!"
      ALLIANCE: "Альянс"
      ATTACK: "Атака"
    mission:
      COLONIZE_FAILED: "Колонизация не удалась."
    error:
      PLANET_UNDER_ATTACK: "Планета атакована, действие невозможно."
    ...

  Backend:
    internal/i18n/
      loader.go      -- загружает ru.yml + en.yml в immutable map
      lookup.go      -- Tr(lang, group, key, args...) -> string
                        Если ключ отсутствует в запрошенном языке,
                        падает в ru (fallback). Если и в ru нет —
                        возвращает "[group.key]" как маркер.

  Использование:
    - AutoMsg (§5.19) рендерит шаблоны, беря текст из i18n:
        i18n.Tr(user.Lang, "mission", "ATTACK_REPORT_SUBJECT", ...)
    - HTTP-ошибки наружу — по ключу error.*, а не по строке.
    - Fronтенд: получает язык пользователя + fetch /api/i18n/{lang}
      один раз на сессию, кеширует.

Frontend i18n:
  - src/i18n/dicts/ru.ts, en.ts — копия YAML, сгенерированная
    скриптом gen:i18n (читает configs/i18n/*.yml).
  - Хук useTranslation(group) возвращает t(key, args) с
    TypeScript-автодополнением благодаря codegen.

Правила:
  - Не переизобретать ключи. Если в legacy есть UNKOWN_MISSION
    (да, там опечатка), добавляем как UNKOWN_MISSION —
    НЕ исправляем на UNKNOWN. Это критично для совместимости
    с сохранёнными боевыми отчётами и документацией сообщества.
  - Новые ключи — в новых группах или с префиксом NOVA_
    (NOVA_REPAIR_STARTED), чтобы в diff были видны добавления.
  - Перевод на английский — обязательная часть PR, даже если
    черновой. Ревизор-native-speaker правит отдельным PR.

================================================================================
12. ДОРАБОТКИ ИЗ OGAME (ЧЕГО НЕТ В OXSAR2)
================================================================================

В этом разделе — механики, которых в oxsar2 либо нет, либо реализованы
частично. Все они обязательны к реализации в oxsar-nova.

12.1. Маскировка и баллистика
-----------------------------
Технологии UNIT_BALLISTICS_TECH (103) и UNIT_MASKING_TECH (104) уже
реализованы в production java-движке d:\Sources\oxsar2-java
(см. Participant.java: ballisticsLevel, maskingLevel и их применение
в Assault.java). «TODO: apply masking & ballistics» стоит только в
прототипе oxsar2/BattleComponent.php, который мы всё равно не берём
за основу (см. 5.7).

Задача для oxsar-nova — не «придумать формулу», а портировать
готовый Java-код в Go без изменения семантики. Все числовые
коэффициенты, cap'ы и целевые классы юнитов (для кого работает
masking) берутся из Assault.java/Participant.java. Никаких новых
формул не вводим — паритет с Java-движком.

12.2. Формула появления луны
----------------------------
OGame: P = min(debris_metal + debris_silicon, 1_000_000) / 1_000_000 * 20%.
В oxsar2 формула упрощена. Сделать по OGame, результат лога — в
moon_creation_stats, событие EVENT_MOON_DESTRUCTION_RESULT (не путать
с уничтожением).

12.3. Уничтожение луны (RIP-миссия)
-----------------------------------
- Только на ударах DEATH_STAR (UNIT_DEATH_STAR=42).
- Вероятности успеха и гибели звезды смерти — по формулам OGame:
    chance_destroy = (100 - sqrt(moon_diameter)) * sqrt(rip_count)
    chance_rip_die = sqrt(moon_diameter) / 2
- Событие EVENT_ATTACK_DESTROY_MOON уже описано в consts.php; довести
  логику до паритета.

12.4. Межпланетные ракеты
-------------------------
- UNIT_INTERPLANETARY_ROCKET (52). Атакуют только оборонительные юниты
  цели.
- Дальность: impulse_engine_level * 5 - 15 систем (формула OGame).
- Перехват: UNIT_INTERCEPTOR_ROCKET (51) сбивает по 1:1.
- Миссия — не флотовая, отдельный event EVENT_ROCKET_ATTACK. В oxsar2
  описана в consts.php, но код неполный. Реализовать до конца.

12.5. Astrophysics / параллельные исследования
----------------------------------------------
- Astrophysics в OGame даёт +1 слот колонизации на каждые два уровня
  и до 3 параллельных исследований (через интергалактическую сеть).
- В oxsar2 аналога astrophysics нет, только expo_tech.
- Добавить новое исследование ASTRO_TECH (новый unit_id, например 112)
  с такой же механикой.

12.6. Интергалактическая исследовательская сеть
-----------------------------------------------
- Research network в OGame позволяет суммировать лаборатории N планет
  (N = network_level). В oxsar2 частично присутствует через
  research_factor, но сети нет.
- Добавить: research_network_level на user (новый технологический
  unit, условно 113). На расчёте времени исследования суммируется
  min(N, network_level) планет с research_lab >= level_of_research.

12.7. Полный каталог достижений
-------------------------------
- В oxsar2 есть achievements, но ACHIEVEMENTS_ENABLED=false в DM и
  частично заполнен только newbie-набор.
- Список обязательных achievements (configs/achievements.yml):
    NEWBIE_END, FIRST_COLONY, FIRST_MOON, FIRST_ATTACK_WIN,
    DESTROYED_FLEETS_x (tiers), BUILT_SHIPS_x, RESEARCH_x,
    POINTS_x, EXPEDITIONS_x, MOON_DESTROYED, DEATH_STAR_BUILT.

12.8. Офицеры
-------------
- В oxsar2 уже есть таблица na_officer, но без контента.

    Admiral               — +1 слот флота

12.9. Ежедневные награды, достижения входа, событийные турниры

12.10. Lifeforms (опционально, флаг конфигурации)

M5.0.2. Cron resync-artefacts — периодический пересчёт factor-полей
    (эквивалент oxsar2 CronCommand::actionResyncArtefacts +
    scripts/cron-win32/ResyncArtefacts.bat).
M5.1. Artefact Market (§5.17) — UI покупки за credit.
M5.2. Alien AI (§5.18) — EVENT_ALIEN_* полный набор.
M6. Альянсы, чат, сообщения, биржа.
M7. Офицеры, достижения, ежедневные награды, туториал (§5.20).

строк Java. Сроки милестоунов M4–M7 пересмотрены в большую
сторону, M4 теперь имеет M4.1/M4.2, M5 — M5.1/M5.2, M7 — M7.1.


================================================================================
17. СТАНДАРТЫ РАЗРАБОТКИ
================================================================================

Раздел фиксирует обязательные практики. Цель — чтобы oxsar-nova с
первого коммита был проектом, в который легко зайти новому разработчику
и легко поддерживать через 5 лет. Legacy-анти-практики oxsar2
(прямые $_POST, отсутствие миграций, отсутствие тестов, нет CI,
нет CLAUDE.md) не допускаются.

17.1. Репозиторий и ветвление
-----------------------------
- Монорепо: /backend (Go), /frontend (TS), /configs, /migrations,
  /testdata, /docs, /deploy. Один репозиторий, общий CI.
- Trunk-based development: main всегда deployable. Фичи — короткие
  ветки feat/<slug>, fix/<slug>, chore/<slug>. Rebase, не merge
  commits. Длительные фичи — под feature-flag в config.
- Conventional Commits (feat:, fix:, chore:, refactor:, test:,
  docs:, perf:, build:, ci:). Обязательно для changelog-генерации.
- Semantic Versioning для API и релизных тэгов.
- Squash-merge в main. История PR сохраняется в описании.
- Защищённый main: запрещён прямой push, обязателен PR + 1 review +
  зелёный CI.
- Все PR требуют заполненного шаблона: что/зачем/как тестировалось/
  риски.
- Changelog генерируется автоматически из commit-сообщений (release
  please или аналог).

17.2. Go backend — стандарты
----------------------------
- Target: Go 1.23+, один go.mod в /backend.
- Форматирование: gofmt + goimports (strict). CI падает при
  расхождении.
- Линтеры: golangci-lint с набором: errcheck, govet, staticcheck,
  revive, gocritic, gosec, bodyclose, contextcheck, nilerr,
  prealloc, errorlint, misspell. Конфиг в .golangci.yml.
- Запрещено: init() с побочными эффектами, глобальные изменяемые
  переменные (кроме конфига, загруженного один раз на старте),
  panic в прод-коде (кроме main.go bootstrap), interface{}/any в
  публичных API без явной причины.
- Обработка ошибок: errors.Is / errors.As, обёртка через
  fmt.Errorf("context: %w", err). Никаких голых strings-ошибок
  между слоями. Domain-ошибки — typed (например, battle.ErrInvalid
  Input).
- Контекст: ctx первым параметром у любого IO/долгого вызова.
  Передаётся по цепочке. Запрещён context.Background() в бизнес-
  коде (только main.go и воркеры).
- Структурное логирование: log/slog с полями user_id, planet_id,
  event_id, trace_id. Никаких log.Println/fmt.Println в прод-коде.
- Dependency injection: руками через конструкторы (New*). Никаких
  DI-контейнеров. Никаких service locator'ов.
- БД-доступ: только через sqlc-сгенерированные методы. Сырой SQL
  допустим только для сложных агрегатов и тоже лежит в /queries/.
  Транзакции — через явный repo.InTx(ctx, fn). Запрещён доступ из
  handler в БД минуя service-слой.
- Конкурентность: errgroup + context для фанаута, мьютексы только
  в пакетах с явным state (кеш), в остальном — каналы и immutable
  данные. go-routine-ы всегда имеют понятный конец жизни.
- Тайм-ауты: любой HTTP-клиент и любой SQL-вызов имеют deadline.
- Graceful shutdown: SIGTERM -> прекратить приём запросов, дождаться
  in-flight, остановить воркеры, закрыть пул БД. Таймаут 30 сек.
- Публичные пакеты имеют godoc для всех экспортируемых сущностей.
  Внутренние — там, где смысл неочевиден.

17.3. TypeScript frontend — стандарты
-------------------------------------
- tsconfig: strict, noUncheckedIndexedAccess, noImplicitOverride,
  exactOptionalPropertyTypes.
- ESLint + typescript-eslint (recommended-type-checked),
  eslint-plugin-react, eslint-plugin-react-hooks,
  eslint-plugin-import, eslint-plugin-jsx-a11y.
- Prettier — единственный форматтер. Конфликт с ESLint решён через
  eslint-config-prettier.
- Запрещено: any, @ts-ignore (только @ts-expect-error с описанием),
  default exports (кроме роут-модулей), console.log в прод-коде.
- API-слой генерируется из OpenAPI (openapi-typescript). Ручные
  типы DTO запрещены.
- Компоненты: функциональные, без classComponent. Код-сплиттинг
  по роутам. Suspense + ErrorBoundary на каждом маршруте.
- State: server state — TanStack Query, UI state — Zustand.
  Смешивать запрещено.
- Accessibility: jsx-a11y в CI, все интерактивные элементы имеют
  role/aria-label/keyboard.
- Формы: react-hook-form + zod. Валидация на клиенте ИДЕНТИЧНА
  серверной (zod-схема генерируется из того же источника, что и
  go-валидация, или пишется парой в одном PR).

17.4. Контракты API
-------------------
- OpenAPI 3.1 — источник истины для REST. В CI проверяется, что
  handler-ы соответствуют спецификации (oapi-codegen или equiv.).
- JSON Schema — источник истины для WebSocket-событий. Из него
  кодогенерация и в Go (pkg/proto), и в TS (src/api/proto).
- Breaking changes в API — только через версионирование (/v1, /v2).
  Deprecated-эндпоинты помечаются заголовком и удаляются не раньше,
  чем через 1 major-релиз.
- ID и временные метки в API — строки (uuid, RFC3339). Никаких
  числовых time_t и autoincrement int в публичных контрактах.

17.5. База данных
-----------------
- Миграции: goose, только append-only, down-миграции обязательны
  для первых 30 дней после релиза.
- Именование: snake_case, таблицы — во множественном числе
  (users, planets). Никаких префиксов na_ (как в oxsar2).
- PK: bigint generated always as identity или uuid v7.
- Индексы — с префиксом ix_<table>_<cols>. Unique — ux_.
- Все денежные/ресурсные поля — numeric(20, 0) для целых,
  numeric(20, 4) для дробей. Никаких float для ресурсов (в oxsar2
  было double — копить 1e15 металла уже ломало точность).
- Timestamptz, не timestamp. TIME ZONE — UTC везде.
- Soft-delete через deleted_at для пользовательского контента
  (сообщения, заметки), hard-delete для машинных таблиц (events
  после обработки).
- Партиционирование по времени для growing-таблиц: res_log,
  assaults, events_processed, chat_messages.

17.6. Тестирование
------------------
- Coverage threshold: backend ≥ 70% изменённых строк, критичные
  пакеты (battle, economy, event) ≥ 85%. CI блокирует merge при
  деградации.
- Пирамида: unit (быстрые, без IO) > integration (dockertest) >
  e2e (Playwright). Обратная пирамида (много e2e, мало unit) —
  запрещена.
- Детерминизм: все тесты должны проходить локально и в CI с
  одинаковым результатом. Flaky-тесты квотируются — retry 3 раза,
  если не стабилизировался за сутки, карантин + тикет.
- Тесты боя — property-based (gopter/rapid) + golden files
  (testdata/battle/*.json).
- Нет shared mutable state между тестами. t.Parallel() по умолчанию.
- Seed в тестах боя фиксированный для репродьюса.

17.7. Наблюдаемость
-------------------
- OpenTelemetry traces на все HTTP/WS-запросы и обработку событий.
- Метрики Prometheus: RED (Rate, Errors, Duration) для HTTP,
  USE (Utilization, Saturation, Errors) для БД-пула и воркеров,
  продуктовые (новые регистрации/час, активных флотов, время
  между fire_at и processed_at).
- Логи: JSON, ELK-совместимые, с trace_id. PII (email, IP) —
  либо хешируется, либо попадает только в отдельный stream
  с коротким retention.
- Alerting (Alertmanager): SLO-burn, event-loop лаг > 30 сек,
  ошибка 5xx > 1%, очередь задач растёт быстрее, чем обрабатывается.
- Dashboards в Grafana: по одному на домен (Auth, Planets, Fleet,
  Battle, Alliance). Разворачиваются как код (Jsonnet/Grafonnet).

17.8. CI/CD
-----------
- GitHub Actions. Джобы: lint, test, build, docker, migration-
  check, openapi-breaking-check, frontend-bundle-size.
- Все зависимости пиннутся: go.mod с sums, package-lock.json в
  git, Docker base-images по дайджесту (не :latest).
- Security-сканы в CI: gosec, npm audit, trivy для образов,
  gitleaks для секретов. Critical-findings блокируют merge.
- SBOM (Syft) генерируется на каждый релиз.
- Образы подписываются cosign. Прод-деплой принимает только
  подписанные образы.
- Деплой: blue/green через feature-флаги или canary (5% → 50% →
  100%). Автороллбек по SLO-burn.

17.9. Документация и AI-ready кодовая база
------------------------------------------
- CLAUDE.md в корне репозитория: как запустить, где что лежит,
  правила стиля. В oxsar2 отсутствовал — в nova это обязательное
  артефакт первого коммита.
- /docs: ADR (Architecture Decision Records) для всех нетривиальных
  решений. Формат — MADR.
- README.md на каждом верхнем пакете Go и каждой feature-папке
  фронта.
- Комментарии в коде — только про «почему», никогда про «что»
  (это задача нейминга).

17.10. Безопасность как процесс
-------------------------------
- Threat model на фичу с внешним поверхностным слоем (регистрация,
  OAuth, рынок, ракеты) документируется как ADR.
- Зависимости обновляются через Dependabot/Renovate еженедельно.
  Критические CVE — out-of-band релиз в течение 48 часов.
- Все секреты — через secret manager окружения. В репо — только
  .env.example. pre-commit hook gitleaks.
- Rate-limit на все публичные эндпоинты (Redis token bucket).
- Аудит-лог: любое административное действие + любое изменение
  прав, балансов, артефактов — в audit_log с before/after JSONB.

17.11. Производительность
-------------------------
- Бюджеты: P99 /api/* < 250 мс, событие в event-loop от fire_at
  до processed_at < 10 сек при нормальной нагрузке, < 60 сек при
  пиковой.
- N+1 к БД — баг. В PR с новым SQL обязателен pg_stat или
  комментарий про план.
- Пагинация: только keyset (seek method), никаких OFFSET на
  больших таблицах.
- Горячие чтения (галактика, highscore) — через Redis-кеш с явной
  инвалидацией по событию. TTL — как fallback, не как основной
  механизм.

17.12. Код-ревью
----------------
- Минимум один approve, для изменений в battle/economy/security —
  два, один из которых — maintainer соответствующего домена.
- Review checklist (в PR-шаблоне): тесты, миграции вверх и вниз,
  обратная совместимость API, обновлена ли документация,
  покрытие, нет ли лишней абстракции.
- Размер PR: целевой ≤ 400 строк diff. Большие фичи — серия PR
  за флагом.


================================================================================
18. БЕЗОПАСНЫЕ УЛУЧШЕНИЯ МЕХАНИКИ, ПРИНЯТЫЕ СРАЗУ
================================================================================

Раздел перечисляет изменения относительно oxsar2, которые признаны
безопасными (не ломают игровой баланс, не требуют отдельного
обсуждения у геймдизайна) и делаются сразу в v1. Каждое изменение —
с обоснованием «почему безопасно».

18.1. Численные типы ресурсов
-----------------------------
В oxsar2 metal/silicon/hydrogen хранились как MySQL double. На
длинных горизонтах (триллионы ресурсов) появлялась потеря точности.
В nova: numeric(20,4) в Postgres, int64 в Go на уровне бизнес-логики.
  Почему безопасно: уровень точности выше, чем был; баланс
  не трогаем.

18.2. Отказ от Sim-таблиц
-------------------------
В oxsar2 была параллельная схема sim_* и дубли моделей Sim*_YII.php
для симулятора боя. В nova симулятор — чистая функция.
  Почему безопасно: это реорганизация кода, не поведения.
  Результаты прогона Sim и Real должны совпадать — в oxsar2 этого
  гарантии не было, мы эту гарантию вводим.

18.3. Идемпотентность event-loop
--------------------------------
В oxsar2 CronCommand при падении в середине обработки мог оставить
полусостояние. В nova каждый обработчик события — идемпотентный,
событие помечается processed только при коммите транзакции.
  Почему безопасно: устраняет баги, эффект для игрока — меньше
  "застрявших" очередей и дубликатов при ретраях.

18.4. Детерминированный бой по seed
-----------------------------------
В Java-движке используется системный Random. В nova seed пишется в
отчёт боя и лог. По seed + входу — полный репродьюс.
  Почему безопасно: меняется инфраструктура расчёта, не распределение
  вероятностей. RNG-алгоритм переносим так, чтобы статистически
  совпадал с Java-оригиналом (см. 14.4).

18.5. Отмена очереди за 15 секунд
---------------------------------
В oxsar2 EV_ABORT_SAVE_TIME = 15 сек — корректная, безопасная
механика. Сохраняем без изменений, но явно выносим в config и
покрываем тестом.
  Почему безопасно: ровно то же поведение, только верифицировано.

18.6. Bashing / Ally-block — включить по умолчанию
--------------------------------------------------
В oxsar2 BASHING_PERIOD = 0 и BASHING_MAX_ATTACKS = 0 — фактически
выключено, что приводило к стейкингу одной цели. В nova: включаем
по умолчанию 6 атак/24 часа на цель, BLOCK_TARGET_BY_ALLY_ATTACK_
USED_PERIOD = 10 минут.
  Почему безопасно: это отраслевой дефолт OGame, защищает слабых
  игроков. Значения — в конфиге, админ вселенной может вернуть 0.

18.7. Защита новичков уточнена
------------------------------
Оставляем пороги 3000/10000/1000000 очков. Добавляем OGame-правило:
если жертва неактивна более 7 дней, верхний порог не применяется
(чтобы «спящие» не были безнаказанной кормушкой-невидимкой).
  Почему безопасно: это OGame-стандарт; защищает активных новичков,
  не ломает экономику ветеранов.

18.8. Режим отпуска — минимум 48 часов
--------------------------------------
В oxsar2 VACATION_DISABLE_TIME был выражен в секундах без явного
нижнего порога. В nova — минимум 48 часов включённого umode
(как OGame), досрочно выйти нельзя.
  Почему безопасно: стандарт OGame, защищает от абуза umode как
  щита во время атаки.

18.9. Правильное округление ресурсов
------------------------------------
В oxsar2 встречались floor/ceil без единого правила (в одном месте
шахта даёт целые числа, в другом дробные). В nova: добыча —
аккумулируется в fixed-point, списание стоимостей — ceil в пользу
системы, начисление лута — floor в пользу игрока. Документировано
единообразно.
  Почему безопасно: устраняет копеечные расхождения, общая
  экономика не сдвигается.

18.10. res_log: обязательный причинно-следственный код
------------------------------------------------------
В oxsar2 na_res_log писался не везде. В nova: любое изменение
ресурса проходит через централизованный RecordDelta(reason, ...).
Отсутствие reason — ошибка времени компиляции (enum).
  Почему безопасно: улучшает наблюдаемость и борьбу с жалобами
  «куда делись ресурсы». Баланс не трогаем.

18.11. Единый справочник юнитов (unit registry)
-----------------------------------------------
В oxsar2 ID юнитов хардкодились через define() в consts.php и
дублировались в Java-движке и JS. В nova: единый YAML-реестр,
из которого генерируются Go-константы, TS-типы и seed базы.
  Почему безопасно: ID сохраняются теми же, меняется только
  способ их объявления. Исключает рассинхрон между слоями.

18.12. Rapidfire-таблицы — декларативные
----------------------------------------
В oxsar2 rapidfire собирался SQL-запросами к na_rapidfire. В nova —
YAML (configs/rapidfire.yml), загруженный в immutable-структуру
на старте, с проверкой консистентности (нет rapidfire по
несуществующему юниту).
  Почему безопасно: значения берутся один-в-один из Java-движка;
  меняется только механизм загрузки.

18.13. Явный таймзон
--------------------
В oxsar2 TIMEZONE = "Europe/Moscow" влиял на cron. В nova: все
внутренние расчёты и хранение — UTC. Пользовательский таймзон —
только для отображения в UI.
  Почему безопасно: техника, невидимая игроку; устраняет баги с
  переходом на летнее время.

18.14. Отказ от хардкод-админов в конфиге
-----------------------------------------
В oxsar2 $GLOBALS["ADMIN_USERS"] = [1, 3] — хардкод userid в PHP.
В nova: роль (role=ADMIN) в таблице users + admin-панель для
назначения.
  Почему безопасно: функционально эквивалентно, безопаснее и
  управляемо без деплоя.

18.15. Пароли: argon2id, отказ от единого соли
----------------------------------------------
В oxsar2 PASSWORD_SALT — глобальная строка в консте. В nova: per-
password соль в argon2id, без глобального секрета. Старые хеши
не мигрируются (новая БД).
  Почему безопасно: криптографически сильнее, без потери
  совместимости (новый универс).

18.16. IP-check действительно работает
--------------------------------------
В oxsar2 «TODO: Make ipcheck!» — функция не реализована. В nova
включена по умолчанию на уровне /24 с явным opt-out в настройках.
  Почему безопасно: снижает шанс угона сессии; у легитимных
  игроков с мобильным интернетом ложные срабатывания
  смягчаются privacy-настройкой.

18.17. Чат — модерируемый, ссылки проверяются
---------------------------------------------
В oxsar2 чат принимал HTML с минимальной фильтрацией. В nova:
текст + разрешённая подмножество markdown, ссылки
санитайзятся DOMPurify на фронте и валидируются на беке.
  Почему безопасно: закрывает XSS-вектор, функция чата
  эквивалентна.

18.18. Формула луны — как в OGame
---------------------------------
См. 12.2. Приводим к стандартной формуле сразу, не переносим
упрощённую из oxsar2.
  Почему безопасно: это общепринятый OGame-стандарт,
  игроки-ветераны ожидают именно его. Фиксируем в testdata/
  moon_probability.json.

18.19. Единообразные коды выхода миссий
---------------------------------------
В oxsar2 RETREAT_FLEET_* и RETURN_* имели пересекающиеся смыслы.
В nova: enum MissionOutcome { OK, AlreadyDone, Blocked, NotOwner,
  TooLate, TargetVanished, Protected }. Возврат клиенту —
этим enum'ом.
  Почему безопасно: чище API, UX-сообщения точнее.

18.20. Валидация координат
--------------------------
В oxsar2 проверка координат галактики делалась частично в UI.
В nova: DB-constraint + валидация в service-слое.
  Почему безопасно: закрывает класс багов «флот в координате 0:0:0».

18.21. ОТОЗВАНО (users.exchange_rate остаётся полем игрока)
-----------------------------------------------------------
Изначально в этом пункте предлагалось выпилить users.exchange_rate
и хранить курс биржи только в configs/market.yml. ОТОЗВАНО в версии
ТЗ 1.4 после разбора d:\Sources\oxsar2\www\game\Artefact.class.php.

Корректное понимание:
- users.exchange_rate — это НЕ «глобальный курс биржи», а ИМЕННО
  персональный модификатор курса у конкретного игрока.
- Значение меняется артефактом ARTEFACT_MERCHANTS_MARK (id=300) с
  дефолта 1.2 на улучшенное 1.03 при активации и обратно на 1.2
  при деактивации (Artefact.class.php:1061, 1187).
- Это один из шести персональных множителей, которые артефакты
  переключают:
    users.exchange_rate       MERCHANTS_MARK            set 1.2 ↔ 1.03
    users.research_factor     SUPERCOMPUTER             ± 1
    planets.build_factor      ROBOT_CONTROL_SYSTEM      ± 1 (planet)
    planets.produce_factor    CATALYST                  ± 0.1 (all user planets)
    planets.energy_factor     POWER_GENERATOR           ± 0.15 (all user planets)
    planets.storage_factor    ATOMIC_DENSIFIER          ± 0.15 (all user planets)
- Функция Artefact::resyncUser() (Artefact.class.php:1225) сбрасывает
  все эти поля в дефолт, что используется cron-ом ResyncArtefacts.
  Это официальный «пересчёт», а не хак.

Что ДЕЙСТВИТЕЛЬНО надо сделать в nova:
- users.exchange_rate и users.research_factor ОСТАЮТСЯ полями
  игрока (с начальными значениями 1.2 и 1 соответственно).
- planets.{build,produce,energy,storage}_factor ОСТАЮТСЯ полями
  планеты.
- Базовый курс/коэффициент вселенной (дефолты 1.2, 1, 1, ...) —
  в configs/market.yml + configs/planets.yml как константы, чтобы
  resyncUser() знал, чем сбрасывать.
- Доменный пакет artefact (§5.10) публикует функции Apply(user,
  artefactID) / Revert(user, artefactID), которые под капотом
  делают точно то же, что Artefact.class.php::activateArtefact
  (см. switch по type_id: MERCHANTS_MARK, SUPERCOMPUTER, ROBOT_
  CONTROL_SYSTEM, CATALYST, POWER_GENERATOR, ATOMIC_DENSIFIER и
  остальные эффекты).
- Добавить тест-golden: activate -> revert должно вернуть
  все шесть полей к исходным значениям. Это не обязательно
  восстановит «побитовый дефолт», потому что другой артефакт мог
  быть активен параллельно, но математика должна быть обратимой.

Ошибка предыдущей версии ТЗ (1.3): я прочитал «users.exchange_rate
= 1.2 по дефолту» как «это дефолт курса вселенной, торчащий не в
том месте». На деле — это ПОЛЕ СОСТОЯНИЯ игрока, отражающее
активные артефакты. Правильная архитектура — его сохранить, а не
убирать.

18.22. Unicode-имена игроков
----------------------------
В oxsar2 username — varbinary/varchar с кастомной фильтрацией.
В nova: CITEXT с NFC-нормализацией, список запрещённых unicode-
атрибутов (homoglyph check). Длина 3..24.
  Почему безопасно: решает проблему дубликатов "Player" и
  "Pl᧐yer", никакой механики не трогает.

18.23. Наблюдаемость по ресурсному балансу
------------------------------------------
Нет в oxsar2. В nova: ежечасный джоб проверяет сумму добытого
минус списанного по всем res_log = дельта балансов. Если
расходится — алерт.
  Почему безопасно: пассивная проверка, чисто наблюдаемость.
  Помогает быстро ловить баги экономики.

19. ЗАКРЫТЫЕ РАНЕЕ ОТКРЫТЫМИ ВОПРОСЫ
================================================================================

Правило, по которому закрываются открытые вопросы: **стабильность и
логика OGame приоритетнее легаси-квирков**. Зафиксировано в v1.5.

19.1. Artefact probability
--------------------------
Вопрос был: сохранять ли legacy-таблицу na_artefact_probobility
один-в-один? Есть подозрение, что баланс перекошен.

РЕШЕНИЕ: опираемся на OGame-стандарт выпадения артефактов из
экспедиций/боёв с инопланетянами (базовые вероятности 1–5%,
редкие — <1%). Legacy-таблица используется как ОРИЕНТИР для
«какие артефакты бывают в каких контекстах», но абсолютные
вероятности пересчитываются под OGame.

Где хранить: configs/artefact_probability.yml — список с полями
{artefact_key, source: expedition|alien_battle|boss|market,
 base_chance, min_chance, max_chance, scaling_formula}.

Плюсы стабильного подхода: баланс вселенной не зависит от
конкретных legacy-цифр oxsar.ru, которые могли быть hotfix'ами
под рекламные акции. Game-design может тюнинговать через PR без
риска «откатить продюсерский хак».

19.2. Импорт legacy-боёвых отчётов
----------------------------------
Вопрос был: как импортировать бой-репорты oxsar2, если в nova нет
MySQL?

РЕШЕНИЕ: НЕ импортируем. Nova стартует с чистой БД (§1.2 не-цели).
Legacy-отчёты сохраняются «снимком» — старый хостинг oxsar.ru
можно оставить read-only как архив, ссылки «посмотреть отчёт
с прошлого универса» ведут туда.

Для тестов боевого движка (§14.4) используем не импорт, а
синтетические фикстуры + параллельный прогон PHP/Java/Go на
одних и тех же входах. Это даёт верификацию без зависимости от
исторической БД.

19.3. Seasonal-серверы / ротация вселенных
------------------------------------------
Вопрос был: вводить ли seasonal-серверы?

РЕШЕНИЕ: да, как в OGame (universe-система). Каждая вселенная —
отдельный логический инстанс со своим universe_id.

Архитектура (зафиксирована в v1.5, детали в M8):
  - Все таблицы игрового состояния (users, planets, fleets,
    events, artefacts_user, ...) получают колонку universe_id.
  - Запросы фильтруются по universe_id middleware'ом.
  - Старт новой вселенной = запись в таблицу universes + прогон
    galaxy-генератора для её namespace.
  - Merge старого универса в main — по OGame-схеме: персонажи
    импортируются как сохранённая статистика, но прогрессия
    обнуляется. Детали — отдельный ADR при подходе к M8.

Это даёт: управляемый lifecycle, честную возможность балансовых
корректировок между сезонами, меньше риск «накопленной руды»
ломающей экономику.

19.4. Кого считать «неактивным»
-------------------------------
Вопрос был: применимо ли к nova правило oxsar2 LAST_TIME_ON_
VACATION_DISABLE = 20 дней?

РЕШЕНИЕ: ПОЛЬЗОВАТЕЛЕЙ НЕ УДАЛЯЕМ. Никогда, даже при очень долгой
неактивности. Это сознательное решение (v1.5).

Почему:
  - Удаление = безвозвратная потеря прогресса игрока, даже если
    он вернулся через год.
  - Юридические вопросы (GDPR-запросы «удалить меня» обрабатываем
    отдельно через support-endpoint, см. §9).
  - Экономические: «спящий» аккаунт — это потенциальный таргет
    атаки для других игроков (fair game при соблюдении защиты
    новичков §5.13).
  - Стабильность: отсутствие фонового cron-remove снимает целый
    класс багов «удалили активного игрока по ошибке».

Что делаем вместо удаления:
  - Флаг users.inactive_since (timestamptz) — выставляется через
    N дней без логина (N из config, default 30).
  - Правила над этим флагом:
      • в highscore отображается приглушённо/скрывается (OGame-
        style);
      • защита новичков смягчается для атакующих (§18.7);
      • экспедиции не трогаются;
      • ресурсы/флот остаются.
  - При возврате (очередной логин) флаг сбрасывается автоматически.

Единственное исключение — soft-delete ручным админским банхаммером
для нарушителей правил (users.deleted_at + anonymize email/
username). Данные на диске остаются, просто скрыты из UI.

Никаких массовых cron-утилит «удалить всех неактивных» в nova нет
и не будет.
