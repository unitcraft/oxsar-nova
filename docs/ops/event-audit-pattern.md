# Паттерн: не удалять события ради аудита

**Дата**: 2026-04-24
**Контекст**: план 17 B1 (полное уничтожение флота при outcome=lost в экспедиции)

## Проблема

При реализации 17 B1 первая попытка (commit draft) использовала
`DELETE FROM events WHERE id=$returnEventID` внутри `expLoss`, чтобы
флот не вернулся после полной потери.

Это ломает **аудит**: если завтра игрок спросит «почему мой флот
исчез», в таблице `events` не останется записи, что он вообще
планировал возвращаться. Непосредственное наблюдение пользователя
(правильное): «это приводит к изменению записи в событиях, нельзя
восстановить какие события были в случае необходимости анализа
какой-то проблемной ситуации».

## Что в проекте принято

Смотрел как устроено:
- `events.state ∈ {'wait', 'start', 'ok', 'error'}` — PostgreSQL ENUM
  ([migrations/0001_init.sql:7](../../migrations/0001_init.sql)).
  `'cancelled'` нет.
- В [`backend/internal/fleet/transport.go:449`](../../backend/internal/fleet/transport.go)
  функция `Recall` **уже использует** `DELETE FROM events` для отмены
  arrive-события. Это существующий прецедент, но он тоже ломает аудит
  (техдолг, не закрывается этим планом).

## Решение для expedition lost

Не удалять return-event и не добавлять новое состояние. Вместо этого:

1. `expLoss` удаляет `fleet_ships` и всё.
2. Handler выставляет `fleets.state='done'` (для outcome='lost') вместо
   `'returning'`.
3. **Return-event оставляем нетронутым**. Когда он сработает,
   `ReturnHandler` (events.go:223) увидит `state='done'` и молча
   завершится в `state='ok'` — это ранний выход, уже существующий
   в коде.

**Что остаётся в БД после потери флота**:
- `events`: обе записи (expedition + return) с полной историей,
  `state='ok'`, нормальные `processed_at`.
- `fleets`: `state='done'`, `arrive_at`, `return_at`.
- `expedition_reports`: запись с `outcome='lost'`.
- `messages`: сообщение игроку с подробностями.
- Удалено только `fleet_ships` — у мёртвого флота кораблей нет.

## Общий принцип (на будущее)

При обработке «отмены» — по возможности **не удалять** записи в
`events`, а использовать естественный early-exit handler'а через
смежное состояние (fleet.state, planet.destroyed_at и т.п.). История
события (wait → ok/error) остаётся в БД для аудита.

DELETE допустим только если:
- Событие точно не должно быть видно игроку как «произошло»
- Нет handler'а, который бы естественно no-op-нулся
- Или у нас появится `state='cancelled'` в enum

## Технический долг

`transport.go:449` — `Recall` делает `DELETE FROM events WHERE kind=7`
для отмены arrive-события. То же рассуждение применимо. Не трогаем в
рамках плана 17 B1, но желательно переделать через аналогичный
early-exit: handler arrive видит `fleets.state='returning'` и молча
пропускает.

Когда появится время — отдельный план, потенциально вместе с
добавлением `'cancelled'` в ENUM `event_state` (миграция
`ALTER TYPE event_state ADD VALUE 'cancelled'`).
